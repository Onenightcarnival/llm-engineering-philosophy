# CLAUDE.md -- 项目指引

## 项目定位

一本关于大模型应用开发最佳实践的技术著作，也是一个结构化的知识库。不是教程，不是文档汇编，不是框架推荐指南，而是一个人对"软件工程在大模型时代应该是什么样子"的系统性回答。

双重读者：人和大模型。如果一份文档不能同时对人和机器保持清晰，那它的结构本身就有问题。

## 写作语气校准

- 让观点自己说话。不通过罗列作者背景来为观点增加权威性。
- 避免自证式叙事：不用"因为我做过 X，所以我有资格说 Y"的结构。如果一个观点需要作者简历来支撑，说明观点本身的论证不够有力。
- 跨领域经验（数学、工程、投资）可以作为类比和思维工具的来源，但以抽象原则的形式出现，不以个人经历的形式出现。
- 锋利但不傲慢。明确的判断和明确的偏好是本书的特色，但"我比别人看得清"不是。
- 自省要真诚：承认局限性时，指向真实的盲区，而非变相的自夸。

## 核心原则

### 写作立场

- 表达个人观点与哲学观，不追求从众，不依赖权威背书。
- 偏重对未来的构想以及对事物本质结构的揭示，而非对既有事实的罗列。
- 概念边界清晰，论证链条完整，能区分"是什么"、"为什么"和"应该是什么"。
- 允许跨领域类比（投资、数学、物理），但类比必须揭示结构性同构，而非表面相似。

### 写作风格

- 中文为主。不使用 emoji、icon 或任何装饰性符号。
- 行文严肃但不刻板，允许锋利的判断和明确的偏好。
- 标题层级不超过二级（h1-h2）。
- 代码块仅用于：（1）架构蓝图（Agent 循环、RAG 流程等结构模式）；（2）Prompt 的代码化表达（如 Pydantic 模型）；（3）好写法与坏写法的代码层面对比。概念分类、决策框架、思维模型、简单算术用散文、表格或列表表达。

### 技术倾向

声明式优于命令式，类型系统优于运行时检查，组合优于继承，显式依赖优于隐式约定，结构驱动优于流程驱动，简单方案优于"看起来高级"的方案。对"框架崇拜"持批判态度。

### 思维模型

- **Strategy > Analysis** -- 架构决策先于实现细节。
- **收敛性判据** -- 迭代过程应有明确的收敛判据：是否在接近目标，何时停止，何时发散。
- **数值稳定性** -- 理论等价的路径在实际执行中可能产生截然不同的结果。
- **约束传播** -- 约束越精确，解空间越小，系统行为越可预测。
- **结构性同构** -- 不同领域具有相同数学结构的问题，解法可以迁移。

## 项目结构

```
awesome-llm-practice/
  CLAUDE.md                  -- 本文件
  README.md                  -- 项目入口，书籍目录
  chapters/                  -- 各章节内容
    00-序章/                  -- 为什么写这本书
    01-认识论/                -- 大模型的本质与边界
    02-不确定性与决策/         -- 不确定性环境下的工程决策框架
    03-人机协作的软件过程/      -- 开发方法论的重构（工作方式先于技术细节）
    04-声明式提示与类型契约/    -- 从命令式到声明式的范式跃迁，类型系统作为约束工具
    05-架构与编排/             -- LLM 应用的结构设计与动态执行
    06-测试评估与可观测性/      -- 不确定性系统的质量保障
    07-反模式与陷阱/           -- 值得警惕的实践
    08-终章/                  -- 软件工程的下一个形态
```

章节目录下包含 `00-概述.md`（章节概述与文章索引）和按阅读顺序编号的独立 `.md` 文章。Markdown 文件命名格式：`序号-中文名.md`，序号两位数字。

## 全书结构逻辑

9 章遵循"三个地基 → 两层构建 → 一层保障 → 一层警示 → 一个收束"的推演链：

- **第 0-3 章（地基）：** 序章划定边界；第一章建立哲学地基（自回归本质、语言作为接口、确定性与概率性的张力）；第二章建立方法论地基（不确定性环境下的工程决策框架）；第三章建立过程地基（人机协作的开发方式，先于技术细节建立工作习惯）。
- **第 4-5 章（构建）：** 第四章从 prompt 工程的范式跃迁到类型系统作为声明工具（组件级约束）；第五章从系统级架构模式到工作流的动态编排（系统级设计）。
- **第 6 章（保障）：** 测试、评估与可观测性，回答"怎么知道系统在正常工作"。
- **第 7 章（警示）：** 反模式与陷阱，回答"什么不该做"。
- **第 8 章（收束）：** 终章回到哲学层面与第一章呼应，构想软件工程的下一个形态。

## 协作规范

- 理解本项目的立场和品味后再动笔。不写空话、套话、正确的废话。
- 不使用"让我们"、"大家都知道"之类的口语化表达。
- 不回避争议性判断。作者观点与主流不同时，忠实表达，不做对冲。
- 论证靠逻辑和证据，不靠作者背景。"这个判断是对的因为 X"，而非"这个判断是对的因为作者做过 Y"。
- 每篇文章应有清晰的论点，而非话题的泛泛介绍。
- 跨领域类比必须揭示结构性同构，能指向具体的工程启示。
- 概念首次出现时给出精确定义。
- 论证链条完整：主张 -> 理由 -> 证据/示例 -> 局限性讨论。
- 数学概念的引用要准确，不为类比牺牲数学含义的严谨性。
- 引用其他章节时使用相对路径。
