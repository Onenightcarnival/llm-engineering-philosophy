# CLAUDE.md -- 项目指引

## 项目定位

本项目是一本关于大模型应用开发最佳实践的技术著作，同时也是一个结构化的知识库。
它不是教程，不是文档汇编，不是框架推荐指南，而是一个人对"软件工程在大模型时代应该是什么样子"这个问题的系统性回答。

这本书既是给人阅读的技术著作，也是给大模型提供的知识库。这种双重身份不是巧合，而是刻意的设计——如果一份文档不能同时对人和机器保持清晰，那它的结构本身就有问题。

## 作者画像

理解作者是谁，才能理解这本书为什么是这样而不是那样。

### 学术背景

计算机博士，软件工程与数据科学双分支。同时持有数学学位，计算数学方向。

这三个学科塑造了不同层面的思维方式：

- 软件工程训练了对系统结构、抽象层级和工程权衡的直觉。它提供的是"如何构建可靠系统"的方法论。
- 数据科学训练了对不确定性的量化能力和对实验方法的尊重。它提供的是"如何在噪声中发现信号"的认识论。
- 计算数学训练了对收敛性、数值稳定性和形式化的敏感。它提供的不是具体算法，而是一种将模糊问题精确化的直觉：定义变量、建立约束、求解或证明无解。

这三个维度在本书中不是分割的，而是交织的。讨论 prompt 设计时会借用数值稳定性的隐喻，讨论架构决策时会引入期望值和方差的思维框架，讨论测试策略时会运用统计假设检验的方法。

### 决策哲学

作者是一个身经百战的草根投资者，拥有超过十几年的成功投资经验。家族背景是私募量化，但本人追求自由，只管理自己的资金。

这段经历不是简历装饰，而是本书多处论证的底层思维来源：

**Strategy 大于 Analysis。** 分析可以无穷无尽，但战略层面的判断才是决定成败的关键。2022 年英伟达股价跌至 100 美元附近时 all in 计划仓位，2025 年 4 月关税战标普回测 4800 点时 all in 计划仓位——这些决策不来自更多的分析，而来自对市场结构的深刻理解和对人性弱点的克服。战略正确时，战术错误都可以修正；战略错误时，战术再精湛也无法挽回。

**知行合一。** 好的判断如果无法执行就毫无价值。知道应该做什么和真正做到之间，隔着人性的弱点：恐惧、从众、对复杂性的迷恋、对简单方案的不信任。投资如此，工程亦然。

**不确定性是约束条件，不是敌人。** 波动性不是市场的缺陷，是市场的本质属性。概率性输出不是 LLM 的缺陷，是 LLM 的本质属性。在两个领域中，正确的做法都不是消除不确定性，而是在给定不确定性水平下做出最优决策。

这些原则贯穿全书，尤其集中体现在第二章"不确定性与决策"中。

### 性格特质

独立。不为别人管钱，不为迎合读者写作。这本书表达的是个人品味，不是共识。

不从众。在极端恐慌时逆向操作的能力，来自对第一性原理的信任和对人性弱点的认知。同样的思维方式投射到技术写作中：当所有人都在追捧某个框架时，更值得问的问题是"它解决了什么本质问题"。

追求本质结构。无论是投资标的的内在价值、数学问题的形式化表述，还是软件系统的架构模式——关注的始终是表象之下的结构。

## 核心原则

### 写作立场

- 表达个人观点与哲学观，不追求从众，不依赖权威背书。
- 一句话的正确与否不在于是谁说的，在于它是否经得起推敲。
- 偏重对未来的构想以及对事物本质结构的揭示，而非对既有事实的罗列。
- 博士水准意味着：概念边界清晰，论证链条完整，能区分"是什么"、"为什么"和"应该是什么"。
- 允许跨领域类比（投资、数学、物理），但类比必须揭示结构性同构，而非表面相似。

### 写作风格

- 语言：中文为主，后续扩展英文翻译。
- 不使用 emoji、icon 或任何装饰性符号。
- 行文严肃但不刻板，允许锋利的判断和明确的偏好。
- 代码即论据：每个观点尽可能有可执行的 Python 示例支撑。
- 标题层级不超过四级（h1-h4），避免过度碎片化。

### 技术倾向

- 偏好声明式而非命令式。
- 偏好类型系统而非运行时检查。
- 偏好组合而非继承。
- 偏好显式依赖而非隐式约定。
- 偏好结构驱动而非流程驱动。
- 偏好简单方案而非"看起来高级"的方案。
- 对"框架崇拜"持批判态度，重视对底层机制的理解。

### 思维模型

以下思维模型在全书中反复出现，理解它们有助于把握行文逻辑：

- **Strategy > Analysis** -- 战略层面的正确性优先于战术层面的精细度。架构决策先于实现细节。
- **收敛性判据** -- 借自计算数学。任何迭代过程（prompt 优化、架构演进、系统调优）都应该有明确的收敛判据：是否在接近目标，何时应该停止，什么条件下会发散。
- **数值稳定性** -- 借自计算数学。理论上等价的两条路径，在实际执行中可能产生截然不同的结果。语义等价的两种 prompt 表述，可能导致质量的巨大差异。
- **约束传播** -- 借自类型理论。好的约束不是限制，是对问题空间的精确建模。约束越精确，解空间越小，系统行为越可预测。
- **结构性同构** -- 不同领域的问题如果具有相同的数学结构，那么解法也可以迁移。投资决策和架构决策的同构性不是修辞，是可操作的分析框架。

### 章节内文件约定

每个章节目录下包含：
- `README.md` -- 章节概述与文章索引
- 一篇或多篇独立的 `.md` 文章
- 如有配套代码，放在同目录下的 `.py` 文件中，文件名与文章对应

### 文件命名

- Markdown 文件使用英文命名，以连字符分隔，全小写。
- 示例：`declarative-chain-of-thought.md`
- Python 文件与对应的 Markdown 文件同名：`declarative-chain-of-thought.py`

## 全书结构逻辑

12 章（含序章和终章）遵循一条从认识论到方法论再到未来构想的推演链：

### 第零章 序章：为什么写这本书

划定边界。声明这本书不是什么，作者的偏见是什么。开宗明义。

### 第一章 认识论：大模型的本质与边界

全书的哲学地基。从计算理论和语言哲学的角度理解大模型到底是什么。不讨论参数量和训练技巧，而是讨论：自回归模型的生成本质、语言作为接口的含义、确定性与概率性的根本张力。这一章决定后续所有章节的基调。

### 第二章 不确定性与决策

全书的方法论地基。将作者从投资实战和计算数学中提炼的决策框架引入软件工程语境。核心主张：LLM 应用开发的核心困难不是技术实现，而是在不确定性环境下做出正确的工程决策。Strategy 大于 analysis，知行合一，恐慌时刻的逆向思维，决策的数学结构——这些跨领域迁移的思维工具在此展开。

### 第三章 提示工程作为软件工程

从已有文章"Declarative Chain-of-Thought"延伸。核心主张：提示工程不是手艺，是软件工程的一个分支。从命令式到声明式的范式跃迁，"Code as Prompt"的工程原则。

### 第四章 类型系统与契约

第三章的自然延伸。如果 prompt 应该是声明式的，那么类型系统就是最好的声明工具。Pydantic 作为 Prompt DSL、JSON Schema 作为契约、约束传播机制在 LLM 应用中的释放。

### 第五章 架构模式

从单次调用扩展到系统级设计。RAG 的本质、Agent 的结构分解、LLM 的架构定位。包含对主流架构模式的批判性分析。

### 第六章 工作流与编排

静态结构（第五章）的动态展开。编排的层次、状态机模型、错误传播。包含对主流编排框架的批判。

### 第七章 测试与可靠性

回答"怎么知道系统在正常工作"。把不确定性纳入测试框架而非试图消除它。

### 第八章 数据科学与计算思维

作者三重学科背景的交汇点。不仅是数据科学方法论的应用，更引入计算数学的思维工具：收敛性分析、数值稳定性、迭代法的判据。这些抽象工具在 prompt 优化和系统调优中的投射。

### 第九章 人机协作的软件过程

前面八章讨论"做什么"和"怎么做"，这一章讨论"以什么方式工作"。LLM 同时改变了被开发的对象和开发的过程本身。

### 第十章 反模式与陷阱

反面教材。在一个快速演化、缺乏共识的领域，"不做什么"的指导价值不亚于"做什么"。

### 第十一章 终章：软件工程的下一个形态

回到哲学层面，与第一章形成呼应。从认识论开始，以方法论的远景结束。

## 协作规范

### 对 Claude 的要求

- 理解本项目的立场和品味后再动笔。不写空话、套话、正确的废话。
- 不要在文章中使用"让我们"、"大家都知道"之类的口语化表达。
- 不要回避争议性判断。如果作者的观点与主流不同，忠实表达，不做对冲。
- 代码示例必须是可运行或接近可运行的 Python，不写伪代码（除非明确标注）。
- 每篇文章应有清晰的论点，而非话题的泛泛介绍。一篇文章解决一个问题或确立一个观点。
- 引用其他章节时使用相对路径。
- 不使用 emoji、icon 或任何装饰性符号。
- 跨领域类比（投资、数学、物理）是本书的特色之一，但每个类比必须揭示结构性同构，而非表面修辞。如果一个类比无法指向具体的工程启示，就不要用它。

### 质量标准

- 概念首次出现时给出精确定义。
- 区分事实陈述与价值判断，但不回避价值判断。
- 论证链条完整：主张 -> 理由 -> 证据/示例 -> 局限性讨论。
- 代码示例与正文论述严格对应，不出现"示意性"的断裂。
- 数学概念的引用要准确，不能为了类比而牺牲数学含义的严谨性。
