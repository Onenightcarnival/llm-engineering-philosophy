# CLAUDE.md -- 项目指引

## 项目定位

一本关于大模型应用开发最佳实践的技术著作，也是一个结构化的知识库。不是教程，不是文档汇编，不是框架推荐指南，而是一个人对"软件工程在大模型时代应该是什么样子"的系统性回答。

双重读者：人和大模型。如果一份文档不能同时对人和机器保持清晰，那它的结构本身就有问题。

## 作者画像

- **学术背景：** 计算机科学硕士（数据科学），数学学士（计算数学）。数据科学提供了对不确定性的量化能力；计算数学提供了将模糊问题精确化的直觉（收敛性、数值稳定性、形式化）。两者在全书中交织：prompt 设计借用数值稳定性隐喻，架构决策引入期望值和方差框架，测试策略运用统计假设检验。
- **工程背景：** 一线大厂数年实战，参照 Spring 设计哲学独立设计过 Python 开发框架。这提供了"从内部看框架"的视角——对 IoC/DI、声明式配置、AOP、分层架构等模式有内化的理解，不是作为使用者，而是作为设计者。对"框架崇拜"的批判态度是造过框架的人对框架局限性的亲身认知。
- **决策哲学：** 十几年成功投资经验，家族背景私募量化，本人只管理自己的资金。核心原则：（1）Strategy 大于 Analysis——战略正确时战术错误可修正，反之不然；（2）知行合一——好的判断无法执行就毫无价值；（3）不确定性是约束条件而非敌人——概率性输出是 LLM 的本质属性，正确做法是在给定不确定性下做最优决策。
- **性格特质：** 独立，不从众，追求本质结构。这本书表达的是个人品味，不是共识。

## 核心原则

### 写作立场

- 表达个人观点与哲学观，不追求从众，不依赖权威背书。
- 偏重对未来的构想以及对事物本质结构的揭示，而非对既有事实的罗列。
- 概念边界清晰，论证链条完整，能区分"是什么"、"为什么"和"应该是什么"。
- 允许跨领域类比（投资、数学、物理），但类比必须揭示结构性同构，而非表面相似。

### 写作风格

- 中文为主。不使用 emoji、icon 或任何装饰性符号。
- 行文严肃但不刻板，允许锋利的判断和明确的偏好。
- 标题层级不超过二级（h1-h2）。
- 代码块仅用于：（1）架构蓝图（Agent 循环、RAG 流程等结构模式）；（2）Prompt 的代码化表达（如 Pydantic 模型）；（3）好写法与坏写法的代码层面对比。概念分类、决策框架、思维模型、简单算术用散文、表格或列表表达。

### 技术倾向

声明式优于命令式，类型系统优于运行时检查，组合优于继承，显式依赖优于隐式约定，结构驱动优于流程驱动，简单方案优于"看起来高级"的方案。对"框架崇拜"持批判态度。

### 思维模型

- **Strategy > Analysis** -- 架构决策先于实现细节。
- **收敛性判据** -- 迭代过程应有明确的收敛判据：是否在接近目标，何时停止，何时发散。
- **数值稳定性** -- 理论等价的路径在实际执行中可能产生截然不同的结果。
- **约束传播** -- 约束越精确，解空间越小，系统行为越可预测。
- **结构性同构** -- 不同领域具有相同数学结构的问题，解法可以迁移。

  chapters/                  -- 各章节内容

    02-不确定性与决策/         -- 不确定性环境下的工程决策框架
    03-提示工程作为软件工程/    -- 从手艺到工程的范式跃迁
    04-类型系统与契约/         -- 用编程语言的形式化能力约束 LLM
    05-架构模式/              -- LLM 应用的结构设计
    06-工作流与编排/           -- 复杂任务的分解与组合
    07-测试与可靠性/           -- 不确定性系统的质量保障

章节目录下包含 `00-概述.md`（章节概述与文章索引）和按阅读顺序编号的独立 `.md` 文章。Markdown 文件命名格式：`序号-中文名.md`，序号两位数字。

## 全书结构逻辑

12 章遵循从认识论到方法论再到未来构想的推演链：

- **第 0-2 章（地基）：** 序章划定边界；第一章建立哲学地基（自回归本质、语言作为接口、确定性与概率性的张力）；第二章建立方法论地基（不确定性环境下的工程决策框架）。
- **第 3-6 章（构建）：** 从 prompt 工程的范式跃迁（第三章），到类型系统作为声明工具（第四章），到系统级架构模式（第五章），到工作流的动态编排（第六章）。各章之间是自然递进关系。
- **第 7-8 章（保障）：** 测试与可靠性（第七章）回答"怎么知道系统在正常工作"；数据科学与计算思维（第八章）是作者学术背景的交汇投射。
- **第 9-11 章（反思与展望）：** 人机协作的软件过程（第九章），反模式与陷阱（第十章），终章回到哲学层面与第一章呼应。

## 协作规范

- 理解本项目的立场和品味后再动笔。不写空话、套话、正确的废话。
- 不使用"让我们"、"大家都知道"之类的口语化表达。
- 不回避争议性判断。作者观点与主流不同时，忠实表达，不做对冲。
- 每篇文章应有清晰的论点，而非话题的泛泛介绍。
- 跨领域类比必须揭示结构性同构，能指向具体的工程启示。
- 概念首次出现时给出精确定义。
- 论证链条完整：主张 -> 理由 -> 证据/示例 -> 局限性讨论。
- 数学概念的引用要准确，不为类比牺牲数学含义的严谨性。
- 引用其他章节时使用相对路径。
