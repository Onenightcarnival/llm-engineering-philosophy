# CLAUDE.md -- 项目指引

## 项目定位

本项目是一本关于大模型应用开发最佳实践的技术著作，同时也是一个结构化的知识库。
它不是教程，不是文档汇编，而是一个人对"软件工程在大模型时代应该是什么样子"这个问题的系统性回答。

作者背景：计算机博士，软件工程与数据科学双分支。
内容底色：软件工程哲学，第一性原理推演，结构性洞见。

## 核心原则

### 写作立场

- 表达个人观点与哲学观，不追求从众，不依赖权威背书。
- 一句话的正确与否不在于是谁说的，在于它是否经得起推敲。
- 偏重对未来的构想以及对事物本质结构的揭示，而非对既有事实的罗列。
- 博士水准意味着：概念边界清晰，论证链条完整，能区分"是什么"、"为什么"和"应该是什么"。

### 写作风格

- 语言：中文为主，后续扩展英文翻译。
- 不使用 emoji、icon 或任何装饰性符号。
- 行文严肃但不刻板，允许锋利的判断和明确的偏好。
- 代码即论据：每个观点尽可能有可执行的 Python 示例支撑。
- 标题层级不超过四级（h1-h4），避免过度碎片化。

### 技术倾向

- 偏好声明式而非命令式。
- 偏好类型系统而非运行时检查。
- 偏好组合而非继承。
- 偏好显式依赖而非隐式约定。
- 偏好结构驱动而非流程驱动。
- 对"框架崇拜"持批判态度，重视对底层机制的理解。

## 内容概要

### 第零章 序章：为什么写这本书

不是又一本 prompt 手册。这本书试图回答的问题是：当软件系统的核心组件从确定性函数变成概率性语言模型时，软件工程的基本原则会发生什么变化，又有什么不会变。

### 第一章 认识论：大模型的本质与边界

从计算理论和语言哲学的角度理解大模型到底是什么。不讨论参数量和训练技巧，而是讨论：自回归模型的生成本质、语言作为接口的含义、确定性与概率性的根本张力。这是全书的哲学地基。

### 第二章 提示工程作为软件工程

提示工程不应该是一门手艺，而应该是软件工程的一个分支。本章讨论从命令式到声明式的范式跃迁，抽象层级的演化规律在 LLM 领域的再现，以及"Code as Prompt"的核心主张。已有文章"Declarative Chain-of-Thought"属于本章。

### 第三章 类型系统与契约

类型系统是人类在软件工程中发明的最强大的约束传播机制之一。本章讨论如何将这种力量投射到 LLM 应用中：Pydantic 作为 Prompt DSL、JSON Schema 作为契约、Literal 类型作为决策空间的显式枚举、验证器作为运行时不变量。

### 第四章 架构模式

LLM 应用并非没有架构可言。本章提炼若干核心架构模式：RAG 的本质是什么（检索增强还是上下文注入）、Agent 的结构分解、状态管理与会话设计、LLM 作为胶水层 vs LLM 作为核心引擎。

### 第五章 工作流与编排

当任务复杂度超出单次调用的承载能力，就需要编排。本章讨论：隐式编排（结构驱动）vs 显式编排（DAG 驱动）、多轮交互的状态机模型、错误传播与补偿机制、为什么大多数 LLM 编排框架过度设计了。

### 第六章 测试与可靠性

如何测试一个输出不确定的系统？本章讨论：结构测试（Schema 层面）vs 语义测试（含义层面）、基于属性的测试在 LLM 应用中的适配、评估即测试、可观测性设计、优雅降级策略。

### 第七章 数据工程视角

数据科学训练带来的独特视角。本章讨论：prompt 的版本控制与实验管理、输出质量的统计度量、数据飞轮的构建、特征工程思维在 prompt 设计中的投射、何时该微调而非继续prompt。

### 第八章 人机协作的软件过程

LLM 不仅是被开发的对象，也是开发过程的参与者。本章讨论：AI 辅助编程的正确姿势、代码审查中的人机分工、文档驱动开发在 LLM 时代的复兴、知识库作为活的系统规格说明。

### 第九章 反模式与陷阱

值得用一整章来讨论"不应该做什么"。包括：prompt 拼接的脆弱性、过度依赖单一模型、忽视成本模型、把 LLM 当数据库用、框架锁定、评估缺失综合征、以及最隐蔽的反模式——把不确定性当 bug 而非特性。

### 第十章 终章：软件工程的下一个形态

软件工程从未停止演化。从结构化编程到面向对象，从瀑布到敏捷，每次范式转换都重新定义了"什么是好的软件"。LLM 带来的不是又一个工具，而是又一次范式转换。本章是对未来的构想。

## 协作规范

### 对 Claude 的要求

- 理解本项目的立场和品味后再动笔。不写空话、套话、正确的废话。
- 不要在文章中使用"让我们"、"大家都知道"之类的口语化表达。
- 不要回避争议性判断。如果作者的观点与主流不同，忠实表达，不做对冲。
- 代码示例必须是可运行或接近可运行的 Python，不写伪代码（除非明确标注）。
- 每篇文章应有清晰的论点，而非话题的泛泛介绍。一篇文章解决一个问题或确立一个观点。
- 引用其他章节时使用相对路径。

### 质量标准

- 概念首次出现时给出精确定义。
- 区分事实陈述与价值判断，但不回避价值判断。
- 论证链条完整：主张 -> 理由 -> 证据/示例 -> 局限性讨论。
- 代码示例与正文论述严格对应，不出现"示意性"的断裂。
