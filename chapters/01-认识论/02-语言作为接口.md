# 语言作为接口：比 API 更古老，比 API 更模糊

## 接口的历史与自然语言的回归

软件系统的接口经历了一条从模糊到精确、再从精确回到模糊的历史弧线。

最早的计算机通过拨动开关和接线来交互——物理层面的接口，精确但极度受限。然后是命令行，用受限的自然语言子集（命令 + 参数）与系统交互。然后是 API，用形式化的函数签名定义输入和输出的类型契约。然后是 GUI，用视觉隐喻将操作映射到按钮和菜单。每一步演化都在扩大用户的范围，同时在接口的精确性和表达力之间寻找新的平衡点。

API 是这条演化路径上的一个高点：它用类型系统、参数名、返回值定义了一个无歧义的交互契约。调用 `get_user(id=42)` 的含义没有任何模糊空间。这种精确性是过去几十年软件工程得以工业化的基础——系统之间的集成、测试、文档、版本控制，全部建立在"接口契约是无歧义的"这个前提之上。

大模型把接口从形式化语言拉回了自然语言。用户用英语、中文、日语向系统描述需求，系统用自然语言返回结果。这在可达性上是巨大的进步——交互的门槛大幅降低了。但从精确性的角度看，这是一次倒退：自然语言是人类最强大的表达工具，同时也是最不精确的。

## 自然语言的模糊性不是缺陷，是特征

自然语言的模糊性来源至少有三个层面，每个层面都有对应的工程后果。

**词汇歧义。** 同一个词在不同语境下有不同含义。"bank"可以是银行也可以是河岸。"苹果"可以是水果也可以是公司。"表"可以是数据库的表、时间表、仪表。在人类对话中，语境通常足以消歧。但在 prompt 中，消歧依赖于模型对上下文的统计推断，而不是形式化的绑定。当 prompt 说"提取文本中的所有表"，模型是提取 table 还是 list 还是 schedule？这取决于上下文中其他 token 激活的概率分布。

**句法歧义。** "用望远镜看到了那个男人"——是用望远镜看的，还是看到了一个拿着望远镜的人？"Flying planes can be dangerous"——飞行的飞机危险，还是驾驶飞机这件事危险？这类歧义在 prompt 中同样存在。"用 JSON 格式总结以下内容"——是输出格式为 JSON，还是用 JSON 的结构方式来组织摘要？

**语用歧义。** "能把窗户打开吗"是疑问句，但语用含义是请求。"这个方案不错"可能是认可，也可能是讽刺。语用层面的歧义是最难处理的，因为它依赖于说话者的意图、双方的共享知识、对话的历史——这些信息在一个 prompt 中通常是不完整的。

这三个层面的模糊性在传统 API 设计中被完全消除了——类型系统不允许歧义存在。`def get_user(id: int) -> User` 中的每一个符号都有且仅有一个含义。但在自然语言接口中，歧义被重新引入了，而且没有编译器来捕获它。

以上三种歧义不是理论构造，它们在真实的 prompt 工程中反复出现。以下是几个典型场景及其工程后果：

**"提取文本中的关键信息"** 是词汇歧义的典型代表。"关键信息"可以被解读为命名实体（人名、地名、组织名），可以是核心论点和论据，可以是数值数据和统计信息，也可以是与用户查询最相关的段落。同一个 prompt 在不同调用中可能提取出完全不同类型的内容，因为"关键"这个词的语义锚定完全依赖模型当时的上下文推断。工程后果是直接的：输出内容不可预测，下游的处理管道无法对输入的结构做任何假设。

**"简短地回答以下问题"** 是语用歧义的典型代表。"简短"对不同的人意味着不同的东西——一个词、一句话、还是一个段落但不要太长？模型对"简短"的理解会随上下文、问题复杂度、甚至采样温度的变化而波动。工程后果在于输出长度不可控，而长度的不可控直接影响后续处理流程的稳定性和 token 成本的可预测性。

**"如果信息不足，请说明"** 同样是语用歧义。"说明"的含义可以是在输出中附加一段解释文字，可以是输出一个特定标记（如 N/A），可以是返回部分结果外加说明，也可以是完全拒绝回答。这些不同的行为模式对应着完全不同的错误处理路径。当系统无法可靠地区分"正常输出"和"异常情况"时，任何下游的容错逻辑都建立在不可靠的假设之上。

这些例子揭示的共同模式是：自然语言 prompt 中几乎每一个看似明确的指令，都包含着被忽视的歧义空间。而这些歧义空间不会被编译器、类型检查器或任何静态分析工具捕获——它们只在运行时以不可预测的方式显现。

## API 契约与自然语言"契约"的结构性差异

形式化 API 和自然语言接口之间的差异，不是程度的差异，而是范畴的差异。

形式化 API 的契约具有以下性质：

- **明确性**：每个参数的类型、每个返回值的结构、每个错误条件的处理方式都有形式化定义。
- **可验证性**：契约的遵守与否可以被自动化工具检查。类型检查器、契约验证器、接口测试——这些工具存在的前提是契约本身是形式化的。
- **可组合性**：两个 API 可以通过类型匹配来组合。`f: A -> B` 和 `g: B -> C` 可以组合为 `g . f: A -> C`，前提是 B 的类型定义是共享的。
- **版本可控性**：API 的变更可以被语义化版本控制（semver）追踪。breaking change 有明确的定义：参数类型变了、返回值结构变了、错误条件变了。

自然语言"契约"（即 prompt）不具备这些性质：

- **模糊性**：prompt 的含义依赖于模型的"理解"，而这种"理解"是统计性的、不可形式化的。
- **不可验证性**：无法自动化检查模型是否"正确理解"了 prompt 的意图。只能检查输出是否符合某种预期，但"预期"本身通常也是模糊的。
- **弱组合性**：两个 prompt 的组合不像两个函数的组合那样可预测。将两段 prompt 拼接在一起，效果不等于两段 prompt 分别执行的组合。
- **版本不可控性**：prompt 的"含义"会随着模型版本的变化而漂移。同一个 prompt 在 GPT-4 和 GPT-4o 上可能产生不同质量的输出，而且这种差异无法被 semver 追踪。

这四组对比揭示的核心问题是：形式化 API 的契约是语法层面的约束，可以被机器解析和执行；自然语言"契约"是语义层面的期望，只能被人类（或另一个语言模型）近似地理解。形式化 API 的输入验证可以做到完备——任何不符合类型签名的输入在编译期就会被拒绝。自然语言接口的输入验证几乎不存在——任何字符串都是"合法"的 prompt，这意味着输入层面的保护网是空的。

更根本的差异体现在输出验证上。形式化 API 的输出满足后置条件可以被自动化地、确定性地检查。而自然语言输出的正确性在概念上就是不可形式化定义的——什么叫"正确理解了用户意图"？什么叫"回答是准确的"？这些判断要么依赖人工评估，要么依赖另一个语言模型的近似判断，要么依赖启发式规则对输出格式和内容的粗粒度检查。无论哪种方式，都不具备类型系统那种数学意义上的可靠性。

## 模糊性的工程代价

自然语言接口的模糊性不是理论问题，是有可量化代价的工程问题。

**调试成本倍增。** 当确定性 API 的行为不符合预期时，调试路径是清晰的：检查输入是否满足前置条件，检查实现逻辑是否正确，检查输出是否满足后置条件。当自然语言接口的行为不符合预期时，调试路径是模糊的：是 prompt 写得不好？是模型"理解"有误？是采样策略不合适？是上下文中的干扰信息导致了偏移？这些可能性之间没有清晰的隔离边界。

**测试复杂度激增。** 确定性 API 的测试是基于等价的：对于每一类输入，存在一个（且仅一个）正确输出。自然语言接口的测试是基于分布的：对于每一个 prompt，跑很多次会产生不同的输出，测试需要验证的是整体的输出质量——平均水平如何、波动有多大、最差的情况是否可接受。这从根本上改变了测试方法论。

**组合爆炸。** 当系统中有多个 LLM 调用需要串联时，每个调用的输出模糊性会在下游放大。如果第一步的输出有 3 种可能的解读，第二步对每种解读有 3 种可能的响应，两步串联就有 9 种可能的路径。这和数值计算中的误差传播是同一类问题，但比数值误差更难量化——因为语义层面的"误差"没有自然的度量。

## 从模糊到精确：工程策略

承认自然语言接口的模糊性是本质属性，不意味着放弃精确性的追求。工程上的策略是：在自然语言接口的外层包裹精确性，而不是试图让自然语言本身变得精确。

**策略一：用类型系统约束输出空间。** 不要让模型自由生成自然语言，而是要求模型在一个预定义的类型结构中填充内容。Pydantic 模型、JSON Schema、枚举类型——这些工具的本质是将模型的输出从"任意字符串"约束到一个有限的、可验证的空间中。这是第四章的核心主题。

**策略二：用程序化 prompt 替代自然语言 prompt。** 与其用自然语言描述你要什么，不如用代码结构来声明你要什么。代码中的变量名、类型注解、文档字符串本身就是比自然语言更精确的规格说明。这是第三章 "Code as Prompt" 主张的核心。

**策略三：将语义验证嵌入系统设计。** 既然输出的正确性无法被类型系统完全捕获，就需要在系统中嵌入额外的验证层。这些验证不是 assert 语句那样的硬约束，而是统计性的质量检查：输出是否包含期望的关键信息？格式是否符合预期？长度是否在合理范围内？语义是否与输入一致？这是第七章的核心主题之一。

```python
from pydantic import BaseModel, Field
from typing import Literal


class SentimentResult(BaseModel):
    """
    通过类型定义将"情感分析"从一个模糊的自然语言任务
    转化为一个有精确输出契约的结构化任务。

    对比两种 prompt：

    模糊版本："分析这段文本的情感倾向"
    -> 模型可能返回任何格式的自然语言描述

    精确版本：要求模型填充以下结构
    -> 输出空间被约束为一个可验证的数据结构
    """
    sentiment: Literal["positive", "negative", "neutral"]
    confidence: float = Field(ge=0.0, le=1.0)
    key_phrases: list[str] = Field(
        min_length=1, max_length=5,
        description="支撑情感判断的关键短语"
    )
    reasoning: str = Field(
        max_length=200,
        description="一句话解释判断理由"
    )
```

这个 Pydantic 模型做了几件事：将 `sentiment` 从一个开放的自然语言回答约束为三个枚举值之一；将 `confidence` 约束为 0 到 1 之间的浮点数；将 `key_phrases` 约束为 1 到 5 个字符串的列表；将 `reasoning` 约束为不超过 200 字符的字符串。每一个约束都是在缩小输出空间，增加可预测性。

## 语言作为接口的不可逆性

尽管自然语言接口存在以上所有问题，它的普及是不可逆的。原因不在技术层面，在经济层面：自然语言接口极大地降低了软件系统的使用门槛。过去需要学习 SQL 才能查询数据库，学习 API 才能调用服务，学习编程才能自动化任务——现在一句话就够了。

这意味着软件工程不能选择"回到纯 API 时代"。正确的策略是：在接受自然语言作为外层接口的前提下，在内部用类型系统、契约和结构化验证来恢复被自然语言牺牲的精确性。

外层模糊，内层精确。用户看到的是自然语言的便利，系统内部运行的是类型安全的管道。这个架构模式是本书后续多个章节的共同基础。
