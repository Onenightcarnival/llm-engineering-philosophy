# 收敛性思维与提示迭代

## 没有收敛判据的迭代是随机游走

计算数学中，迭代法是求解方程的基本策略：从一个初始猜测出发，通过反复应用某个变换逼近真实解。牛顿法求根、梯度下降求最优值、Jacobi 法解线性方程组——这些方法的共同结构是：每一步的输出成为下一步的输入，经过足够多的步骤后，序列趋向目标值。

但迭代法的核心问题从来不是"如何迭代"，而是"迭代是否收敛"。一个迭代过程如果不收敛——序列不趋向任何值，或趋向错误的值——那么再多的迭代都只是在浪费计算资源。因此，计算数学在提出任何迭代法时，必须同时证明其收敛条件，并给出收敛速度的估计。这是数学家的纪律。

Prompt 的迭代优化面临完全相同的结构性问题，却普遍缺少这种纪律。

一个开发者拿到一个 prompt，运行几个测试用例，发现效果不理想，修改措辞，再运行，再修改。这个循环可以持续数天甚至数周。在这个过程中，几乎没有人会问三个数学家一定会问的问题：这个迭代过程是否在收敛？收敛的速度如何？什么条件下应该停止？

## 收敛的精确定义

在计算数学中，迭代序列 {x_n} 收敛到 x* 的定义是：对于任意给定的精度 epsilon > 0，存在 N 使得当 n > N 时，|x_n - x*| < epsilon。直观地说，迭代的结果越来越接近目标，且接近的程度可以任意精确。

将这个定义映射到 prompt 迭代，需要三个对应物：

第一，目标值 x* 对应质量目标。这必须是一个可量化的指标（如准确率 95%、格式符合率 99%），而非模糊的"变好"。[上一篇文章](./02-输出质量的统计度量.md)讨论了如何定义和度量这些指标。

第二，距离 |x_n - x*| 对应当前质量与目标之间的差距。每次迭代后，这个差距应该被计算并记录。

第三，精度 epsilon 对应可接受的质量偏差。达到这个精度范围内就可以停止迭代——不需要追求完美，只需要足够好。

有了这三个要素，就可以将 prompt 迭代从"凭感觉调整"转化为一个有明确终止条件的优化过程。

```python
from dataclasses import dataclass


@dataclass
class ConvergenceTracker:
    target: float  # 质量目标，如 0.95
    epsilon: float  # 可接受偏差，如 0.02
    max_iterations: int  # 最大迭代次数
    history: list[float]  # 每次迭代的质量指标

    def has_converged(self) -> bool:
        """是否已收敛到目标附近。"""
        if not self.history:
            return False
        return abs(self.history[-1] - self.target) <= self.epsilon

    def should_stop(self) -> tuple[bool, str]:
        """判断是否应该停止迭代，并给出原因。"""
        if self.has_converged():
            return True, "已收敛：当前质量在目标范围内"
        if len(self.history) >= self.max_iterations:
            return True, "已达最大迭代次数，未收敛"
        if len(self.history) >= 3 and self.is_stagnant():
            return True, "迭代停滞：连续三次无显著改善"
        if len(self.history) >= 3 and self.is_diverging():
            return True, "迭代发散：质量在恶化"
        return False, "继续迭代"

    def is_stagnant(self, window: int = 3, threshold: float = 0.005) -> bool:
        """最近 window 次迭代的改善幅度是否低于阈值。"""
        if len(self.history) < window:
            return False
        recent = self.history[-window:]
        return max(recent) - min(recent) < threshold

    def is_diverging(self, window: int = 3) -> bool:
        """质量是否在持续恶化。"""
        if len(self.history) < window:
            return False
        recent = self.history[-window:]
        return all(
            recent[i] > recent[i + 1]  # 假设越低越差
            for i in range(len(recent) - 1)
        )
```

## 收敛速度与迭代效率

收敛与否只是第一个问题。第二个问题是收敛速度。

计算数学根据收敛速度将迭代法分级：线性收敛意味着每步将误差缩小一个固定比例，超线性收敛意味着误差缩小的速度越来越快，二次收敛（如牛顿法在良好条件下）意味着每步将有效精度翻倍。

Prompt 迭代的收敛速度取决于迭代策略的质量。盲目的措辞调整——换一个同义词、调整句子顺序——通常只能带来微弱的改善，对应的是缓慢的线性收敛甚至停滞。结构性的修改——改变 prompt 的整体框架、引入新的约束维度、重新定义任务分解方式——可能带来跳跃式的改善，对应的是超线性收敛。

这里的工程启示是明确的：如果连续几轮迭代的改善幅度在递减，继续在同一层面上微调是低效的。应该升级到更高的抽象层面——从措辞调整升级到结构重构，从 prompt 优化升级到架构调整。[第二章](../02-不确定性与决策/01-战略大于分析.md)讨论的 Strategy 大于 Analysis 原则在此有精确的数学对应：战略层面的调整（改变迭代算法）比战术层面的调整（在同一算法内多迭代几步）能带来更大的收敛速度提升。

## 发散条件：何时应该放弃

迭代法不仅可能不收敛，还可能发散——离目标越来越远。计算数学中，识别发散条件并及时终止计算，是数值方法的基本素养。

Prompt 迭代的发散有两种典型形态。

第一种是震荡发散：质量指标在两个值之间来回跳动，没有趋向任何稳定值。这通常意味着迭代方向是矛盾的——每次修改解决了一个问题但引入了另一个问题。比如为了提高输出的详细程度而增加了指令，结果降低了格式符合率；为了恢复格式符合率而增加了格式约束，结果又压缩了输出的详细程度。

第二种是缓慢恶化：每次迭代都在某个维度上略微变差，但由于变化缓慢而未被察觉。这通常发生在 prompt 的复杂度不断膨胀的过程中——每次迭代加入新的指令、新的约束、新的示例，prompt 越来越长，模型的注意力被稀释，整体质量反而下降。

识别发散的关键是量化监控。如果每次迭代后都计算并记录质量指标，发散模式可以在早期被检测到。如果只靠肉眼判断，发散往往在造成显著质量损失后才被察觉。

## 何时停止迭代

综合以上分析，prompt 迭代的停止条件有四种：

一、收敛停止：质量指标进入目标范围（|当前值 - 目标值| < epsilon），目标已达成。

二、停滞停止：连续 k 次迭代的改善幅度低于阈值，继续迭代的边际收益不足以证明成本投入。此时应该评估是否需要升级到更高层面的改变——从 prompt 优化升级到架构调整，或重新审视任务定义本身。

三、发散停止：质量指标出现恶化趋势，当前的迭代方向有问题。应该回滚到最优历史版本，然后尝试不同的优化方向。

四、预算停止：达到预设的最大迭代次数或时间预算。这是一个工程约束，防止在一个可能无解的问题上无限投入。

这四种停止条件不是互斥的。一个健全的迭代框架应该同时监控所有四种条件，以最先触发的那个为准。

## 局限性

收敛性思维的映射有一个根本性的简化：数学中的迭代法操作在连续的度量空间上，而 prompt 的修改是离散的、非结构化的。数学中可以精确定义"迭代方向"和"步长"，在 prompt 空间中这些概念只能近似地对应。这意味着数学中关于收敛速度的精确定理不能直接套用到 prompt 优化中，但收敛性思维作为一种结构化的分析框架——是否在接近目标、何时该停止、何时该换方向——仍然具有实用价值。

此外，prompt 优化的"目标函数"本身可能是非平稳的：模型提供商的静默更新可能导致同一个 prompt 在不同时间点的表现不同。这相当于在迭代过程中目标值本身在移动——一个计算数学中也存在但更棘手的问题。
