# 输出质量的统计度量

## "看起来不错"不是质量标准

LLM 应用开发中最普遍的质量评估方法是：开发者运行几个测试用例，肉眼扫一遍输出，觉得"看起来不错"，然后部署。这个流程的问题不是它不严谨——不严谨只是表象——而是它在认识论层面就是错的：人类对文本质量的主观判断具有高方差、低一致性和系统性偏差，这三个特征恰好是最差的度量工具应有的全部缺陷。

高方差：同一个人在不同时间对同一段文本的评价可能不同。低一致性：不同的人对同一段文本的评价往往不同。系统性偏差：人倾向于高估流畅但错误的输出（因为它"读起来通顺"），低估笨拙但正确的输出（因为它"读起来别扭"）。

[第二章](../02-不确定性与决策/00-概述.md)确立了一个基本原则：不确定性是约束条件，不是敌人。但要管理不确定性，首先需要量化它。无法度量的东西无法管理，而"看起来不错"恰恰是拒绝度量。

## 定义可度量的质量维度

质量是一个多维概念，不同任务关注的维度不同。将模糊的"质量"分解为可度量的维度，是统计度量的第一步。

对于大多数 LLM 应用场景，质量维度可以归纳为五个基本类别：正确性（输出在事实层面是否正确）、完整性（输出是否覆盖了所有要求的内容）、格式符合度（输出是否遵循了指定的结构和格式约束）、一致性（多次运行对同一输入的输出是否稳定）、以及相关性（输出是否与输入的意图对齐）。

每个维度都需要被操作化为可计算的指标。以格式符合度为例：

```python
import json
from pydantic import BaseModel, ValidationError


def format_compliance_rate(
    outputs: list[str],
    schema: type[BaseModel],
) -> dict[str, float]:
    """计算一批输出的格式符合率和各字段的填充率。"""
    total = len(outputs)
    valid_count = 0
    field_fill_counts: dict[str, int] = {
        f: 0 for f in schema.model_fields
    }

    for output in outputs:
        try:
            parsed = schema.model_validate_json(output)
            valid_count += 1
            for field_name in schema.model_fields:
                if getattr(parsed, field_name) is not None:
                    field_fill_counts[field_name] += 1
        except (ValidationError, json.JSONDecodeError):
            continue

    return {
        "overall_compliance": valid_count / total,
        **{
            f"field_{name}_fill_rate": count / total
            for name, count in field_fill_counts.items()
        },
    }
```

格式符合度是最容易度量的维度，因为它有明确的二元判据：要么符合 schema，要么不符合。正确性和相关性的度量要困难得多，往往需要领域专家标注或 LLM-as-judge 方法——但即便使用不完美的度量，也好过没有度量。

## 建立评估基线

度量指标定义后，下一步是建立基线（baseline）。基线是一个锚点：所有后续的改进都相对于基线来衡量。没有基线的优化是盲目的——你不知道自己是在进步还是在原地踏步。

基线的构建遵循三个原则。

第一，基线必须基于固定的评估数据集。评估数据集一旦确定，在一个评估周期内不应变更。如果每次评估都用不同的数据，指标的波动就无法归因——是 prompt 变了，还是数据变了？这和[上一篇文章](./01-提示的版本控制与实验管理.md)讨论的控制变量原则一脉相承。

第二，基线必须包含多次运行的统计量。由于 LLM 的输出具有随机性（即便 temperature 设为 0，不同时间点的 API 调用也可能产生不同结果），单次运行的指标不能代表系统的真实水平。基线应该记录均值和标准差，而非单一数值。

第三，基线必须记录环境信息。模型版本、API 端点、运行时间——这些因素都可能影响输出质量。一个经验教训是：模型提供商的静默更新（同一个 API 端点背后的模型权重被更新）可能导致基线失效，而你完全不知情。

```python
import numpy as np
from dataclasses import dataclass


@dataclass
class EvaluationBaseline:
    metric_name: str
    mean: float
    std: float
    n_runs: int
    confidence_interval_95: tuple[float, float]
    model_version: str
    eval_dataset_version: str
    created_at: str

    @classmethod
    def from_scores(
        cls,
        metric_name: str,
        scores: list[float],
        model_version: str,
        eval_dataset_version: str,
        created_at: str,
    ) -> "EvaluationBaseline":
        arr = np.array(scores)
        mean = float(arr.mean())
        std = float(arr.std(ddof=1))
        n = len(scores)
        se = std / np.sqrt(n)
        ci_95 = (mean - 1.96 * se, mean + 1.96 * se)
        return cls(
            metric_name=metric_name,
            mean=mean,
            std=std,
            n_runs=n,
            confidence_interval_95=ci_95,
            model_version=model_version,
            eval_dataset_version=eval_dataset_version,
            created_at=created_at,
        )
```

## 统计显著性：区分信号与噪声

prompt 从 v1 改到 v2，准确率从 85% 变成 88%。这是真实的改进，还是随机波动？如果不做统计检验，无从判断。

统计显著性检验的核心逻辑是：假设两个版本没有真实差异（零假设），观察到的指标差异纯粹由随机性导致的概率有多大？如果这个概率（p 值）足够小，我们就有理由拒绝零假设，认为差异是真实的。

对于 LLM 输出质量的对比，最常用的检验方法取决于指标类型。对于二元指标（通过/不通过），使用 McNemar 检验——它专门用于配对的二元分类比较，比独立样本的卡方检验更合适，因为两个 prompt 版本是在同一批输入上评估的。对于连续指标（评分），使用配对 t 检验或 Wilcoxon 符号秩检验。

```python
from scipy import stats


def compare_binary_metrics(
    baseline_results: list[bool],
    variant_results: list[bool],
    alpha: float = 0.05,
) -> dict:
    """McNemar 检验：比较两个 prompt 版本在同一数据集上的通过率差异。"""
    assert len(baseline_results) == len(variant_results)
    # 构建 2x2 列联表
    b_pass_v_fail = sum(
        b and not v
        for b, v in zip(baseline_results, variant_results)
    )
    b_fail_v_pass = sum(
        not b and v
        for b, v in zip(baseline_results, variant_results)
    )
    result = stats.binomtest(
        b_fail_v_pass,
        b_fail_v_pass + b_pass_v_fail,
        p=0.5,
    )
    return {
        "p_value": result.pvalue,
        "significant": result.pvalue < alpha,
        "baseline_pass_rate": sum(baseline_results) / len(baseline_results),
        "variant_pass_rate": sum(variant_results) / len(variant_results),
    }
```

一个关键的注意事项：当评估数据集较小（少于 100 条）时，统计检验的统计功效（power）很低，即使存在真实差异也可能检测不到。这不是方法的失败，而是样本量不足的必然后果。增大样本量是提高检验灵敏度的最直接途径，但也意味着更多的 API 成本。

## 常见陷阱

**陷阱一：用平均分掩盖分布信息。** 准确率 85% 可能意味着"所有用例都在 80%-90% 之间"，也可能意味着"一半用例 100%，另一半 70%"。两种分布对应完全不同的问题和对策。始终报告分布的形状（直方图、分位数），而不仅是均值。

**陷阱二：在同一数据集上反复评估导致过拟合。** 如果根据评估结果调整 prompt，然后在同一数据集上再次评估，这个循环会导致 prompt 过拟合到评估集。解决方案和 ML 中一样：将数据分为开发集和保留测试集，开发集用于迭代，保留测试集只在最终决策时使用一次。

**陷阱三：忽视评估指标与业务目标的对齐。** 技术指标（准确率、BLEU 分数）和业务指标（用户满意度、转化率）之间不一定存在线性关系。一个准确率从 90% 提升到 95% 的改进，如果集中在用户不关心的边缘场景，业务价值可能为零。

## 局限性

统计度量方法的根本局限在于：它只能度量可量化的维度。LLM 输出的某些质量属性——比如语气是否得体、逻辑是否连贯、表达是否有洞察力——很难被还原为数值指标。对这些维度的评估，目前仍然依赖人工判断或 LLM-as-judge 方法，而后者本身又引入了新的不确定性层。

此外，统计显著性不等于实际显著性。一个 p < 0.01 的差异如果效果量（effect size）很小，在实际应用中可能毫无意义。始终同时报告显著性和效果量，避免被小 p 值误导。
