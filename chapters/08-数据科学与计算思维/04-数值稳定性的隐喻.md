# 数值稳定性的隐喻

## 等价的计算，不等价的结果

计算数学教给工程师的一个核心教训是：理论上等价的两个计算路径，在有限精度的浮点运算下可能产生截然不同的结果。

经典案例是求解一元二次方程 ax^2 + bx + c = 0。求根公式给出 x = (-b +/- sqrt(b^2 - 4ac)) / (2a)。当 b > 0 且 b^2 >> 4ac 时，计算 (-b + sqrt(b^2 - 4ac)) 涉及两个接近的大数相减，这在浮点运算中会导致灾难性抵消（catastrophic cancellation）——有效精度从 16 位可能骤降到 2-3 位。数学上等价的替代公式 x = 2c / (-b - sqrt(b^2 - 4ac)) 避免了大数相减，在相同条件下能保持完整精度。

两个公式数学上完全等价。差异不在数学，在计算。浮点数的有限精度就是那个将"理论等价"打破为"实践不等价"的约束条件。

LLM 的 prompt 设计存在结构上同构的现象。

## Prompt 中的"有限精度"

LLM 的"有限精度"是一系列比浮点精度更复杂的约束：注意力窗口的有限容量、上下文信息的位置敏感性、指令与内容之间的干扰效应、以及模型在不同表述模式下的行为差异。这些约束共同构成了一个类似于浮点精度的限制——语义上等价的两种 prompt 表述，在通过这些约束时可能产生质量上的显著差异。

考虑一个简单的分类任务：将客户反馈分为"正面"、"负面"和"中性"。以下两种 prompt 在语义上是等价的：

表述 A：直接指令式。

```
将以下客户反馈分类为正面、负面或中性。只输出分类结果。

反馈：{text}
```

表述 B：角色设定加结构化输出。

```
你是一个客户反馈分析专家。你的任务是对客户反馈进行情感分类。

分类标准：
- 正面：客户表达满意、赞扬或积极情绪
- 负面：客户表达不满、投诉或消极情绪
- 中性：客户陈述事实，无明显情感倾向

请对以下反馈进行分类，只输出"正面"、"负面"或"中性"。

反馈：{text}
```

两种表述要求做的事情完全相同。但在实际运行中，表述 B 通常会在边界案例上表现更好——那些情感倾向不明显、需要更细致判断的输入。原因不是表述 B 包含了更多"信息"（分类标准的定义是常识性的，模型早已知道），而是表述 B 的结构特征——明确的判断维度、分条列举的格式——更好地激活了模型的相关能力。

这和数值稳定性的逻辑完全一致：差异不在"内容"（数学等价意味着内容相同），而在"路径"（计算过程如何组织）。

## 识别"数值不稳定"的 Prompt 模式

在计算数学中，数值不稳定性有几种经典的来源：大数相减、小数做除数、误差在迭代中累积放大。对应到 prompt 设计，可以识别出若干类似的不稳定模式。

**模式一：过度压缩的指令。** 将多个复杂要求压缩在一句话中，类似于将多步计算压缩为一个复合表达式。数值计算中，复合表达式的中间结果不可控，容易触发精度损失；prompt 中，过度压缩的指令让模型难以分解任务，容易遗漏要求。

不稳定版本：

```
提取文本中的日期、人名和金额，按时间排序后以 JSON 格式输出，忽略不完整的记录。
```

稳定版本将任务显式分解：

```
请按以下步骤处理文本：

第一步：提取所有出现的日期、人名和金额。
第二步：检查每条记录是否同时包含日期、人名和金额。丢弃不完整的记录。
第三步：将完整记录按日期从早到晚排序。
第四步：以 JSON 数组格式输出，每个元素包含 date、name、amount 三个字段。
```

两个版本在语义上等价——要求完全相同。但分步版本的输出质量显著更稳定，尤其在输入文本较长或包含噪声信息时。这与计算数学中"避免复合表达式、显式保留中间结果"的稳定性策略直接对应。

**模式二：隐式否定。** 告诉模型"不要做什么"而非"做什么"。这类似于数值计算中依赖隐式条件而非显式约束——隐式条件在边界情况下容易被违反。

不稳定：

```
总结这篇文章。不要包含作者观点，不要超过200字，不要使用专业术语。
```

稳定：

```
总结这篇文章的事实内容，使用通俗语言，控制在200字以内。
```

否定指令（"不要做X"）对模型的约束力弱于肯定指令（"做Y"），这是 LLM 的一个已知行为特征。从数值稳定性的视角看，否定指令是一种"间接计算路径"——它要求模型先理解X是什么，然后避开X，最后生成满足其他条件的输出。每一步间接性都引入误差放大的机会。

**模式三：歧义性锚点。** prompt 中包含可以被多种方式解释的表述，类似于数值计算中的条件数（condition number）过大——输入的微小变化导致输出的巨大变化。

当 prompt 包含"适当地"、"合理地"、"简要地"等模糊修饰语时，不同的输入文本可能触发模型对这些修饰语的不同解读，导致输出质量的高方差。用精确的量化约束（"200字以内"、"3个要点"、"只包含数值数据"）替换模糊修饰语，就像降低计算过程的条件数——减小了输入扰动对输出的放大效应。

## 稳定性检测方法

识别 prompt 的"数值不稳定性"有一个朴素但有效的方法：对同一个 prompt 运行多次（相同输入、相同参数），观察输出的方差。如果方差异常高，说明 prompt 处于不稳定状态——模型在多种同样"合理"的输出之间犹豫不决。

```python
def stability_score(
    outputs: list[str],
    similarity_fn: callable,
) -> float:
    """计算一组输出之间的平均配对相似度，作为稳定性度量。

    similarity_fn 接受两个字符串，返回 [0, 1] 之间的相似度。
    返回值越接近 1，稳定性越高。
    """
    n = len(outputs)
    if n < 2:
        return 1.0
    total_sim = 0.0
    pairs = 0
    for i in range(n):
        for j in range(i + 1, n):
            total_sim += similarity_fn(outputs[i], outputs[j])
            pairs += 1
    return total_sim / pairs
```

稳定性分数低的 prompt 就像条件数大的矩阵——即使当前的输出"看起来不错"，也不意味着在新的输入上能保持同样的质量。将稳定性检测纳入[第一篇文章](./01-提示的版本控制与实验管理.md)讨论的实验管理流程，可以在 prompt 变更上线前发现潜在的不稳定性问题。

## 局限性

数值稳定性隐喻的核心局限在于：浮点计算的不稳定性有严格的数学分析工具（误差传播理论、条件数分析），而 prompt 的"不稳定性"目前只能通过经验性的观察和统计度量来评估。我们可以检测到不稳定现象，但无法像计算数学那样从理论上预测哪些 prompt 结构会不稳定、不稳定的程度有多大。

这意味着本文提供的不稳定模式识别（过度压缩、隐式否定、歧义性锚点）是经验规律而非数学定理。它们在大多数情况下成立，但不保证普遍性。随着对 LLM 内部机制的理解加深，未来可能发展出更严格的"prompt 稳定性理论"——但在那之前，经验性的启发式方法是我们最好的工具。
