# 提示的版本控制与实验管理

## 被忽视的双重身份

Prompt 的工程管理之所以混乱，根源在于一个未被正视的事实：prompt 同时是代码和配置。

说它是代码，因为 prompt 的变更直接影响系统行为，其影响范围和修改一个核心函数没有本质区别。说它是配置，因为 prompt 的变更频率远高于代码，往往由非工程角色（产品经理、领域专家）发起，且同一套系统可能需要在不同场景下使用不同的 prompt 变体。

这种双重身份意味着，prompt 既不能用纯粹的代码管理方式（git commit、code review、CI/CD），也不能用纯粹的配置管理方式（配置中心、热更新、灰度发布）。它需要两者的结合——而这正是 ML 实验管理多年前已经解决的问题。

在机器学习领域，模型的超参数具有完全相同的双重身份：它们是"代码"（直接决定模型行为），也是"配置"（需要在实验过程中频繁调整和对比）。ML 工程师不会把超参数硬编码在训练脚本里然后用 git 管理变更，也不会把它们扔进一个配置文件然后祈祷不出错。他们用 MLflow、Weights & Biases 这样的实验管理平台，将每一次实验的参数、结果、环境信息完整记录并关联。

Prompt 工程需要同样的纪律。

## Prompt 作为实验对象

将 prompt 管理从"版本控制"升级为"实验管理"，关键的认知转变是：每一次 prompt 的修改都是一次实验，而非一次"改进"。

这个区分不是语义游戏。称之为"改进"预设了方向——新版本一定比旧版本好。称之为"实验"则承认了不确定性——新版本可能更好，也可能更差，需要数据来判断。[第二章](../02-不确定性与决策/00-概述.md)讨论的不确定性管理原则在这里有直接的投射：在结果未经验证之前，任何 prompt 变更都只是假设，不是事实。

一个合格的 prompt 实验记录至少包含以下要素：

```python
from dataclasses import dataclass, field
from datetime import datetime


@dataclass
class PromptExperiment:
    experiment_id: str
    parent_id: str | None  # 基于哪个版本修改
    hypothesis: str  # 这次修改预期解决什么问题
    prompt_content: str
    model: str
    temperature: float
    eval_dataset_version: str  # 评估数据集的版本
    metrics: dict[str, float] = field(default_factory=dict)
    created_at: datetime = field(default_factory=datetime.now)
    author: str = ""
    notes: str = ""
```

`parent_id` 记录的是变更的谱系关系，`hypothesis` 迫使实验者在修改前明确预期，`eval_dataset_version` 确保不同实验之间的评估基准一致。这些字段看似繁琐，但每一个都对应着实际工程中反复出现的问题：不知道当前版本是从哪个版本演化来的，不记得当初为什么做了某个修改，用不同的测试集评估然后得出了不可比的结论。

## 实验设计：控制变量的纪律

ML 实验管理的核心纪律是控制变量：每次实验只改变一个因素，其他因素保持不变。这个纪律在 prompt 迭代中被普遍违反。

典型的反模式是：修改了 prompt 的措辞，同时升级了模型版本，同时调整了 temperature，然后发现输出质量变了——但无法归因到底是哪个变更导致的。这种混淆变量的做法在数据科学中是初学者错误，在 prompt 工程中却是常态。

控制变量的操作化实现：

```python
from dataclasses import dataclass


@dataclass
class ExperimentConfig:
    prompt_template: str
    model: str
    temperature: float
    max_tokens: int
    system_message: str


def validate_single_variable_change(
    baseline: ExperimentConfig,
    variant: ExperimentConfig,
) -> str | None:
    """确保两个配置之间只有一个变量不同。返回变更的字段名，或在多个字段变更时抛出异常。"""
    changes = []
    for f in ExperimentConfig.__dataclass_fields__:
        if getattr(baseline, f) != getattr(variant, f):
            changes.append(f)
    if len(changes) == 0:
        raise ValueError("基线与变体完全相同，不构成有效实验")
    if len(changes) > 1:
        raise ValueError(
            f"同时变更了 {changes}，违反单变量原则。"
            f"请拆分为 {len(changes)} 个独立实验。"
        )
    return changes[0]
```

这段代码的价值不在于它的技术复杂度——它极其简单——而在于它将一个纪律性要求编码为可执行的约束。[第四章](../04-类型系统与契约/00-概述.md)讨论的"约束传播"原则在此适用：好的约束不是限制，是对问题空间的精确建模。

## 变更追踪与效果归因

版本控制回答"改了什么"，实验管理回答"改了之后怎么样"。两者的结合才能回答真正重要的问题："这次修改是否值得保留"。

效果归因的最低标准是：对于每一次被采纳的 prompt 变更，能够说出它在评估集上带来的量化改善，以及这个改善是否具有统计显著性。[下一篇文章](./02-输出质量的统计度量.md)会详细讨论统计显著性的判定方法，这里只强调一个原则：没有量化归因的变更决策，本质上是在掷骰子。

实践中的最小可行方案不需要复杂的平台。一个结构化的 JSON Lines 文件加上一个简单的 Python 脚本就能实现核心功能：记录每次实验的配置和结果，对比任意两个版本的指标差异，按时间线回溯 prompt 的演化路径。关键不是工具的精良程度，而是纪律的执行力度。

对于已经达到一定规模的团队，MLflow 的实验追踪模块可以直接复用于 prompt 管理。prompt 内容作为参数记录，评估指标作为 metrics 记录，prompt 文件作为 artifact 存储。不需要为 prompt 管理专门搭建基础设施——ML 领域十年的工具积累已经足够成熟。

## 局限性

实验管理方法论有一个根本前提：评估函数是稳定且可信的。如果评估本身不可靠——评估标准模糊、评估数据不具代表性、评估指标与业务目标不对齐——那么再精细的实验管理也只是在追踪噪声。这指向[第七章](../07-测试与可靠性/00-概述.md)讨论的可靠性问题：评估系统的可靠性是整个实验管理体系的地基。

另一个局限是成本。严格的控制变量和量化归因意味着更多的 API 调用、更长的评估周期、更高的人力投入。对于探索阶段的原型项目，这种纪律的投入产出比可能不合理。Strategy 大于 Analysis 的原则在此适用：实验管理的精细程度应该匹配项目阶段，早期原型不需要生产级的实验基础设施。
