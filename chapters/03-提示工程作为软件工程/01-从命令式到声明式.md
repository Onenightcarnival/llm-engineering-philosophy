# 从命令式到声明式：软件工程史的一个切面

## 一个反复发生的范式跃迁

从汇编到 C，从 C 到 SQL，从 jQuery 到 React，从 Makefile 到 Terraform——软件工程的历史中有一条反复出现的弧线：从命令式到声明式。每一次跃迁都遵循相同的结构性规律，都在做同一件事——将"怎么做"的负担从开发者转移到系统，让开发者只需要声明"要什么"。

这条弧线在 LLM 应用开发中正在上演又一次重演。从手工编写自然语言 prompt（命令式）到用类型系统和代码结构声明期望输出（声明式），这是软件工程史上同一个结构性模式的最新实例。

理解这个历史模式的价值不在于知识考古，而在于它提供了一组可操作的预测：每一次命令式到声明式的跃迁都经历了相似的阶段，遇到了相似的阻力，最终产生了相似的工程收益。LLM 提示工程的声明化进程正处于这条弧线的早期，看清历史模式有助于做出正确的战略判断。

## 汇编到高级语言：第一次抽象层级提升

最早的程序直接用机器指令编写。汇编语言是对机器指令的薄层封装——用助记符替代操作码，但程序员仍然在精确控制每一步操作：把这个值加载到寄存器 A，把那个值加载到寄存器 B，执行加法，把结果存回内存地址 0x3F00。

C 语言做的事情看起来微不足道——它只是让你写 `a = b + c` 而不是四条汇编指令。但这个看似简单的变化实质上是一次范式跃迁：程序员从"怎么加"变成了"要什么结果"。寄存器分配、指令调度、内存布局——这些"怎么做"的决策被交给了编译器。

这次跃迁中值得注意的结构性特征：

**抽象的代价是可控的。** C 编译器生成的代码不如手写汇编那么极致优化，但差距足够小。对绝大多数场景，这个代价是值得的——开发效率提升了数量级，而运行效率的损失在百分之几以内。

**新的约束反而增加了可靠性。** C 的类型系统（尽管以今天的标准看非常薄弱）引入了约束：`int` 和 `float` 不能随意混用，函数有签名。这些约束在汇编时代不存在——程序员可以往任何地址写任何值。约束看起来是限制，实际上是保护。

**旧范式并没有消失。** 在性能关键路径上，C 程序员仍然可以插入内联汇编。新范式将旧范式压缩到一个更小的、确实需要精细控制的领域中。

## SQL：声明式的典范

SQL 是声明式编程最纯粹的实例之一。

```sql
SELECT name, revenue
FROM companies
WHERE industry = 'Technology' AND revenue > 1000000
ORDER BY revenue DESC
LIMIT 10;
```

这段 SQL 没有告诉数据库任何关于"怎么做"的信息。它没有指定：用哪个索引、按什么顺序扫描表、在内存还是磁盘上排序、是否需要并行执行。所有这些执行细节由查询优化器决定。

SQL 出现之前，程序员用导航式数据库编程：手动遍历数据结构、手动维护指针链、手动实现排序和过滤。从导航式到声明式的跃迁产生了几个关键结果。

**表达力与实现解耦。** 同一条 SQL 语句在不同数据库引擎上有不同的执行计划，但结果相同。这意味着用户可以切换底层引擎而不需要改写查询——实现层面的优化对声明层面是透明的。

**优化的责任转移。** 在命令式范式下，性能优化是程序员的责任——选错了遍历顺序，查询就慢几个数量级。在声明式范式下，优化是系统的责任——查询优化器比绝大多数程序员更擅长选择执行计划。这是一个关键的洞察：将"怎么做"的决策交给专门为此设计的系统，通常比人工决策更优。

**可组合性浮现。** 声明式查询天然可以组合。子查询、视图、CTE——这些组合机制在命令式代码中需要复杂的控制流才能实现，在 SQL 中是语法的自然延伸。

## React：从 DOM 操作到 UI 声明

jQuery 时代的 UI 开发是命令式的：

```javascript
// 命令式：描述每一步操作
$('#button').click(function() {
    var count = parseInt($('#counter').text());
    count++;
    $('#counter').text(count);
    if (count > 10) {
        $('#counter').addClass('warning');
    }
});
```

React 将其转化为声明式：

```javascript
// 声明式：描述 UI 应该是什么样子
function Counter({ count }) {
    return (
        <div className={count > 10 ? 'warning' : ''}>
            {count}
        </div>
    );
}
```

jQuery 代码描述的是"当按钮被点击时怎么修改 DOM"——一系列具体的操作步骤。React 代码描述的是"给定状态，UI 应该是什么样子"——一个从状态到视图的纯函数映射。

这次跃迁揭示了一个重要的模式：**声明式不意味着丧失对细节的控制，而是将控制从"过程"转移到"规格说明"。** React 组件通过 props、状态和类型来声明 UI 的规格，而具体的 DOM 更新由 React 的 reconciliation 算法处理。开发者仍然在做设计决策，但决策的粒度从"怎么操作 DOM 节点"提升到了"怎么定义组件的接口和状态结构"。

## 共同的结构性规律

从这些历史案例中可以提炼出命令式到声明式跃迁的结构性规律：

| 领域 | 命令式形态 | 声明式形态 | 被委托的决策 | 约束语言 | 抽象代价 |
|------|----------|----------|------------|---------|---------|
| 系统编程 | 汇编语言 | C 语言 | 寄存器分配、指令调度 | 类型系统和函数签名 | 生成代码不如手写汇编极致优化 |
| 数据查询 | 导航式数据库编程 | SQL | 索引选择、扫描策略、排序算法 | 关系代数的 SQL 语法 | 查询优化器的决策偶尔不如手写计划 |
| UI 开发 | jQuery DOM 操作 | React 组件 | DOM diff 和增量更新 | 组件 props 和状态的类型定义 | 虚拟 DOM 的额外内存和计算开销 |
| 基础设施 | 手动配置服务器 | Terraform / K8s YAML | 资源创建顺序、依赖解析、状态收敛 | HCL / YAML 的资源声明 | 声明式配置的调试成本较高 |
| LLM 提示工程 | 自然语言 prompt | 类型化结构声明 | 推理步骤执行顺序和输出格式遵循 | Pydantic / JSON Schema / 类型注解 | Schema 的 token 开销 |

五次跃迁的核心动作具有结构性同构：将一类"怎么做"的决策从人转移到系统，同时引入一种新的约束语言来表达"要什么"。从汇编到 C，被委托的是寄存器和指令层面的决策；从导航式到 SQL，被委托的是索引和扫描策略层面的决策；从自然语言 prompt 到类型化声明，被委托的是推理步骤和输出格式层面的决策。形式不同，结构相同。约束语言的精确程度决定了系统的行为可预测性——这就是为什么每一次跃迁都伴随着某种形式化约束的引入：C 的类型系统、SQL 的关系代数语法、React 的 props 类型定义、Pydantic 的 Schema。约束是跃迁得以成立的条件。

## LLM 提示工程中的命令式遗留

当前主流的 prompt 工程实践本质上是命令式的。

一个典型的命令式 prompt：

```
你是一个专业的文档分析师。请按照以下步骤分析给定的文档：

第一步：通读全文，识别文档的主题和类型。
第二步：提取文档中的所有关键实体（人名、组织、日期、金额）。
第三步：分析实体之间的关系。
第四步：基于以上分析，生成一份结构化摘要。

摘要应包含以下部分：
- 文档概述（不超过 3 句话）
- 关键实体列表
- 关系图谱描述
- 结论和建议

请确保输出为 JSON 格式。
```

这段 prompt 存在命令式编程的所有典型问题：

**控制流是隐式的。** "第一步...第二步..."的顺序依赖模型的自然语言理解，没有强制执行机制。模型可能跳过某一步、合并某几步、或者以不同的顺序执行——调用者无从得知。

**输出契约是弱约束。** "请确保输出为 JSON 格式"是一个愿望，不是一个契约。模型可能在 JSON 前后添加解释文字，可能输出不合法的 JSON，可能遗漏某些字段。

**不可组合。** 如果另一个任务只需要"提取关键实体"这一步，无法从这段 prompt 中干净地提取出来复用。整段 prompt 是一个不可分割的整体。

**不可测试。** 无法在不调用 LLM 的情况下验证这段 prompt 的正确性。"步骤是否完整""输出格式是否明确"——这些属性没有形式化的检查手段。

这些问题和汇编语言时代、jQuery 时代面临的问题在结构上是同一类：开发者在用错误的抽象层级工作，把本应由系统处理的细节手动管理。

## 声明式提示工程的本质

声明式提示工程的核心转变是：用类型定义替代过程描述，用结构约束替代自然语言指令。

同样的文档分析任务，声明式的表达：

```python
from pydantic import BaseModel, Field
from typing import Literal


class Entity(BaseModel):
    name: str = Field(description="实体名称")
    entity_type: Literal["person", "organization", "date", "amount", "other"] = Field(
        description="实体类型"
    )


class Relationship(BaseModel):
    subject: str = Field(description="关系的主体")
    predicate: str = Field(description="关系的类型，如 '创立'、'投资'、'位于'")
    object: str = Field(description="关系的客体")


class DocumentAnalysis(BaseModel):
    document_type: Literal[
        "contract", "report", "letter", "memo", "other"
    ] = Field(description="文档类型")

    overview: str = Field(
        max_length=300,
        description="文档核心内容概述"
    )

    entities: list[Entity] = Field(
        min_length=1,
        description="从文档中识别出的所有关键实体"
    )

    relationships: list[Relationship] = Field(
        description="基于上述 entities 中的实体，提取它们之间的关系"
    )

    conclusion: str = Field(
        max_length=500,
        description="综合 overview、entities 和 relationships 的信息，"
                    "给出分析结论和行动建议"
    )
```

这段代码做了和前面那段自然语言 prompt 相同的事情，但方式根本不同。

**控制流由结构隐式定义。** [第一章](../01-认识论/autoregressive-generation.md)已经解释了自回归生成的序列性——模型按字段定义顺序生成内容。`overview` 在 `entities` 之前生成，`relationships` 在 `entities` 之后生成（且其描述明确引用了"上述 entities"）。推理路径通过字段顺序和依赖声明来确定，而不是通过"第一步...第二步..."的自然语言指令。

**输出契约是强约束。** `document_type` 必须是五个枚举值之一，`overview` 不超过 300 字符，`entities` 至少有 1 个元素。这些约束是可在运行时验证的硬契约。验证失败意味着输出被拒绝并触发重试，而不是默默接受一个不合规的结果。

**天然可组合。** `Entity` 和 `Relationship` 是独立的类型定义，可以在其他任务中直接复用。如果另一个任务只需要实体提取，直接使用 `Entity` 类型即可，不需要从一段长 prompt 中拆解。

**可独立测试。** 类型定义本身就是可执行的规格说明：

```python
from pydantic import ValidationError


def test_entity_type_constraint():
    """验证实体类型枚举约束。"""
    Entity(name="张三", entity_type="person")  # 合法

    try:
        Entity(name="张三", entity_type="animal")  # 不合法
        assert False, "应该抛出 ValidationError"
    except ValidationError:
        pass  # 预期行为


def test_document_analysis_requires_entities():
    """验证文档分析至少需要一个实体。"""
    try:
        DocumentAnalysis(
            document_type="report",
            overview="一份报告",
            entities=[],  # 违反 min_length=1
            relationships=[],
            conclusion="无",
        )
        assert False, "应该抛出 ValidationError"
    except ValidationError:
        pass
```

这些测试在不调用任何 LLM 的情况下验证了约束逻辑的正确性。这和 SQL 的 schema 验证是同一个思路——约束定义本身是可以独立于执行引擎来验证的。

## 为什么不是所有人都接受这个跃迁

每一次命令式到声明式的跃迁都遭遇了类似的阻力。

**"我能用命令式做得更好。"** 汇编程序员认为 C 编译器生成的代码太慢。DBA 认为手写的查询计划比优化器的更好。jQuery 开发者认为直接操作 DOM 比虚拟 DOM 更高效。prompt 工程师认为精心调优的自然语言 prompt 比结构化声明更灵活。

这个观点在某些情况下是正确的——在性能关键路径上，手写汇编确实比 C 更快。但它忽略了一个关键事实：绝大多数情况下，系统的瓶颈不在这里。命令式的精细控制能力是一种稀缺资源——它需要高技能的个人、大量的调试时间、以及对底层机制的深入理解。声明式范式的意义不在于在极端情况下比命令式更好，而在于它将"可接受的结果"从少数专家扩展到了多数工程师。

**"声明式太受限了。"** 确实，声明式范式在表达力上通常弱于命令式。SQL 无法表达所有可能的数据操作，React 组件无法表达所有可能的 UI 行为，Pydantic 模型无法表达所有可能的 prompt 意图。

但这恰恰是声明式范式的设计意图——通过缩小表达空间来增加可预测性。[第二章](../02-不确定性与决策/uncertainty-as-constraint.md)讨论的"约束作为工程工具"在这里有直接的体现：约束的本质是对问题空间的精确建模。一个不能表达所有可能性的约束语言，如果能精确表达实际需要的所有可能性，就是一个好的约束语言。

**"学习成本太高了。"** 每一次跃迁都要求开发者学习一种新的思维方式。从过程性思维到声明式思维的转换远超学一个新语法——它要求开发者改变思考问题的方式：不再想"怎么一步步实现"，而是想"结果应该满足什么约束"。

这个学习成本是真实的，但它是一次性的。一旦思维方式转换完成，新范式带来的生产力提升是持续的。

## 命令式的保留地

声明式范式不是要消灭命令式。在 LLM 提示工程中，有些场景仍然适合命令式的方式：

**开放式创意任务。** "写一首关于秋天的诗"这类任务，输出空间本身就是开放的，用类型系统约束它会扼杀创造性。这和在性能关键路径上使用内联汇编是同一个逻辑：当声明式的抽象代价超过了它的收益，就回到命令式。

**探索性原型阶段。** 在尚不确定任务结构时，自然语言 prompt 是快速探索的好工具——它的灵活性在这个阶段是优势而非劣势。但一旦任务结构稳定下来，就应该将 prompt 升级为声明式的类型定义。这和用脚本快速原型验证、然后用强类型语言重写生产代码是同一个工程节奏。

**与用户的直接对话。** 聊天机器人的对话本身不适合强类型约束——用户的输入是自由的自然语言，输出也应该是自然的回应。但这不意味着对话系统的内部不能是声明式的：意图识别、实体提取、对话状态管理——这些内部组件完全可以用声明式方式构建，只有最终面向用户的那一层保持自然语言的灵活性。

选择命令式还是声明式是一个可以用输出空间的结构特征来量化的工程判断：

| 任务类型 | 输出空间 | 结构稳定性 | 推荐范式 | 理由 |
|---------|---------|----------|---------|-----|
| 结构化数据提取 | 有界 | 稳定 | 声明式 | 输出结构明确，字段和类型已知，声明式约束可以最大化可靠性 |
| 创意写作 | 开放 | 未知 | 命令式 | 输出空间本质上是开放的，类型约束会扼杀创造性 |
| 客服意图分类 | 有界 | 稳定 | 声明式 | 输出是有限枚举集合，Literal 类型是天然的表达 |
| 代码审查 | 有界 | 演化中 | 混合 | 问题列表的结构是确定的（声明式），但每个问题的描述需要自由表达（命令式） |

判断的核心变量是输出空间的可枚举性和结构的稳定程度。输出空间有界且结构稳定时，声明式范式的优势最大；输出空间开放且结构未知时，命令式范式的灵活性不可替代；介于两者之间时，混合范式是务实的选择。

## 跃迁的时机判断

回到[第二章](../02-不确定性与决策/strategy-over-analysis.md)的核心命题：Strategy 大于 Analysis。选择命令式还是声明式是一个战略决策。

用命令式 prompt 快速验证想法是正确的——原型阶段不应该被类型定义的开销拖慢。但一旦原型验证通过、任务结构确定下来，继续使用命令式 prompt 进入生产就是一个战略错误——它意味着选择了一个可维护性低、可测试性差、可靠性不可预测的技术路线。

历史模式提供了一个清晰的信号：每一次命令式到声明式的跃迁都是不可逆的。没有人在 React 之后回到 jQuery 做大规模 UI 开发，没有人在 SQL 之后回到导航式数据库编程。跃迁的方向是确定的，不确定的只是时间节点。

对 LLM 提示工程来说，这个时间节点取决于两个变量：模型理解结构化约束的能力（供给侧）和工程师对声明式范式的认知（需求侧）。前者正在被 Function Calling、Structured Output、JSON Mode 等机制快速推进。后者——这正是本章试图推动的。
