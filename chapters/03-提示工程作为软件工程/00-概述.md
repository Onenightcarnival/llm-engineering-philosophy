# 第三章 提示工程作为软件工程

## 核心问题

提示工程不应该是一门手艺。它应该是软件工程的一个分支，遵循软件工程的基本原则，同时发展出适应新范式的特有方法。

当 prompt 从原型阶段的临时脚本变成生产系统的核心组件，它就必须像代码一样被对待：版本控制、回归测试、模块化设计、可维护性保障。区别在于 prompt 没有编译器帮你兜底——这个缺失不是放弃工程纪律的理由，而是需要更强纪律的原因。

## 文章

- [从命令式到声明式：软件工程史的一个切面](01-从命令式到声明式.md) -- 从汇编到 C，从 jQuery 到 React，从自然语言 prompt 到 Code as Prompt——每次抽象层级的提升都遵循相同的结构性规律。这个转变不是新事物，而是软件工程史上命令式到声明式跃迁的最新实例。理解历史模式有助于做出正确的战略判断。
- [提示的可维护性问题](02-提示的可维护性.md) -- 当 prompt 成为生产系统的组成部分，它就面临所有代码面临的问题：版本管理、回归测试、重构风险、技术债务。区别在于 prompt 没有编译器、类型检查器和 IDE 的支撑。声明式方法将 prompt 的可维护性问题转化为代码的可维护性问题——而后者已经有了成熟的解决方案。
- [提示与代码的同构性](03-提示与代码的同构性.md) -- 一段好的 prompt 和一段好的代码在结构上共享深层的设计原则：单一职责、关注点分离、最小惊讶、显式优于隐式。这些原则不是"编程原则"，而是"规格说明原则"——适用于所有向执行者传达意图的精确沟通。同构性的边界同样重要：确定性 vs 概率性、静态分析的极限、组合的不可预测性。
- [声明式思维链](04-声明式思维链.md) -- Code as Prompt 的技术实现。通过 Pydantic 类型系统和声明式约束，将提示工程从手艺提升为工程。结构定义即推理路径、类型系统即约束传播、嵌套结构即递归推理——技术细节的完整展开。

## 本章的推演逻辑

四篇文章构成一条从历史规律到工程原则再到技术实现的推演链：

第一篇从软件工程史的宏观视角出发，论证"命令式到声明式"的跃迁是一个反复发生的结构性规律，LLM 提示工程正处于这条弧线的早期。第二篇将视角收缩到工程实践层面，展开 prompt 作为生产制品面临的可维护性挑战——版本管理、回归测试、技术债务——以及声明式方法如何将这些挑战转化为已有解决方案的问题。第三篇在更深的层面论证 prompt 与代码的结构性同构，以及这种同构性意味着软件工程的设计原则可以直接迁移到 prompt 设计中。第四篇是 Code as Prompt 理念的技术实现——Declarative Chain-of-Thought 的完整技术细节。

本章的核心主张——提示工程是软件工程的一个分支——是[第四章](../04-类型系统与契约/00-概述.md)的直接前提。如果 prompt 应该是声明式的，那么类型系统就是最好的声明工具。本章建立的"Code as Prompt"原则在第四章中通过 Pydantic、JSON Schema 和约束传播机制得到系统性的展开。
