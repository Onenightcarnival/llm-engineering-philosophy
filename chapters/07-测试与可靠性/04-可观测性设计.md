# 可观测性设计

## 为什么 LLM 应用需要更强的可观测性

传统软件的可观测性主要关注性能（延迟、吞吐量）和健康度（错误率、可用性）。LLM 应用在这两个维度之外，还有一个传统系统不存在的维度：**输出质量**。

一个 HTTP 200 的响应在传统系统中意味着成功。在 LLM 应用中，HTTP 200 只意味着 API 调用没有失败——LLM 返回了格式正确的 JSON，但内容可能是幻觉、可能答非所问、可能忽略了关键约束。你需要可观测性来发现这些"成功的失败"。

## 三大支柱的具体化

可观测性的三大支柱——日志、追踪、指标——在 LLM 应用中有特殊的实现要求。

**日志：记录什么。** LLM 应用的日志应该记录完整的调用上下文：输入的 prompt（包括 system prompt 和用户消息）、LLM 的原始响应、解析后的结构化结果、验证是否通过。这些信息是事后分析质量问题的唯一依据。

```python
import logging
import json

logger = logging.getLogger("llm")

def call_llm_with_logging(messages, model_class):
    response = client.chat.completions.create(
        model="gpt-4",
        messages=messages,
        response_format={"type": "json_object"}
    )

    raw_output = response.choices[0].message.content

    logger.info(json.dumps({
        "event": "llm_call",
        "model": "gpt-4",
        "input_tokens": response.usage.prompt_tokens,
        "output_tokens": response.usage.completion_tokens,
        "messages_count": len(messages),
        "raw_output_preview": raw_output[:500],
        "latency_ms": response.response_ms,
    }))

    try:
        result = model_class.model_validate_json(raw_output)
        logger.info(json.dumps({"event": "parse_success", "model_class": model_class.__name__}))
        return result
    except Exception as e:
        logger.warning(json.dumps({"event": "parse_failure", "error": str(e)}))
        raise
```

注意隐私考量：完整的 prompt 和响应可能包含用户的敏感数据。日志的存储、访问控制和保留策略需要与数据隐私要求对齐。

**追踪：端到端的因果链。** 在多步工作流中，一次用户请求可能触发多次 LLM 调用、多次工具调用和多次数据库查询。分布式追踪（trace）将这些操作串联为一条因果链，每个操作是一个 span。

追踪在 LLM 应用中的特殊价值是**推理过程的可回溯性**。当最终输出有问题时，追踪帮助你定位问题发生在哪个步骤——是检索到了无关文档（RAG 步骤），还是 LLM 忽略了关键信息（生成步骤），还是工具返回了错误数据（工具调用步骤）。

**指标：量化质量。** 除了标准的运维指标（延迟 P50/P99、错误率、吞吐量），LLM 应用需要额外的质量指标：

- 结构合规率：LLM 输出通过 Schema 验证的比例。
- 重试率：需要重试才能获得合规输出的调用比例。
- Token 消耗：输入和输出 token 的分布，直接关联成本。
- 降级率：触发降级逻辑的调用比例。

## Prompt 版本与输出质量的关联

可观测性的一个高级应用是将 prompt 版本与输出质量关联。当你修改了一个 prompt 并部署到生产环境，可观测性系统应该能回答：这次修改是否改善了输出质量？

实现方式：在每次 LLM 调用的日志中记录 prompt 的版本标识。在指标系统中按版本分组统计质量指标。对比新旧版本的指标分布，判断变更的效果。

这将可观测性从"发现问题"提升到"指导优化"——不是被动地等待问题出现，而是主动地量化每次变更的影响。

## 成本可观测性

Token 消耗是 LLM 应用的主要运营成本。成本可观测性意味着能够回答：

- 每个功能模块消耗了多少 token？
- 成本的时间趋势是什么？是否有异常增长？
- 哪些用户或请求类型是成本的主要贡献者？

实现方式很直接：每次 LLM 调用后，将 token 消耗记录为按功能模块和模型标注的计数器指标。有了这些数据，就可以按模块归因成本、检测异常增长、识别高消耗的请求类型。

成本可观测性不是可选的。一个没有成本监控的 LLM 应用，就像一个没有限速的 API——迟早会因为意外的流量或使用模式导致账单失控。
