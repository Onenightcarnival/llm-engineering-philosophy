# 结构测试与语义测试

## 两个层面的正确性

LLM 输出的"正确"有两个独立的层面。

**结构正确**：输出符合预期的格式——JSON 可解析、字段齐全、类型匹配、值在合法范围内。结构正确性是第四章讨论的类型系统和验证器的职责，可以被自动化地、确定性地检验。

**语义正确**：输出的含义符合预期——摘要准确反映了原文、分类结果符合业务定义、推理过程逻辑自洽。语义正确性无法用简单的程序逻辑检验，因为它涉及"意思是否对"的判断。

一个输出可以结构正确但语义错误（JSON 格式完美，但情感分类把正面评价分类为负面）。也可以结构错误但语义正确（回答的内容正确，但不是预期的 JSON 格式）。两个层面的测试缺一不可，且需要不同的策略。

## 结构测试的自动化

结构测试是 LLM 应用测试中最容易也最应该自动化的部分。

```python
import pytest
from pydantic import ValidationError

def test_output_structure():
    """验证 LLM 输出是否符合预定义的 Schema"""
    response = call_llm(test_prompt)

    # 结构验证：Pydantic 解析即测试
    try:
        result = ExpectedModel.model_validate_json(response)
    except ValidationError as e:
        pytest.fail(f"输出结构不符合 Schema: {e}")

    # 字段完整性
    assert result.summary is not None
    assert len(result.key_points) > 0

    # 值域约束
    assert 0 <= result.confidence <= 1
    assert result.category in VALID_CATEGORIES
```

结构测试的策略直接继承传统软件测试的方法论：

- **正向测试**：给定典型输入，验证输出符合 Schema。
- **边界测试**：给定极端输入（超长文本、空输入、特殊字符），验证系统不会崩溃。
- **回归测试**：Schema 变更后，验证旧版本的典型输出仍能被新 Schema 兼容解析。

结构测试的特殊之处在于非确定性：同一个测试用例多次运行可能产生不同的输出。应对策略是多次运行取通过率——如果 100 次运行中 95 次通过结构验证，结构合规率是 95%。这个数字本身就是一个有价值的质量指标。

## 语义测试的策略

语义测试的核心困难在于：没有编译器可以告诉你"这个摘要是否准确"。但这不意味着语义测试无法工程化。以下策略可以在不同程度上覆盖语义正确性。

**基于规则的断言。** 某些语义约束可以被转化为可程序化检查的规则。

```python
def test_summary_contains_key_entities():
    """摘要应该提及原文中的关键实体"""
    source_entities = extract_entities(source_text)  # 用NER工具提取
    summary = generate_summary(source_text)
    summary_entities = extract_entities(summary)

    # 关键实体的召回率应该高于阈值
    recall = len(source_entities & summary_entities) / len(source_entities)
    assert recall >= 0.7, f"关键实体召回率过低: {recall:.2%}"
```

**参考答案对比。** 构建一组"黄金标准"输入-输出对，将 LLM 输出与参考答案做相似度比较。不要求完全匹配——使用 embedding 相似度或 ROUGE 等文本相似度指标，设定合理的阈值。

**LLM-as-Judge。** 用另一个 LLM（通常是更强或者不同的模型）来评估第一个 LLM 的输出质量。这种方法有递归的不确定性问题（用不确定的评判者评判不确定的输出），但在实践中，经过精心设计 prompt 的 Judge 模型可以达到可接受的一致性。关键是将 Judge 的评估也纳入统计验证——看多次评估的分布，而非单次评估的结论。

## 测试的统计性质

LLM 应用的测试结果天然是统计性的。同一个测试用例多次运行可能产生不同的结果。这是被测系统的本质。

工程应对方式是接受统计性质，建立统计化的质量标准：

- 结构合规率 >= 98%（100 次运行中至少 98 次通过结构验证）
- 语义准确率 >= 85%（在标注测试集上的评估得分）
- 一致性 >= 90%（同一输入多次运行，输出的核心判断一致的比例）

这些阈值不是固定的——它们取决于业务场景的容错度。关键在于：**有一个明确的、可量化的质量标准，好过没有标准的"看起来不错"。**

## 测试金字塔的适配

传统测试金字塔（大量单元测试 → 适量集成测试 → 少量端到端测试）在 LLM 应用中需要适配。

**底层：结构测试。** 大量、自动化、确定性。每次 CI 运行，每次 Schema 变更，每次模型切换。

**中层：组件级语义测试。** 适量，针对单个 LLM 调用的语义质量。使用标注测试集和自动化评估指标。

**顶层：端到端语义测试。** 少量，覆盖完整业务流程。通常需要人工评估或高质量的 LLM-as-Judge。

底层保证系统不会以结构错误的方式失败。中层保证单个 LLM 调用的质量可接受。顶层保证整个系统满足业务需求。三层测试的覆盖面和成本呈金字塔分布，与传统测试金字塔的精神一致。
