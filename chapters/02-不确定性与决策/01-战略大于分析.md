# Strategy 大于 Analysis：工程决策的第一原则

## 分析的陷阱

LLM 应用开发中最常见的失败模式发生在决策层面。具体来说，是把有限的时间和注意力花在了错误的层面上。

一个团队花三周时间对比了五个大模型在某个 benchmark 上的表现差异，最终选择了分数最高的那个。然后花两周时间微调 prompt，把准确率从 87% 提升到 91%。然后花一周时间集成了一个编排框架。六周后系统上线，发现核心问题是：这个任务根本不应该用 LLM 来做——用一个正则表达式加规则引擎就能解决，准确率 100%，延迟低两个数量级，成本低三个数量级。

这是一种结构性的思维偏差：当手里有了锤子，看什么都像钉子。当团队掌握了 LLM 的使用方法，就倾向于用 LLM 解决所有问题，然后在 LLM 的框架内做无穷无尽的优化。这些优化在战术层面是正确的，但在战略层面是错误的——方向错了，走得越远偏得越多。

## Strategy 大于 Analysis 的精确含义

Strategy 大于 Analysis 不是反智主义，不是鼓励拍脑袋决策。它的精确含义是：在资源（时间、注意力、成本）有限的条件下，战略层面的正确性对系统成败的影响权重远大于战术层面的精细度。

用数学语言表述：设系统的成功概率为 P(S)，战略决策为 D_s，战术执行为 D_t，则：

P(S | D_s 正确, D_t 一般) >> P(S | D_s 错误, D_t 优秀)

这个不等式在投资领域有同样的结构。2022 年英伟达股价跌至 100 美元附近时，战略判断是"AI 基础设施的长期需求被严重低估"。这个判断一旦正确，买入时点偏差 10%、仓位管理粗糙 20%，都不影响最终的丰厚回报。反过来，如果战略判断错误——比如认为 AI 是泡沫——那么再精湛的止损策略和仓位管理都是在优化一个注定亏损的头寸。

在 LLM 工程中，战略决策包括但不限于：

- 这个任务是否应该用 LLM？
- 如果用 LLM，它在系统中的角色是什么——核心引擎还是辅助组件？
- 不确定性的容忍边界在哪里？系统能承受多高的失败率？
- 是追求单次调用的高精度，还是通过架构设计来容忍单次调用的低精度？

战术决策包括但不限于：

- 用 GPT-4o 还是 Claude 3.5？
- temperature 设多少？
- prompt 的措辞怎么优化？
- 用哪个编排框架？

战术决策不是不重要，但它们的重要性以战略决策正确为前提。在战略决策错误的框架下优化战术，是一种高效率的浪费。

## LLM 应用的战略决策框架

把战略决策从模糊的直觉转化为可操作的分析框架，需要回答三个层级的问题。

**第一层：任务适配性判断。** 这个任务是否在 LLM 的能力边界之内？[第一章](../01-认识论/what-llms-cannot-do.md)已经划定了边界：精确计算、状态维护、一致性保障、实时数据处理——这些不在边界内。意图解析、自然语言生成、文本转换、模式识别——这些在边界内。判断一个任务的适配性，不是看 LLM "能不能做"（它几乎什么都能尝试），而是看 LLM 做这件事是否比替代方案有结构性优势。

任务适配性是一个光谱，而非二元判断。强适配意味着 LLM 具有结构性优势，其他方案无法在合理成本内达到同等效果。中等适配意味着 LLM 可用，但传统方案也能胜任，选择取决于工程权衡。弱适配意味着 LLM 勉强能做，但替代方案更简单、更可靠、更经济。不适配意味着任务的核心需求落在 LLM 的能力边界之外，强行使用只会制造复杂性。

**第二层：不确定性容忍度评估。** 假设任务适合 LLM，下一个问题是：系统能承受多大的不确定性？一个内部工具的文本摘要功能，偶尔输出质量不佳是可以接受的——用户可以刷新重试。一个面向客户的合同生成系统，任何错误都可能导致法律后果。不确定性容忍度决定了系统需要多厚的验证层、多高的冗余度、以及多严格的人工审核流程。

容忍度从高到低大致分为四个区间：高容忍度适用于内部工具和创意辅助场景，偶尔出错完全可以接受；中等容忍度允许低频错误，但系统需要具备检测和恢复机制；低容忍度对应客户交互和财务文档等场景，错误代价高昂，需要严格的校验流程；零容忍适用于医疗和法律领域的关键判断，此时 LLM 不应作为最终决策者。

**第三层：架构定位决策。** LLM 在系统中扮演什么角色？这个决策直接影响系统的可靠性上限。如果 LLM 是核心引擎——系统的主要功能由 LLM 直接实现——那么系统的可靠性上限就是 LLM 的可靠性。如果 LLM 是辅助组件——核心逻辑由确定性代码实现，LLM 负责自然语言理解和生成——那么系统的可靠性可以远高于 LLM 本身。

架构定位有三种基本模式。核心引擎模式下，系统的核心功能由 LLM 直接实现，适用于任务本身就是语言理解或生成的场景。智能接口模式下，LLM 处理自然语言输入输出，但核心业务逻辑由确定性代码实现——这是大多数生产系统应该首选的模式。辅助组件模式下，LLM 只负责摘要、分类等辅助任务，系统的主流程不依赖 LLM 的输出。

这三层评估应该在写第一行代码之前完成。它的价值不在于精确量化——分类本身是粗粒度的——而在于迫使团队在正确的层面上思考问题。

考虑两个具体场景。客服邮件分类系统需要将客户邮件自动分类到五个类别。任务适配性强——自然语言理解正是 LLM 的优势区间。不确定性容忍度中等——分类错误导致工单分配失败，但人工可以纠正。LLM 可以作为核心引擎。替代方案包括传统 NLP 分类器或关键词规则引擎，但在类别语义复杂时 LLM 的优势明显。战略评估结论：适合使用 LLM。

发票金额提取与计算系统需要从发票图片中提取金额并计算总价。任务适配性需要拆分看：信息提取适合 LLM，数值计算不适合。不确定性容忍度低——金额错误直接影响财务准确性。因此 LLM 不应作为核心引擎，而应作为智能接口：LLM 负责从非结构化文档中提取结构化数据，确定性代码负责所有计算逻辑。注意一个关键推论：如果将不确定性容忍度设为零并且让 LLM 充当核心引擎，这构成结构性矛盾——零容错要求与概率性输出在原理上冲突，任何战术层面的优化都无法解决这个战略层面的矛盾。

## 战术陷阱的典型形态

识别战术陷阱和识别战略机会同样重要。以下是 LLM 应用开发中最常见的战术陷阱。

**Benchmark 崇拜。** 花大量时间对比不同模型在标准 benchmark 上的得分。问题在于：benchmark 衡量的是模型的一般能力，而你的任务是特定的。模型 A 在 MMLU 上比模型 B 高 3 分，不意味着它在你的发票分类任务上更好。唯一有意义的评估是在你的真实数据上的表现——但这需要先有战略层面的正确方向，才值得投入这个评估成本。

典型症状是团队花两周对比模型 benchmark，却没花两小时讨论任务是否适合 LLM。根本原因是把模型选择这个战术决策当成了最重要的决策，忽略了任务适配性这个战略问题。正确的做法是先在十条真实数据上用任意一个主流模型做原型验证，确认任务可行后再考虑模型选择。

**Prompt 炼金术。** 反复调整 prompt 的措辞，试图找到"完美"的表述。这个过程经常缺少收敛判据：什么时候停止？"准确率提升到什么程度就够了"这个问题如果没有在战略层面预先定义，prompt 优化就会变成一个无终止的循环。更深层的问题是：如果一个任务需要极其精心的 prompt 才能勉强工作，这本身就是一个信号——也许这个任务不适合用 prompt 来解决，也许需要的是更好的架构设计而非更好的 prompt。

典型症状是 prompt 文件已经迭代了四十多个版本，准确率在 88% 到 92% 之间反复震荡。根本原因是没有收敛判据，不知道什么时候该停止优化 prompt、转而改进架构或重新评估任务设计。正确的做法是预先定义质量阈值和最大优化轮次，达不到阈值时升级到架构层面的方案。

**框架选型过度投入。** 在 LangChain、LlamaIndex、CrewAI 等框架之间反复比较。框架解决的是战术问题（如何组织代码、如何管理链路），不解决战略问题（应不应该这么做、做到什么程度）。而且框架锁定的风险被系统性低估了——一旦深度集成了某个框架，迁移成本可能比重写还高。

典型症状是评估了五个编排框架，做了对比矩阵，写了技术选型文档，三周过去了还没写业务代码。根本原因是框架解决的是代码组织问题，不是核心的业务和架构问题——在业务逻辑不清晰时选框架，是在没有地图的情况下选交通工具。正确的做法是先用纯 Python 实现核心逻辑的原型，确认架构方向后再考虑是否需要框架。

## 投资决策与工程决策的结构性同构

Strategy 大于 Analysis 这个原则不是从软件工程中提炼出来的——它是从投资实战中提炼出来的，然后发现在工程领域有精确的对应。

投资决策的结构是：在信息永远不完备的条件下，做出有时间约束的资源分配决定。工程决策的结构是同一个：在需求不完全明确、技术不完全成熟、资源不完全充裕的条件下，做出有交付期限的技术选择。

这个同构可以映射到具体操作层面：

| 决策层级 | 投资决策 | 工程决策 | 可逆性 | 影响范围 | 错误代价 | 反馈周期 |
|---------|---------|---------|--------|---------|---------|---------|
| 战略层 | 选行业/赛道 | 选技术路线/是否用 LLM | 高成本逆转 | 全局性 | 方向性错误，可能需要推翻重来 | 数周到数月 |
| 战术层 | 选个股 | 选模型/选框架 | 中等成本 | 组件级 | 需要重构部分系统 | 数天 |
| 执行层 | 买入时机和仓位 | 开发节奏和资源分配 | 低成本 | 局部 | 局部修改即可纠正 | 数小时 |
| 风控层 | 止损纪律 | 技术方案的退出标准 | 取决于时机 | 全局性 | 取决于触发时机 | 实时到数天 |
| 时间框架 | 长期持有 vs 短线交易 | 长期架构 vs 快速原型 | -- | -- | -- | -- |

关键洞察是：在两个领域中，战略层面的决策失误都无法被战术层面的优化弥补。买了错误行业的股票，无论怎么优化买入价格都不会盈利。选了错误的技术路线，无论怎么优化 prompt 都不会成功。

每一层决策的正确性以上一层的正确性为前提。战术决策再优秀，如果战略方向错误，只是在高效地奔向错误的终点。执行层面的精细打磨，如果战术选择失当，只是在抛光一块不合格的材料。这个层级依赖关系在两个领域中完全对称。

决策层级越高，反馈周期越长，这构成了一个认知陷阱：人倾向于在反馈快的低层级上反复优化，因为每次调整都能立即看到结果；而高层级的战略决策因为反馈慢，其错误往往在数周甚至数月后才暴露。等发现战略方向错误时，已经在错误方向上投入了大量不可回收的资源。

## 战略思维的操作化

Strategy 大于 Analysis 不是一个只在白板前讨论的原则，它可以被操作化为具体的实践。

**实践一：任何 LLM 项目的第一个产出物不应该是代码，而是战略评估文档。** 这个文档回答三个问题：为什么用 LLM（而不是替代方案）？LLM 在系统中的角色是什么？不确定性的容忍边界在哪里？这个文档不需要长——一页纸就够——但它迫使团队在正确的层面上做出决策。

**实践二：设立明确的收敛判据。** 在开始 prompt 优化或模型对比之前，预先定义"什么结果意味着方向正确，什么结果意味着需要战略调整"。没有收敛判据的优化是没有终止条件的循环——它会消耗所有可用时间而不产生结论。

**实践三：原型验证先于系统建设。** 用最少的代码（通常是几十行 Python）验证核心假设：LLM 在这个任务上的表现是否在可接受范围内？然后再决定是否投入更多资源。这和投资中"小仓位试探"的逻辑相同——用最小的成本获取最关键的信息。

战略原型的操作规范是明确的。投入上限控制在四小时以内，代码量不超过五十行，只使用最简单的 prompt 和单个模型。准备十到二十条覆盖核心场景的真实测试用例，而非合成数据。质量阈值在原型开始前定义，而非事后根据结果调整。

原型验证的结论只有三种。第一种是"推进"：通过率达到或超过质量阈值，核心假设成立，可以投入资源进行系统设计，下一步是定义架构、评估模型选项、设计验证流程。第二种是"深入调查"：通过率接近阈值（例如达到阈值的 70% 以上），核心假设部分成立，需要分析失败案例的模式，可能通过更好的 prompt 设计或架构调整来改善，也可能需要混合架构。第三种是"转向"：通过率远低于阈值，核心假设不成立，LLM 在当前形态下不适合这个任务，战略选项是换用传统方案、降低任务复杂度、或等待更强的模型——不要在 prompt 优化上继续投入，那是战术层面的挣扎。

原型验证的价值不在于产出可部署的系统，而在于以最小成本获取战略层面的关键信息。四小时的原型验证比四周的框架选型有更高的战略价值，因为前者直接回答"这条路走不走得通"，后者只是在假设这条路正确的前提下优化行走方式。
