# 第二章 不确定性与决策

## 核心问题

LLM 应用开发的核心困难在于不确定性环境下的工程决策。这个问题与投资决策、数学建模中的问题同构：信息永远不完备，但决策不能无限推迟。

Strategy 大于 analysis。分析可以无穷无尽，但战略层面的判断才是决定系统成败的关键。好的架构决策来自对问题结构的深刻理解。

## 文章

- [Strategy 大于 Analysis：工程决策的第一原则](01-战略大于分析.md) -- 在 LLM 应用开发中，多数团队沉迷于 benchmark 对比、prompt 微调和框架选型的细枝末节，而忽略了战略层面的根本问题：这个任务是否应该用 LLM 来做？用到什么程度？不确定性的容忍边界在哪里？战略判断一旦正确，战术层面的错误都可以修正；战略判断错误，战术再精湛也无法挽回。
- [不确定性不是敌人，是约束条件](02-不确定性是约束条件.md) -- 概率性输出是 LLM 的本质属性，正如波动性是市场的本质属性。工程师的任务是在给定不确定性水平下设计出可靠的系统。不确定性预算、防御性架构模式、可靠性-成本权衡的量化模型。
- [决策的数学结构](03-决策的数学结构.md) -- 工程决策可以用期望值、方差和条件概率来建模。选择模型 A 还是模型 B 可以被形式化为给定约束条件下的优化问题。期望值框架、风险调整、约束优化、贝叶斯更新——这些数学工具的价值在于让隐含假设显式化、让模糊权衡可量化。

## 本章的推演逻辑

三篇文章构成一条从原则到约束再到形式化工具的推演链：

第一篇确立核心原则——战略正确性优先于战术精细度。第二篇将不确定性从"需要消除的障碍"重新定义为"需要管理的约束条件"，展开防御性架构的具体模式。第三篇用数学工具将前两篇的定性讨论推向定量——期望值、方差、约束优化、贝叶斯更新。

本章的方法论地基——在不确定性环境下做出正确的工程决策——是后续章节的前提。[第三章](../03-人机协作的软件过程/00-概述.md)在此基础上讨论开发过程的重构，[第四章](../04-声明式提示与类型契约/00-概述.md)讨论的声明式约束、[第五章](../05-架构与编排/00-概述.md)讨论的架构模式，都是在本章确立的决策框架下的具体实施。
