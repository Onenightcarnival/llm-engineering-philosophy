# 决策的数学结构

## 工程决策可以形式化

"我们团队觉得 Claude 比 GPT 好用。""我个人偏好用 RAG 而非 fine-tune。""这个 prompt 感觉比那个好。"

这些表述用主观感受替代了可量化的分析。工程决策——选择模型 A 还是模型 B、采用架构 X 还是架构 Y、投入资源优化 prompt 还是改进验证层——可以被形式化为结构化的分析过程。形式化不保证得到"完美"答案，但它能做到两件事：把隐含的假设显式化，把模糊的权衡量化。

## 期望值思维

最基础的决策工具是期望值：把每个可能结果的价值乘以它发生的概率，加总后减去固定成本。

这个思路本身很简单。它的工程价值在于迫使决策者把隐含假设写下来：你认为有哪些可能的结果？你给每个结果赋予了什么概率？你如何量化每个结果的价值？当团队围绕"用哪个模型"争论时，让每个人写下自己的概率估计和价值判断，分歧的真实来源会立刻浮现——通常不在"哪个模型好"，而在"我们对失败场景的概率估计完全不同"或"我们对失败后果的严重程度判断相反"。

形式化还有一个作用：暴露被忽略的尾部风险。如果团队只考虑"准确率"这个维度，他们可能忽略了"严重错误导致客户投诉"这个低概率但高代价的结果。写下概率表迫使你面对这些不舒服的场景。

## 方差和尾部风险

期望值是必要但不充分的决策工具。两个期望值相同的方案可能有截然不同的风险特征。

风险调整的核心思想是：回报要除以波动性来看。同样的期望收益，波动越大，风险调整后的价值越低。风险厌恶程度因人而异——有的团队能接受高波动换高上限，有的团队宁可要稳定的中等结果。

另一个有用的视角是看尾部：当事情出差错时，平均会有多差？这比"最坏情况"更有信息量，因为它不只看单个极端事件，而是看整个尾部区域的平均表现。

这两个视角组合起来指向一个在架构决策中反复出现的结论：复杂架构的上限往往更高，但方差也大得多。风险调整后，简单架构经常是更优的选择。一个单次 LLM 调用加 Pydantic 验证的方案，最差情况下可能只是降级处理；一个三步 Agent 加 RAG 加多模型集成的方案，最差情况可能是级联失败、长时间不可用。上限高不等于风险调整后的价值高。

## 约束优化

更完整的决策建模需要引入约束条件。工程决策是在成本、时间、可靠性、团队能力等约束条件下求可行解。

约束优化的流程只有两步：过滤不满足约束的方案，然后对剩余的可行方案按期望值排序。

这个两步过程看似简单，但它的价值在于不会被方案的"酷炫程度"或"技术新颖性"误导。一个期望值最高的方案如果违反了成本约束或超出了团队的技术能力，它根本不在可行域内——无论理论上多好。约束越精确，可行域越小，决策越清晰。没有约束的优化问题要么无解，要么解是平凡的。

实际操作中，约束包括月度成本上限、延迟上限、可靠性下限、开发周期上限、团队掌握的技术栈。把这些约束写成一张检查表，对每个候选方案逐项核实，不满足任何一项即排除。剩下的可行方案通常不多，排序就变得容易了。

## 根据新证据更新判断

工程决策中一个常被忽略的维度：判断应该随着新信息的获取而更新。

更新的关键在于证据的信息量。一条强证据（比如 100 个样本的盲测结果）应该大幅移动你的判断；一条弱证据（比如"上线后暂时没人投诉"）只应该产生微小的推动。单条强负面证据可以逆转之前多条正面证据建立的信心——比如发现一类特殊格式的输入全部处理失败，足以把之前"10 个样本中 9 个正确"建立起来的信心拉回原点。

好的判断更新机制既不固执也不过度反应。不会因为一条弱正面证据就盲目乐观，也不会因为一条弱负面证据就全盘推翻。每条证据的推力和它的信息量成正比，不多不少。

具体场景：评估"LLM 在发票数据提取任务上是否达标"。初始判断五五开。跑了十个样本，九个正确，信心上升。然后发现一类特殊格式全部失败，信心被拉回来。修改 prompt 后特殊格式也能处理，信心又上升。一百个样本的盲测通过，信心大幅上升。上线后一周无投诉，信心微弱上升。整个过程中，判断在持续校准，而非一锤定音。

## 形式化的边界

形式化有清晰的能力边界。

概率估计是主观的。"准确率满足需求的概率是 70%"这个数字是工程师的判断，形式化能让这个判断显式化、可讨论、可修正，但不能让它变成客观事实。

价值量化是困难的。"客户投诉"的损失取决于投诉的严重程度、客户的重要性、品牌的损害——这些因素难以精确量化。形式化提供的是一个框架，让你可以在不同假设下看决策如何变化（敏感性分析）。

现实比模型复杂。上面的思路假设结果互斥、概率已知、决策一次性。现实中结果可能重叠，概率不断变化，决策是序列化的。

总结：形式化让隐含假设显式化，让模糊权衡可量化，让主观判断可讨论。但它不提供客观答案，不消除判断的主观性，不保证模型与现实一致。形式化最有价值的场景是高影响、低可逆性的决策——这些决策值得投入分析成本，因为错误的代价远高于分析的代价。如果决策后果小、可逆性高（比如 prompt 的一个措辞调整），直接做实验比分析更高效。

这回到了本章的第一原则：Strategy 大于 Analysis。形式化分析是分析的工具，但知道什么时候分析、分析到什么程度，本身是一个战略判断。
