# 第二章 不确定性与决策

## 核心问题

LLM 应用开发的核心困难不是技术实现，而是在不确定性环境下做出正确的工程决策。这个问题与投资决策、数学建模中的问题同构：信息永远不完备，但决策不能无限推迟。

Strategy 大于 analysis。分析可以无穷无尽，但战略层面的判断才是决定系统成败的关键。好的架构决策不是来自更多的调研，而是来自对问题结构的深刻理解。

## 文章

- [Strategy 大于 Analysis：工程决策的第一原则](strategy-over-analysis.md) -- 在 LLM 应用开发中，多数团队沉迷于 benchmark 对比、prompt 微调和框架选型的细枝末节，而忽略了战略层面的根本问题：这个任务是否应该用 LLM 来做？用到什么程度？不确定性的容忍边界在哪里？战略判断一旦正确，战术层面的错误都可以修正；战略判断错误，战术再精湛也无法挽回。
- [不确定性不是敌人，是约束条件](uncertainty-as-constraint.md) -- 概率性输出不是 LLM 的缺陷，而是它的本质属性，就像波动性不是市场的缺陷，而是市场的本质属性。工程师的任务不是消除不确定性，而是在给定不确定性水平下设计出可靠的系统。不确定性预算、防御性架构模式、可靠性-成本权衡的量化模型。
- [恐慌时刻的逆向工程](contrarian-engineering.md) -- 当一个新模型发布、一个框架被废弃、或者一种方法论被"证伪"时，多数人的反应是追逐潮流或者恐慌转向。但正确的做法是回到第一性原理：这个变化改变了哪些基本假设？如果基本假设没变，战略就不需要变。技术领域的"极端恐慌"与金融市场的极端恐慌本质相同——它创造的是机会，不是危机。
- [知行合一：从设计到实现的一致性](design-implementation-consistency.md) -- 好的工程决策需要克服人性弱点才能落地。复杂性偏好、沉没成本谬误、从众心理、恐惧驱动的过度设计——这些认知偏差在 LLM 应用开发中反复出现。纪律性不是天赋，是通过预设决策规则、强制性简化约束和定期止损审查来保障的工程能力。
- [决策的数学结构](decision-mathematics.md) -- 工程决策可以用期望值、方差和条件概率来建模。选择模型 A 还是模型 B 不是品味问题，而是一个在给定约束条件下的优化问题。期望值框架、风险调整、约束优化、贝叶斯更新——这些数学工具的价值不在于计算精确答案，而在于让隐含假设显式化、让模糊权衡可量化。

## 本章的推演逻辑

五篇文章构成一条从原则到方法论再到形式化工具的推演链：

第一篇确立核心原则——战略正确性优先于战术精细度。第二篇将不确定性从"需要消除的障碍"重新定义为"需要管理的约束条件"，展开防御性架构的具体模式。第三篇讨论在技术环境剧变时如何保持战略定力——区分基本假设、经验假设和偶然假设的层级。第四篇聚焦于"知道正确答案"到"做到正确答案"之间的鸿沟，提供将纪律性嵌入工程流程的机制。第五篇用数学工具将前四篇的定性讨论推向定量——期望值、方差、约束优化、贝叶斯更新。

本章的方法论地基——在不确定性环境下做出正确的工程决策——是后续章节的前提。[第三章](../03-提示工程作为软件工程/README.md)讨论的 prompt 设计、[第四章](../04-类型系统与契约/README.md)讨论的类型约束、[第五章](../05-架构模式/README.md)讨论的架构模式，都是在本章确立的决策框架下的具体实施。
