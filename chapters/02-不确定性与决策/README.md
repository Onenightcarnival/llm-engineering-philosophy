# 第二章 不确定性与决策

## 核心问题

LLM 应用开发的核心困难不是技术实现，而是在不确定性环境下做出正确的工程决策。这个问题与投资决策、数学建模中的问题同构：信息永远不完备，但决策不能无限推迟。

Strategy 大于 analysis。分析可以无穷无尽，但战略层面的判断才是决定系统成败的关键。好的架构决策不是来自更多的调研，而是来自对问题结构的深刻理解。

## 计划文章

- **Strategy 大于 Analysis：工程决策的第一原则** -- 在 LLM 应用开发中，多数团队沉迷于 benchmark 对比、prompt 微调和框架选型的细枝末节，而忽略了战略层面的根本问题：这个任务是否应该用 LLM 来做？用到什么程度？不确定性的容忍边界在哪里？战略判断一旦正确，战术层面的错误都可以修正；战略判断错误，战术再精湛也无法挽回。

- **不确定性不是敌人，是约束条件** -- 概率性输出不是 LLM 的缺陷，而是它的本质属性，就像波动性不是市场的缺陷，而是市场的本质属性。工程师的任务不是消除不确定性，而是在给定不确定性水平下设计出可靠的系统。这需要的不是更好的 prompt，而是更好的架构。

- **恐慌时刻的逆向工程** -- 当一个新模型发布、一个框架被废弃、或者一种方法论被"证伪"时，多数人的反应是追逐潮流或者恐慌转向。但正确的做法是回到第一性原理：这个变化改变了哪些基本假设？如果基本假设没变，战略就不需要变。技术领域的"极端恐慌"与金融市场的极端恐慌本质相同——它创造的是机会，不是危机。

- **知行合一：从设计到实现的一致性** -- 好的工程决策需要克服人性弱点才能落地。明知应该选择简单方案，却因为"看起来不够高级"而选择复杂框架；明知应该先验证核心假设，却急于搭建完整系统——这些都是知行不一的表现。系统设计中的纪律性，与任何需要长期坚持正确策略的领域一样，是一种可以训练的能力。

- **决策的数学结构** -- 工程决策可以用期望值、方差和条件概率来建模。选择模型 A 还是模型 B 不是品味问题，而是一个在给定约束条件下的优化问题。计算数学训练带来的不是具体的算法知识，而是一种将模糊问题形式化的直觉：定义变量、建立约束、求解或证明无解。
