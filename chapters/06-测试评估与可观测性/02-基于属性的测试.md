# 基于属性的测试在 LLM 应用中的适配

## 为什么基于属性的测试天然适合 LLM

传统单元测试断言具体值：`assert f(3) == 9`。这在确定性系统中工作良好，但在 LLM 应用中失效——同一输入不会产生相同输出。

基于属性的测试（Property-based Testing）提供了一个不同的思路：不断言具体值，而是断言**输出必须满足的性质**。`assert f(x) >= 0`——不管 f(x) 的具体值是多少，它必须是非负的。这种"不关心具体值，只关心性质"的测试策略天然适配 LLM 的非确定性输出。

```python
from hypothesis import given, strategies as st

@given(text=st.text(min_size=10, max_size=1000))
def test_summary_shorter_than_input(text):
    """摘要的长度应该短于原文"""
    summary = generate_summary(text)
    assert len(summary) < len(text)

@given(text=st.text(min_size=50))
def test_sentiment_is_valid(text):
    """情感分析的结果必须是合法的枚举值"""
    result = analyze_sentiment(text)
    assert result.sentiment in {"positive", "negative", "neutral"}
    assert 0 <= result.confidence <= 1
```

Hypothesis 等 PBT 框架自动生成大量随机输入，检查每个输入是否满足声明的属性。这不仅测试了"典型输入"，还自动探索了边界情况——空字符串、超长文本、特殊字符、多语言混合。

## 可测试属性的分类

LLM 输出可以测试的属性大致分为四类：

**结构属性。** 输出的形式化约束。字段存在性、类型正确性、值域范围、数组长度。这类属性最容易定义和检验，与第四章的类型约束直接对应。

**关系属性。** 输入与输出之间的关系。摘要短于原文、翻译保留原文的实体、分类结果不依赖于输入中不相关的变化（加入无关噪声不应改变分类结果）。

**不变性属性。** 特定变换下输出应保持不变。将相同内容用不同措辞表达，情感分析结果应该一致。将文本中的人名替换为其他人名，摘要的结构应该不变。

**一致性属性。** 输出内部的自洽性。如果 confidence 很高，reasoning 应该包含具体证据。如果分类为 negative，key_phrases 中不应全是正面词汇。

## 属性测试的实践约束

LLM API 调用有成本和延迟。Hypothesis 默认对每个测试函数生成 100 个随机输入——对于确定性函数这是合理的，但对于需要调用 LLM API 的测试，100 次调用可能耗时过长、成本过高。

实践中的调整：

**减少采样数量。** `@settings(max_examples=10)` 将每次测试的随机输入减少到 10 个。在 CI 中用少量样本做冒烟测试，定期用大量样本做全面测试。

**缓存 LLM 响应。** 对于相同的输入，缓存 LLM 的响应避免重复调用。注意缓存会掩盖非确定性——缓存模式下测试的是属性是否在"某一次"输出上成立，而非在"任意一次"输出上成立。

**分层属性测试。** 结构属性和值域属性的测试不需要调用 LLM——可以直接对已有的 LLM 输出样本运行。只有关系属性和一致性属性需要实际调用 LLM。

## 与传统测试的互补

基于属性的测试不替代传统的基于示例的测试，而是互补。

基于示例的测试覆盖了"已知的重要场景"——用精心构造的输入和预期输出验证系统在关键业务场景下的表现。基于属性的测试覆盖了"未预见的边界情况"——通过随机探索发现开发者没有想到的输入模式和失败模式。

两者的结合提供了更完整的质量保障：示例测试保证系统在已知场景下正常工作，属性测试保证系统在广泛的输入空间中不会以意外的方式失败。
