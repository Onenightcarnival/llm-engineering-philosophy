# 第六章 测试、评估与可观测性

## 核心问题

如何测试一个输出不确定的系统？传统软件测试建立在"相同输入产生相同输出"的假设之上。当这个假设不再成立，测试的含义需要被重新定义。

前面五章讨论了如何设计和构建 LLM 应用——认识论、决策框架、开发过程、声明式约束、架构与编排。本章回答一个不可回避的问题：怎么知道系统在正常工作？

## 文章

- [结构测试与语义测试](01-结构测试与语义测试.md) -- 两个独立层面的正确性验证。结构测试（输出是否符合 Schema）可以完全自动化；语义测试（输出的含义是否正确）需要基于规则的断言、参考答案对比和 LLM-as-Judge 等策略。测试的统计性质和测试金字塔的适配。
- [基于属性的测试在 LLM 应用中的适配](02-基于属性的测试.md) -- Property-based testing 天然适合不确定性系统：不断言具体值，而是断言输出必须满足的性质。四类可测试属性（结构、关系、不变性、一致性），以及 API 调用成本下的实践约束。
- [实验管理与统计度量](03-实验管理与统计度量.md) -- Prompt 具有代码与配置的双重身份。它需要版本控制，更需要实验管理：A/B 测试、效果对比、变更归因、回滚机制。如何定义可量化的质量维度，如何建立具有统计属性的评估基线。
- [评估即测试](04-评估即测试.md) -- 测试与评估的边界在 LLM 应用中消失了。统计度量（准确率、一致性、幻觉率）本质上是持续运行的测试套件。评估数据集的构建、持续评估的实践、以及评估的成本收益分析。
- [可观测性设计](05-可观测性设计.md) -- 日志、追踪、指标三大支柱在 LLM 应用中的特殊实现。输出质量作为可观测性的新维度，prompt 版本与质量的关联分析，以及成本可观测性。
- [优雅降级与持续改进](06-优雅降级与持续改进.md) -- 降级是正常运行的一部分。五级降级层级设计、触发条件的量化、透明度原则。以及如何通过用户反馈、输出日志和人工标注构建正反馈闭环，将降级数据转化为系统改进的燃料。

## 本章的推演逻辑

六篇文章构成一条从测试方法到运行时保障再到持续改进的推演链：

第一篇建立测试的分类框架——结构测试和语义测试的独立性、各自的策略和局限性。第二篇引入一种特别适合 LLM 应用的测试范式——基于属性的测试，作为传统基于示例测试的互补。第三篇将 ML 实验管理的方法论引入 prompt 工程，建立版本控制、统计基线和质量度量的基础设施。第四篇模糊了测试与评估的边界，将统计度量纳入持续质量保障体系。第五篇从测试转向运行时的可观测性——如果测试是上线前的质量保障，可观测性是上线后的质量监控。第六篇讨论当质量保障失败时的最后一道防线——优雅降级，以及如何将系统运行中产生的数据转化为持续改进的动力。

本章的测试与可靠性方法论回答了"怎么知道系统在正常工作"。[第七章](../07-反模式与陷阱/00-概述.md)则从反面回答"什么做法会让系统出问题"。
