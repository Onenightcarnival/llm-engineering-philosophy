# 第六章 工作流与编排

## 核心问题

当任务复杂度超出单次 LLM 调用的承载能力，就需要将任务分解并编排。问题在于：编排应该在什么层面发生，由谁来控制。

第五章讨论的架构模式是静态的结构描述——系统的组件有哪些、它们之间的关系是什么。本章是这些静态结构的动态展开：组件如何被执行、按什么顺序、错误如何处理、流程如何控制。

## 文章

- [隐式编排与显式编排](01-隐式编排与显式编排.md) -- 两种根本不同的编排范式。隐式编排通过数据结构驱动推理流程，显式编排通过代码定义执行图。隐式编排被严重低估了。以及为什么大多数场景下，Python 本身就是最好的编排语言。
- [多轮交互的状态机模型](02-多轮交互的状态机模型.md) -- 多轮对话本质上是一个有限状态机。用形式化的视角重新审视对话系统设计：显式状态定义、转移函数实现、以及状态机与 Agent 的控制权差异。
- [错误传播与补偿机制](03-错误传播与补偿机制.md) -- LLM 调用的四种失败模态（硬失败、格式失败、语义偏离、幻觉），多步工作流中错误传播的级联效应，以及验证点和补偿策略的设计。
- [为什么大多数 LLM 编排框架过度设计了](04-编排框架的过度设计.md) -- 一篇观点文章。抽象层级膨胀、概念过载和不必要的间接性。框架的正当使用场景，以及"无框架编排"的替代方案。

## 本章的推演逻辑

四篇文章从编排范式到错误处理再到工具选择，构成一条完整的推演链：

第一篇建立概念框架——编排的两种基本范式及其适用条件，为后续讨论提供分类学基础。第二篇聚焦于最常见的编排场景——多轮对话，用状态机的形式化视角提供精确的设计工具。第三篇讨论编排中最关键的工程问题——错误如何在多步工作流中传播和恢复。第四篇回到工具选择的战略层面，对编排框架的过度使用提出批判。

本章讨论了任务如何被分解和执行。但如何知道执行的结果是否正确？这就是[第七章](../07-测试与可靠性/00-概述.md)的主题：不确定性系统的测试与可靠性保障。
