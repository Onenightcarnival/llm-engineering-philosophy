# 多轮交互的状态机模型

## 对话就是状态机

多轮对话系统中，每一轮交互都可以被视为一次状态转移。用户的输入是触发事件，LLM 的处理逻辑是转移函数，系统的回复和内部状态变化是转移的结果。

这不是修辞。有限状态机（FSM）的形式化定义——状态集合、输入字母表、转移函数、初始状态、接受状态——可以精确映射到多轮对话系统的每个组件：

- **状态集合**：对话可能处于的所有阶段（收集信息、确认订单、等待支付、完成）。
- **输入字母表**：用户可能发送的消息类型（意图分类的结果）。
- **转移函数**：给定当前状态和用户输入，系统应该进入什么新状态。
- **初始状态**：对话开始时的状态。
- **接受状态**：对话可以正常结束的状态。

用状态机的形式化视角审视对话系统，揭示了一个关键问题：多数对话系统没有显式的状态定义。对话的"状态"隐含在消息历史中，由 LLM 在每次调用时从历史消息中隐式"推断"。这种隐式状态管理在简单场景下工作良好，但在复杂业务流程中会导致状态丢失、状态回退和非法状态转移。

## 显式状态的工程价值

```python
from enum import Enum
from pydantic import BaseModel, Field
from typing import Literal

class OrderState(str, Enum):
    collecting_info = "collecting_info"
    confirming = "confirming"
    processing = "processing"
    completed = "completed"
    cancelled = "cancelled"

class DialogContext(BaseModel):
    state: OrderState = Field(default=OrderState.collecting_info)
    collected_fields: dict = Field(default_factory=dict)
    confirmed: bool = Field(default=False)
```

显式定义状态带来三个直接的工程收益：

**可验证性。** 可以编写测试断言：在 `collecting_info` 状态下收到"取消"意图时，系统应转移到 `cancelled` 状态。这种基于状态转移的测试比基于对话内容的端到端测试更精确、更稳定。

**可恢复性。** 如果对话中断（用户离开页面、系统重启），显式状态可以被序列化和恢复。用户回来时，系统知道对话处于什么阶段，而不需要重新"理解"整个对话历史。

**可审计性。** 状态转移序列构成了对话的结构化日志。比原始消息历史更容易分析和监控——可以统计每个状态的平均停留时间、最常见的转移路径、非正常结束的比例。

## 转移函数的实现

在 LLM 对话系统中，转移函数是 LLM 意图识别与确定性业务逻辑的组合。

```python
def transition(state: OrderState, user_intent: str, context: DialogContext) -> OrderState:
    transitions = {
        (OrderState.collecting_info, "provide_info"): OrderState.collecting_info,
        (OrderState.collecting_info, "confirm"): OrderState.confirming,
        (OrderState.collecting_info, "cancel"): OrderState.cancelled,
        (OrderState.confirming, "confirm"): OrderState.processing,
        (OrderState.confirming, "modify"): OrderState.collecting_info,
        (OrderState.confirming, "cancel"): OrderState.cancelled,
        (OrderState.processing, "complete"): OrderState.completed,
    }

    next_state = transitions.get((state, user_intent))
    if next_state is None:
        return state  # 非法转移：保持当前状态
    return next_state
```

转移表将所有合法的状态转移显式枚举。非法转移（不在表中的状态-输入组合）被显式处理——保持当前状态，而非让 LLM 自行决定。

这种"LLM 负责意图识别，确定性代码负责状态转移"的分工，是[第五章](../05-架构模式/03-胶水层与核心引擎.md)讨论的胶水层模式的直接应用。LLM 处理模糊的自然语言输入，确定性代码执行精确的状态控制。

## 超越有限状态机

简单的 FSM 在某些场景下不够表达。两种常见的扩展：

**带守卫条件的转移。** 状态转移不仅取决于输入，还取决于上下文条件。例如，从 `collecting_info` 到 `confirming` 的转移要求所有必填字段已收集。守卫条件可以用 Pydantic validator 实现——在尝试进入新状态前，验证上下文是否满足进入条件。

**层级状态机。** 复杂业务流程可以被分解为多层状态机。顶层状态机管理业务流程的主要阶段，每个阶段内部有自己的子状态机处理细节交互。这种分层与第四章讨论的类型嵌套是同构的——通过组合简单的状态机构建复杂的行为。

## 状态机与 Agent 的关系

第五章讨论的 Agent 本质上也是一个状态机——但它是一个**由 LLM 动态定义转移函数的状态机**。Agent 的每一步，LLM 不仅决定输入（选择哪个工具），还隐式地决定了状态转移的方向。

本文讨论的状态机模型与 Agent 的区别在于控制权：状态机的转移规则由开发者预先定义，LLM 只负责输入分类；Agent 的转移规则由 LLM 动态生成。前者更可控但更死板，后者更灵活但更不可预测。

选择标准与业务的确定性相关：如果业务流程是确定的（下单流程、审批流程），用显式状态机；如果任务的执行路径需要动态决定（开放式问题求解、探索性分析），用 Agent。
