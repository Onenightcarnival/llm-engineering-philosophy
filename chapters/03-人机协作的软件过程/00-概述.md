# 第三章 人机协作的软件过程

LLM 不仅是被开发的对象，也是开发过程的参与者。这从根本上改变了软件开发的工作流程和分工模式。

本章放在技术细节（prompt 工程、类型系统、架构模式）之前，因为工作方式的转变是一切技术实践的前提。读者在学习如何写声明式 prompt 或设计 RAG 架构之前，需要先理解：日常写代码、做代码审查、写文档的方式已经变了。带着这个认知去读后续章节，每个技术决策都有过程上下文。

## 文章索引

- [AI 辅助编程的正确姿势](01-AI辅助编程的正确姿势.md) -- 不是"让 AI 写代码，人来审查"这么简单。更准确的模型是：人定义结构和约束，AI 填充实现；人负责"做什么"和"为什么"，AI 负责"怎么做"。
- [代码审查中的人机分工](02-代码审查中的人机分工.md) -- AI 擅长发现模式级问题（风格不一致、常见 bug 模式），人擅长判断设计决策的合理性。如何设计一个充分利用双方优势的审查流程。
- [文档驱动开发在 LLM 时代的复兴](03-文档驱动开发的复兴.md) -- 文档曾是软件工程中最被忽视的产出物。但在 LLM 时代，文档不再只是给人读的——它也是给 LLM 读的上下文。README、注释、类型注解都成为了"prompt 的一部分"。这个认知为[第四章](../04-声明式提示与类型契约/00-概述.md)的类型系统讨论提供了动机：文档是给人和机器双重读者写的规格说明，Pydantic 模型是这种规格说明的形式化表达。
- [知识库作为活的系统规格说明](04-知识库作为活的系统规格说明.md) -- 本项目本身就是一个例子。一个结构良好的知识库，既是给人阅读的技术著作，也是给 LLM 提供的系统规格说明。这种双重身份如何影响内容的组织方式。
