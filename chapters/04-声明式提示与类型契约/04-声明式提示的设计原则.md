# 声明式提示的设计原则

前两篇确立了声明式思维链的两个核心：Code as Prompt（Pydantic 模型即 prompt）和 Schema as Workflow（字段排列即推理路径）。本篇讨论将这两个概念落地时需要遵循的设计原则。

## 约束输出空间

LLM 的默认输出空间是无限的——给定任何 prompt，理论上所有 token 序列都是可能的输出。工程化的核心任务之一，是将这个无限空间压缩到与问题相匹配的有限空间。`Literal` 类型代表了约束的极端形式：将输出空间从无限直接压缩到一个显式枚举的有限集合。

```python
from typing import Literal

# 无约束：LLM 可以返回任何字符串
sentiment: str

# Literal 约束：输出空间被压缩到恰好三个值
sentiment: Literal["positive", "negative", "neutral"]
```

这个压缩同时完成了三件事：降低了输出熵（更少的选择意味着更高的一致性），提高了下游系统的可处理性（有限枚举可以穷举处理），以及将模糊的语义分类转化为精确的计算对象。

**枚举粒度的权衡。** 粒度太粗，每个类别内部差异太大，下游处理不了。粒度太细，相邻类别之间界限模糊，LLM 分不清。枚举粒度应该匹配 LLM 在该任务上的实际区分能力，而不是业务逻辑的理想精度。

实践中的经验法则：一个 Literal 类型中的枚举值不超过 10 个。超过这个数量，引入层级结构——先粗分类，再细分类。

```python
from pydantic import BaseModel, Field
from typing import Literal

class CoarseIntent(BaseModel):
    category: Literal["order", "product", "account", "other"] = Field(
        description="意图的大类"
    )

class OrderIntent(BaseModel):
    sub_intent: Literal[
        "place_order", "cancel_order", "modify_order", "track_order"
    ] = Field(description="订单相关意图的细分类别")
```

层级化分类将一个高维的决策问题分解为多个低维的决策问题。每一层的 Literal 枚举都保持在 LLM 能可靠区分的粒度内。

**Literal 与 Enum 的选择。** 如果枚举只用于约束 LLM 输出的取值范围，用 Literal。如果枚举值本身是业务领域的一等公民（需要附加方法、属性和元数据），用 Enum。

```python
from enum import Enum

class Priority(str, Enum):
    critical = "critical"
    high = "high"
    medium = "medium"
    low = "low"

    @property
    def sla_hours(self) -> int:
        return {"critical": 1, "high": 4, "medium": 24, "low": 72}[self.value]
```

继承 `str` 使得 `Priority` 的值是字符串，可以直接用于 JSON 序列化和 Pydantic 模型。附加的 `sla_hours` 属性将枚举值与业务逻辑绑定。

**枚举值的命名是 prompt 设计问题。** `Literal["positive", "negative", "neutral"]` 中的字符串值本身就是 LLM 的语义线索。选择 "positive" 还是 "pos"、"good" 还是 "favorable"，会影响 LLM 的分类行为。

## 分离关注点

关注点分离在声明式 prompt 中有特殊的意义：不同类型的推理步骤应该被明确区分，因为它们有不同的质量标准和验证策略。

```python
from pydantic import BaseModel, Field
from typing import Optional, Literal


# 关注点 1：数据提取（纯事实，可核查）
class FinancialData(BaseModel):
    company_name: str
    ticker: Optional[str] = Field(
        default=None,
        description="股票代码（如有）"
    )
    revenue_millions: Optional[float] = Field(
        default=None, gt=0,
        description="营收（百万美元），原文未提及则为 null"
    )

# 关注点 2：定性分析（需要判断，有不确定性）
class QualitativeAssessment(BaseModel):
    market_position: Literal["leader", "challenger", "niche", "unclear"] = Field(
        description="基于报告内容判断的市场地位"
    )
    risk_factors: list[str] = Field(
        min_length=0, max_length=5,
        description="报告中提及的主要风险因素"
    )
```

`FinancialData` 的正确性可以用事实核查来验证——营收数字对不对，去原文比对即可。`QualitativeAssessment` 的"正确性"是统计性的——市场地位的判断合不合理，需要评估者的专业判断。如果将它们混在一个扁平结构中，就无法为不同的关注点设定不同的质量标准。

**内容与格式的分离。** 在自然语言 prompt 中，"分析文本的情感"（内容关注点）和"输出为 JSON，包含 sentiment 字段"（格式关注点）通常混杂在一起。在声明式 prompt 中，内容关注点由 Field description 承担，格式关注点由类型系统承担——两者自然分离。

**约束与实现的分离。** "年份在 2000-2030 之间"是约束。"怎么从文本中提取年份"是实现。在自然语言 prompt 中，约束通常嵌入在实现指令中（"提取年份，注意年份应该在 2000-2030 之间"）。在声明式 prompt 中，约束由 `Field(ge=2000, le=2030)` 独立表达，与"怎么提取"完全分离。

## 最小惊讶原则

字段名和 description 应该在语义上一致。LLM 根据字段名和 description 来推断应该生成什么内容。字段名指向一个方向、description 指向另一个方向，会增加输出的不确定性。

```python
from pydantic import BaseModel, Field

# 违反最小惊讶原则
class Confusing(BaseModel):
    summary: str = Field(
        description="详细列出文本中的所有数据点及其来源"
    )
    # "summary" 暗示简短概括，description 要求详细列举——矛盾

# 遵循最小惊讶原则
class Clear(BaseModel):
    data_points: list[str] = Field(
        description="文本中的所有数据点及其来源"
    )
    summary: str = Field(
        description="基于上述 data_points，简短概括核心发现"
    )
```

对维护者的最小惊讶也很重要：字段的顺序应该遵循自然的思维流程（先观察、再分析、最后结论），而不是按照某种"优化过"但反直觉的顺序排列。

## DRY 的重新审视

DRY（Don't Repeat Yourself）在 prompt 设计中需要重新审视。在代码中，重复是有害的，因为修改一处忘记修改另一处会导致不一致。在自然语言 prompt 中，关键约束出现两次可以增加 LLM 遵循约束的概率——因为 LLM 的注意力机制不保证均匀分配。

声明式 prompt 用不同的方式解决了这个问题：

```python
from typing import Literal

# 在自然语言 prompt 中需要重复强调的约束：
# "情感只能是 positive/negative/neutral（请务必只选择这三个之一，
#  不要输出其他值）"

# 在声明式 prompt 中，一次声明即可：
sentiment: Literal["positive", "negative", "neutral"]
# Literal 类型已经将约束精确编码了。
# 类型系统保证了约束只有一个权威来源。
```

DRY 的真正目标是确保每个约束只有一个权威来源（Single Source of Truth）。在声明式 prompt 中，类型定义是权威来源，Field description 是辅助说明。当两者矛盾时，类型系统的约束优先——因为它在验证层面被强制执行。

## 保持一致性：验证器作为运行时不变量

类型注解和 Literal 枚举能表达的约束是"值的属性"级别的：类型、范围、枚举。它们不能表达"值之间的关系"：字段 A 的值决定字段 B 的取值范围、两个字段的时序关系、复杂度评估与子任务数量的匹配。

这类关系性约束由 Pydantic 的 validator 来保障。

**field_validator：字段级不变量。** 作用于单个字段，适合表达超出 JSON Schema 表达能力的单字段约束。

```python
from pydantic import BaseModel, Field, field_validator
import re

class CodeReview(BaseModel):
    file_path: str = Field(description="被审查的文件路径")
    severity: str = Field(description="问题严重程度：critical、warning 或 info")
    line_number: int = Field(description="问题所在行号", ge=1)
    suggestion: str = Field(description="修改建议")

    @field_validator("file_path")
    @classmethod
    def validate_file_path(cls, v: str) -> str:
        if not re.match(r'^[\w./\-]+$', v):
            raise ValueError(f"文件路径包含非法字符: {v}")
        if '..' in v:
            raise ValueError(f"文件路径不允许包含 '..': {v}")
        return v

    @field_validator("suggestion")
    @classmethod
    def validate_suggestion_not_empty(cls, v: str) -> str:
        if len(v.strip()) < 10:
            raise ValueError("修改建议过短，至少需要10个字符的具体描述")
        return v.strip()
```

`file_path` 的验证器防止路径遍历攻击——LLM 可能生成包含 `../` 的路径。`suggestion` 的验证器保证建议有实质内容。注意 `field_validator` 可以修改值（返回 `v.strip()`），同时承担验证和标准化的双重职责。

**model_validator：跨字段不变量。** 在所有字段解析完成后执行，可以访问整个模型实例。

```python
from pydantic import BaseModel, Field, model_validator
from typing import Literal

class TaskDecomposition(BaseModel):
    original_task: str = Field(description="原始任务描述")
    subtasks: list[str] = Field(description="分解后的子任务列表")
    estimated_total_hours: float = Field(description="预估总工时")
    complexity: Literal["low", "medium", "high"] = Field(
        description="任务复杂度评估"
    )

    @model_validator(mode="after")
    def validate_consistency(self):
        if self.complexity == "low" and len(self.subtasks) > 5:
            raise ValueError(
                f"复杂度为 low 但子任务数量为 {len(self.subtasks)}，不一致"
            )
        if self.estimated_total_hours < len(self.subtasks) * 0.5:
            raise ValueError(
                f"预估工时({self.estimated_total_hours}h)相对于"
                f"{len(self.subtasks)}个子任务过少"
            )
        return self
```

这个验证器表达的是业务一致性：如果 LLM 把任务判断为"低复杂度"但分解出了 8 个子任务，这种不一致暗示分析质量有问题。LLM 在生成复杂结构化输出时，不同字段之间的一致性不是自动保证的——每个字段是按顺序生成的，后面的字段虽然能"看到"前面的值，但不意味着它会主动保持一致。显式的 model_validator 是唯一可靠的一致性保障。

**三层验证架构。** 从宽到严，每一层处理上一层无法表达的约束：

1. **Schema 层**：类型、枚举、值域范围。在 LLM 生成时强制执行（如果 API 支持 strict mode）。
2. **字段层**：`field_validator`，单个字段的复杂约束。在 Pydantic 解析时执行。
3. **模型层**：`model_validator`，字段之间的关系约束。在所有字段解析完成后执行。

这种分层对应传统数据库的约束体系：列类型约束 → CHECK 约束 → 触发器。也对应 Bertrand Meyer 的 Design by Contract：前置条件（prompt 的构造逻辑确保输入合理）、后置条件（validator 验证输出满足约定）、类不变量（model_validator 保证系统状态一致）。

区别在于：传统 DbC 中，违反契约意味着 bug；LLM 应用的 DbC 中，违反契约意味着需要重试或降级。

## 处理验证失败

不变量被违反时，系统需要一个恢复策略。在 LLM 应用中，验证失败是正常操作的一部分——LLM 的概率性输出不保证满足所有约束。

**重试。** 捕获 ValidationError，用相同的 prompt 重新调用 LLM。简单直接，适合违反率低、约束明确的场景。

**降级。** 验证失败时使用默认值或简化的处理逻辑。适合非关键路径。

**携带错误信息重试。** 将 ValidationError 的信息附加到下一次 prompt 中，明确告诉 LLM 哪个约束被违反了。这比盲目重试更高效，因为 LLM 获得了关于失败原因的反馈。

```python
from pydantic import BaseModel, ValidationError

def call_with_retry(prompt: str, model_class: type[BaseModel], max_retries: int = 3):
    last_error = None
    for attempt in range(max_retries):
        response = call_llm(prompt if last_error is None else
            f"{prompt}\n\n上次输出的验证错误：{last_error}")
        try:
            return model_class.model_validate_json(response)
        except ValidationError as e:
            last_error = str(e)
    raise RuntimeError(f"经过{max_retries}次重试仍未通过验证：{last_error}")
```

选择哪种策略取决于两个因素：验证失败的频率和业务的容错度。高频率失败暗示约束设计过于严格，低容错度要求必须重试直到成功。

## 同构性的边界

Prompt 和代码共享深层的设计原则，但这个同构性有清晰的边界。

**确定性 vs 概率性。** 代码的执行是确定性的——同一个输入总是产生同一个输出。prompt 的执行是概率性的——同一个 prompt 的多次调用产生不同的输出。所有依赖确定性的设计原则（幂等性、可重现性）在 prompt 中需要重新定义。

**静态分析的极限。** 代码的很多质量属性可以通过静态分析来保障。prompt 的质量属性（清晰性、无歧义性、完备性）目前无法被自动化工具检查。可以检查的是输出结构（通过 Pydantic 验证），但 prompt 文本本身的质量只能通过人工审查或 LLM 辅助审查来评估。

**组合的不可预测性。** 代码的组合是确定性的——`f(g(x))` 的行为完全由 `f`、`g` 和 `x` 决定。prompt 的组合不具备这个性质——两段"工作良好"的 prompt 拼接后可能产生意想不到的注意力交互。声明式方法通过类型组合（而非字符串拼接）来缓解这个问题，但不能完全消除。
