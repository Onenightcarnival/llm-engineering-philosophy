# 验证器作为运行时不变量

## 不变量的定义

不变量（invariant）是一个在系统生命周期中任何时刻都必须为真的条件。数据库的外键约束是不变量，账户余额不能为负是不变量，有向无环图中不存在环是不变量。不变量与普通验证的区别在于：普通验证检查的是"输入是否合法"，不变量保证的是"系统状态是否一致"。

在 LLM 应用中，Pydantic 的 `field_validator` 和 `model_validator` 承担了运行时不变量的角色。它们定义的是"无论 LLM 输出什么，解析后的数据必须满足什么条件"（"LLM 应该输出什么"是 description 的职责）。这个区分至关重要——前者是对 LLM 的指令，后者是对系统的保护。

## 三层验证架构

前三篇文章分别讨论了类型注解（结构约束）、JSON Schema（契约约束）和 Literal 类型（值域约束）。这些约束有一个共同特点：它们都可以在 Schema 层面表达，因此可以被 LLM 的 structured output 机制在生成时就强制执行。

但存在一类约束无法在 Schema 层面表达：字段之间的关系。

```python
from pydantic import BaseModel, Field, model_validator

class TimeRange(BaseModel):
    start_hour: int = Field(description="开始时间（24小时制）", ge=0, le=23)
    end_hour: int = Field(description="结束时间（24小时制）", ge=0, le=23)

    @model_validator(mode="after")
    def validate_time_order(self):
        if self.start_hour >= self.end_hour:
            raise ValueError(
                f"开始时间({self.start_hour})必须早于结束时间({self.end_hour})"
            )
        return self
```

`start_hour < end_hour` 这个约束涉及两个字段的关系，JSON Schema 无法表达。它只能作为运行时不变量存在，在 Pydantic 解析 LLM 输出时执行。如果 LLM 返回 `{"start_hour": 14, "end_hour": 10}`，类型检查和值域检查都会通过（两个值都是 0-23 之间的整数），但 model_validator 会拒绝它。

这构成了一个三层验证架构：

1. **Schema 层**：类型、枚举、值域范围。在 LLM 生成时强制执行（如果 API 支持 strict mode）。
2. **字段层**：`field_validator`，单个字段的复杂约束（格式校验、值变换）。在 Pydantic 解析时执行。
3. **模型层**：`model_validator`，字段之间的关系约束。在所有字段解析完成后执行。

三层从宽到严，每一层处理上一层无法表达的约束。这种分层对应传统数据库的约束体系：列类型约束 → CHECK 约束 → 触发器。

## field_validator：字段级不变量

`field_validator` 作用于单个字段，在该字段被赋值时执行。它适合表达单字段的复杂约束——那些超出 JSON Schema 表达能力的约束。

```python
from pydantic import BaseModel, Field, field_validator
import re

class CodeReview(BaseModel):
    file_path: str = Field(description="被审查的文件路径")
    severity: str = Field(description="问题严重程度：critical、warning 或 info")
    line_number: int = Field(description="问题所在行号", ge=1)
    suggestion: str = Field(description="修改建议")

    @field_validator("file_path")
    @classmethod
    def validate_file_path(cls, v: str) -> str:
        if not re.match(r'^[\w./\-]+$', v):
            raise ValueError(f"文件路径包含非法字符: {v}")
        if '..' in v:
            raise ValueError(f"文件路径不允许包含 '..': {v}")
        return v

    @field_validator("suggestion")
    @classmethod
    def validate_suggestion_not_empty(cls, v: str) -> str:
        if len(v.strip()) < 10:
            raise ValueError("修改建议过短，至少需要10个字符的具体描述")
        return v.strip()
```

`file_path` 的验证器防止路径遍历攻击——LLM 可能生成包含 `../` 的路径，如果不验证就直接用于文件操作，会造成安全漏洞。`suggestion` 的验证器保证建议有实质内容，而不是 LLM 生成的敷衍回复。

注意 `field_validator` 可以修改值（返回 `v.strip()`）。这意味着它同时承担了验证和标准化的双重职责。标准化是一种弱约束：不拒绝输入，而是将其映射到标准形式。去除首尾空白、统一大小写、规范化日期格式——这些标准化操作减少了下游系统需要处理的变异性。

## model_validator：跨字段不变量

`model_validator` 在所有字段解析完成后执行，可以访问整个模型实例。它是表达跨字段关系约束的唯一机制。

```python
from pydantic import BaseModel, Field, model_validator
from typing import Literal

class TaskDecomposition(BaseModel):
    original_task: str = Field(description="原始任务描述")
    subtasks: list[str] = Field(description="分解后的子任务列表")
    estimated_total_hours: float = Field(description="预估总工时")
    complexity: Literal["low", "medium", "high"] = Field(
        description="任务复杂度评估"
    )

    @model_validator(mode="after")
    def validate_consistency(self):
        # 子任务数量与复杂度应该一致
        if self.complexity == "low" and len(self.subtasks) > 5:
            raise ValueError(
                f"复杂度为 low 但子任务数量为 {len(self.subtasks)}，不一致"
            )
        # 工时与子任务数量应该合理
        if self.estimated_total_hours < len(self.subtasks) * 0.5:
            raise ValueError(
                f"预估工时({self.estimated_total_hours}h)相对于"
                f"{len(self.subtasks)}个子任务过少"
            )
        return self
```

这个验证器表达的是业务一致性：如果 LLM 把任务判断为"低复杂度"，但同时分解出了 8 个子任务，这种不一致暗示分析质量有问题。如果总工时少于子任务数量的一半（每个子任务至少半小时），工时估计可能不可靠。

这类一致性检查不能依赖 LLM 的"自我一致性"。LLM 在生成复杂结构化输出时，不同字段之间的一致性不是自动保证的——每个字段是按顺序生成的，后面的字段虽然能"看到"前面的值，但不意味着它会主动保持一致。显式的 model_validator 是唯一可靠的一致性保障。

## 验证失败的处理策略

不变量被违反时，系统需要一个明确的恢复策略。在传统软件中，不变量违反通常意味着 bug——开发者需要修复代码。在 LLM 应用中，不变量违反是正常操作的一部分——LLM 的概率性输出不保证满足所有约束。

三种常见的处理策略：

**重试。** 捕获 ValidationError，用相同的 prompt 重新调用 LLM。简单直接，但有成本（额外的 API 调用）和风险（重试可能返回相同的无效结果）。适合违反率低、约束明确的场景。

**降级。** 当验证失败时，使用默认值或简化的处理逻辑。例如，当情感分析的置信度字段验证失败时，返回一个默认的"不确定"结果。适合非关键路径。

**携带错误信息重试。** 将 ValidationError 的信息附加到下一次 prompt 中，明确告诉 LLM 哪个约束被违反了。这比盲目重试更高效，因为 LLM 获得了关于失败原因的反馈。

```python
def call_with_retry(prompt: str, model_class: type[BaseModel], max_retries: int = 3):
    last_error = None
    for attempt in range(max_retries):
        response = call_llm(prompt if last_error is None else
            f"{prompt}\n\n上次输出的验证错误：{last_error}")
        try:
            return model_class.model_validate_json(response)
        except ValidationError as e:
            last_error = str(e)
    raise RuntimeError(f"经过{max_retries}次重试仍未通过验证：{last_error}")
```

选择哪种策略取决于两个因素：验证失败的频率和业务的容错度。高频率失败暗示约束设计过于严格（需要放宽约束），低容错度要求必须重试直到成功。

## 与 Design by Contract 的对应

Bertrand Meyer 在 Eiffel 语言中提出的 Design by Contract 包含三个元素：前置条件（precondition）、后置条件（postcondition）和类不变量（class invariant）。

在 Pydantic + LLM 的语境中：

- **前置条件**对应 prompt 的构造逻辑——确保发送给 LLM 的输入是合理的（上下文不超长、必要信息完备）。
- **后置条件**对应 field_validator 和 model_validator——验证 LLM 的输出满足约定。
- **类不变量**对应 model_validator 中那些表达系统一致性的约束——在对象的整个生命周期中都必须为真。

区别在于：传统 DbC 中，违反契约意味着 bug；LLM 应用的 DbC 中，违反契约意味着需要重试或降级。契约的含义从"必须正确"变成了"验证并恢复"。这是 DbC 在概率性环境中的适配——契约的定义不变，契约的执行策略变了。
