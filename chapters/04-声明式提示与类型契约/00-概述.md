# 第四章 声明式提示与类型契约

提示工程应该是软件工程的一个分支，遵循软件工程的基本原则，同时发展出适应新范式的特有方法。

本章围绕声明式思维链的两个核心概念展开：**Code as Prompt**——用 Pydantic 模型同时充当类型定义、语义指令和输出验证器，代码本身就是 prompt；**Schema as Workflow**——Schema 的字段排列定义了 LLM 的推理路径，结构即工作流。在两个核心概念之上，第四篇汇总了将声明式思维链落地的设计原则和约束工具。

## 文章

- [从命令式到声明式：软件工程史的一个切面](01-从命令式到声明式.md) -- 从汇编到 C，从 jQuery 到 React，从自然语言 prompt 到类型化结构声明——每次抽象层级的提升都遵循相同的结构性规律。LLM 提示工程正处于这条规律的最新实例。
- [Code as Prompt](02-Code-as-Prompt.md) -- Pydantic 模型的三层语义（类型注解=约束层，Field description=语义层，Validator=不变量层）使代码本身成为 prompt。一处定义，三处生效。JSON Schema 是这个模型的机器可读投影，其表达力有边界，边界之外由运行时验证补全。
- [Schema as Workflow](03-Schema-as-Workflow.md) -- Schema 的字段排列定义了 LLM 的推理路径。每个字段承担一个推理步骤，字段间的显式依赖声明建立推理步骤间的逻辑连接。字段级单一职责是声明式思维链的前提条件。
- [声明式提示的设计原则](04-声明式提示的设计原则.md) -- 约束输出空间（Literal 类型与枚举粒度）、分离关注点（不同推理层有不同质量标准）、保持一致性（验证器作为运行时不变量）、处理验证失败（重试与降级策略）。

## 阅读顺序

第一篇建立历史背景：命令式到声明式的跃迁是一个反复发生的规律。第二、三篇是本章的核心：Code as Prompt 和 Schema as Workflow 分别从"代码即提示"和"结构即工作流"两个角度构建声明式思维链的完整画面。第四篇是实践指南：将核心概念落地时的设计原则和约束工具。

本章确立的组件级约束是[第五章](../05-架构与编排/00-概述.md)系统级架构的基础。
