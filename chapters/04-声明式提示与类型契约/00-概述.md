# 第四章 声明式提示与类型契约

提示工程不应该是一门手艺。它应该是软件工程的一个分支，遵循软件工程的基本原则，同时发展出适应新范式的特有方法。

从汇编到 C，从 jQuery 到 React，从自然语言 prompt 到 Code as Prompt——每次抽象层级的提升都遵循相同的结构性规律：命令式让位于声明式。如果 prompt 应该是声明式的，那么类型系统就是最好的声明工具。Pydantic 的 BaseModel 不仅是数据验证工具，更是一种面向 LLM 的领域特定语言。

本章将"提示工程是软件工程"的立场与"类型系统是最强约束传播机制"的技术主张合为一体：先论证为什么要声明式，再展开用什么工具实现。

## 文章

- [从命令式到声明式：软件工程史的一个切面](01-从命令式到声明式.md) -- 从汇编到 C，从 jQuery 到 React，从自然语言 prompt 到 Code as Prompt——每次抽象层级的提升都遵循相同的结构性规律。这个转变是软件工程史上命令式到声明式跃迁的最新实例。理解历史模式有助于做出正确的战略判断。
- [提示与代码的同构性](02-提示与代码的同构性.md) -- 一段好的 prompt 和一段好的代码在结构上共享深层的设计原则：单一职责、关注点分离、最小惊讶、显式优于隐式。这些原则本质上是"规格说明原则"——适用于所有向执行者传达意图的精确沟通。同构性的边界同样重要：确定性 vs 概率性、静态分析的极限、组合的不可预测性。
- [Pydantic 作为提示 DSL](03-Pydantic作为提示DSL.md) -- Pydantic 的 BaseModel 不仅是数据验证工具，更是一种面向 LLM 的领域特定语言。类型注解是约束层，Field description 是语义层，validator 是不变量层。三者共同构成一套完整的 prompt 规格说明。
- [Literal 类型与决策空间的显式化](04-Literal类型与决策空间.md) -- 枚举的本质是对问题空间的精确建模。将输出空间从无限压缩到有限，同时降低了 LLM 的输出熵，提高了下游系统的可处理性。
- [JSON Schema：机器可读的契约](05-JSON-Schema作为契约.md) -- Schema 的本质是调用者与 LLM 之间的契约，而非单纯的"输出格式要求"。契约的本质是双方的权利与义务，这个视角改变了 Schema 的设计方式。Pydantic 到 JSON Schema 的投影是有损的，这种有损性恰好标记了声明式与命令式的边界。
- [验证器作为运行时不变量](06-验证器作为运行时不变量.md) -- field_validator 和 model_validator 定义的是"无论 LLM 输出什么，系统状态必须满足什么条件"。三层验证架构（Schema 层、字段层、模型层）与 Design by Contract 的思想一脉相承。

## 阅读顺序

前两篇建立认知框架：为什么命令式到声明式的跃迁是一个反复发生的规律，以及 prompt 与代码为什么共享深层的设计原则。后四篇是技术展开：Pydantic 作为 prompt DSL、Literal 类型压缩输出空间、JSON Schema 作为机器可读契约、验证器作为运行时不变量。

本章确立的组件级约束是[第五章](../05-架构与编排/00-概述.md)系统级架构的基础。
