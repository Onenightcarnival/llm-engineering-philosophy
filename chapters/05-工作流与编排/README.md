# 第五章 工作流与编排

## 核心问题

当任务复杂度超出单次 LLM 调用的承载能力，就需要将任务分解并编排。问题在于：编排应该在什么层面发生，由谁来控制。

## 计划文章

- **隐式编排与显式编排** -- 隐式编排通过结构定义（如 Pydantic 的字段顺序和嵌套）驱动推理流程；显式编排通过 DAG 或状态机定义执行图。两者各有适用场景，但隐式编排被严重低估了。
- **多轮交互的状态机模型** -- 多轮对话本质上是一个状态机。每轮交互是一次状态转移。用状态机的形式化视角重新审视对话系统设计。
- **错误传播与补偿机制** -- LLM 调用会失败，而且失败的方式比传统 API 更多样（格式错误、语义偏离、幻觉）。如何设计重试、降级和补偿策略。
- **为什么大多数 LLM 编排框架过度设计了** -- 一篇观点文章。LangChain 式的抽象层级膨胀、概念过载和不必要的间接性，与"做最简单的能工作的事情"原则的冲突。
