# 第四章 类型系统与契约

## 核心问题

类型系统是人类在软件工程中发明的最强大的约束传播机制之一。当 LLM 的输出从自由文本变为结构化数据，类型系统的力量得以在这个新领域释放。

第三章确立了"提示工程是软件工程"的立场，本章是这个立场的直接技术展开：如果 prompt 应该是声明式的，那么类型系统就是最好的声明工具。

## 文章

- [Pydantic 作为提示 DSL](01-Pydantic作为提示DSL.md) -- Pydantic 的 BaseModel 不仅是数据验证工具，更是一种面向 LLM 的领域特定语言。类型注解是约束层，Field description 是语义层，validator 是不变量层。三者共同构成一套完整的 prompt 规格说明。
- [JSON Schema：机器可读的契约](02-JSON-Schema作为契约.md) -- Schema 不是"输出格式要求"，而是调用者与 LLM 之间的契约。契约的本质是双方的权利与义务，这个视角改变了 Schema 的设计方式。Pydantic 到 JSON Schema 的投影是有损的，这种有损性恰好标记了声明式与命令式的边界。
- [Literal 类型与决策空间的显式化](03-Literal类型与决策空间.md) -- 枚举不是限制，是对问题空间的精确建模。将输出空间从无限压缩到有限，同时降低了 LLM 的输出熵，提高了下游系统的可处理性。枚举粒度的权衡与信息论中的量化问题同构。
- [验证器作为运行时不变量](04-验证器作为运行时不变量.md) -- field_validator 和 model_validator 定义的不是"LLM 应该输出什么"，而是"无论 LLM 输出什么，系统状态必须满足什么条件"。三层验证架构（Schema 层、字段层、模型层）与 Design by Contract 的思想一脉相承。

## 本章的推演逻辑

四篇文章构成一条从工具选型到约束分层的推演链：

第一篇回答"用什么工具"——论证 Pydantic BaseModel 作为 prompt DSL 的独特定位，以及它的三层语义结构如何同时服务于类型定义、prompt 指令和输出验证。第二篇将视角从 Python 运行时扩展到协议层面，讨论 JSON Schema 作为机器可读契约的本质，以及从 Pydantic 到 JSON Schema 的有损投影如何划定声明式约束的边界。第三篇聚焦于一种特殊但极其重要的约束形式——Literal 类型将无限输出空间压缩为有限决策空间的机制，以及粒度选择的工程权衡。第四篇回到运行时层面，讨论 validator 如何表达 JSON Schema 无法承载的跨字段关系约束，构成系统不变量的最后防线。

本章与[第三章](../03-提示工程作为软件工程/00-概述.md)的关系是"原则到工具"：第三章论证了声明式方法的方向，本章提供了这个方向上的具体技术载体。本章与[第五章](../05-架构模式/00-概述.md)的关系是"组件到系统"：本章讨论的是单次 LLM 调用的输入输出约束，第五章将视角扩展到包含多个 LLM 调用的系统级架构。
