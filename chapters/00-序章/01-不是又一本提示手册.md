# 不是又一本 prompt 手册

## 一个被反复回答的问题

"大模型应用开发"这个领域正在被大量内容淹没。教程、速查表、框架文档、最佳实践合集、awesome 列表——它们有一个共同特征：默认读者的核心诉求是"告诉我怎么做"。

怎么写一个好的 prompt。怎么搭一个 RAG 系统。怎么用 LangChain 编排工作流。怎么做 Agent。

这些问题不是不值得回答，但它们共享一个盲区：没有人退后一步问，当软件系统的核心组件从确定性函数变成概率性语言模型，软件工程本身需要做出什么调整。

这是一个"怎么想"的问题，不只是"怎么做"。

## 这本书不是什么

**不是 prompt 手册。** 不会教你 few-shot 怎么排列、system prompt 怎么措辞、temperature 怎么调。这些是战术层面的技巧，有价值，但不是本书关注的层面。本书关注的是：prompt 这个东西在软件工程中应该被如何定位，它和类型系统、测试框架、架构模式之间是什么关系。

**不是框架文档。** 不会手把手教你 LangChain 的 chain 怎么接、LlamaIndex 的 retriever 怎么配、CrewAI 的 agent 怎么定义。框架会过时，对框架背后设计决策的理解不会。本书讨论的是框架应该解决什么问题，而不是某个框架的 API 怎么调用。

**不是技术选型指南。** 不会列出"2025 年最值得关注的 10 个 LLM 框架"这样的清单。技术选型的核心不在于横向比较功能列表，而在于理解自己的约束条件和决策标准。本书提供的是决策框架，不是决策结论。

**不是学术综述。** 不会系统性地梳理 Transformer 的演进历史、各种注意力机制的对比、或者 RLHF 的技术细节。这些内容有专门的论文和教材覆盖。本书的起点是：假设你已经知道大模型能做什么，现在的问题是怎么围绕它构建可靠的软件系统。

**不是中立的。** 有明确的偏好、明确的判断、明确的品味。具体是什么，正文会展开。

## 这本书是什么

一个人对"大模型时代的软件工程应该是什么样子"这个问题的系统性回答。

"一个人"意味着：这本书表达的是个人观点，不是行业共识。背后的偏见和局限在[个人立场声明](02-个人立场.md)中显式标注，供读者自行校准。

具体而言，这本书试图做三件事：

**建立认识论地基。** 大模型到底是什么？这里从计算理论和语言哲学的角度来回答。自回归生成意味着什么？概率性输出和确定性函数的根本差异在哪里？这些看似抽象的问题，直接决定了后续每一个工程决策的方向。不理解概率性输出的本质，就会把不确定性当 bug 来修，而这条路没有终点。

**构建方法论框架。** 在认识论的基础上，推导出一套面向不确定性环境的工程方法论。从人机协作（第三章）到声明式提示与类型契约（第四章），从架构与编排（第五章）到测试评估与可观测性（第六章）——每一章都围绕一个核心主张展开论证。

**跨领域迁移。** 软件工程中的决策困境，和计算数学中的收敛性问题，往往具有相同的数学结构。识别这种结构是实用的分析工具。当你意识到 prompt 优化和数值迭代法面对的是同一类收敛性问题时，数值分析中积累了几十年的判据和技巧就可以直接迁移过来。

## 为什么是现在

不是因为大模型是热门话题。

而是因为一个特殊的时间窗口正在打开：软件工程的基本范式正在经历一次罕见的结构性变化，而这次变化尚未被充分理解。

过去六十年，软件工程建立在一个隐含假设上：程序的行为是确定性的。给定相同的输入，程序产生相同的输出。测试、调试、形式化验证——整套工程方法论都以这个假设为地基。即便引入了并发、分布式系统、最终一致性等"不确定性"，那也是实现层面的复杂性，不是语义层面的不确定性。一个分布式数据库在某个时间点可能返回过期数据，但它不会"创造"数据。

大模型打破了这个假设。LLM 是一个概率性的生成器。相同的输入可能产生不同的输出，而且不同的输出可能都是"合理"的。这意味着软件工程六十年来积累的方法论——至少其中依赖确定性假设的那一部分——需要被重新审视。

这个审视工作刚刚开始。行业目前的状态是：大量的实践跑在理论前面，大量的框架在没有清晰方法论指导的情况下被构建和使用，大量的"最佳实践"是从有限案例中归纳出的经验法则而非从原理推导出的方法论。

这本书试图从原理出发，而不是从经验出发。

## 全书结构

[第一章](../01-认识论/00-概述.md)和[第二章](../02-不确定性与决策/00-概述.md)是地基：大模型的本质是什么，如何在不确定性环境下做工程决策。[第三章](../03-人机协作的软件过程/00-概述.md)到[第六章](../06-测试评估与可观测性/00-概述.md)是方法论在具体领域的展开：工作方式、声明式约束、架构编排、测试可观测性。[第七章](../07-反模式与陷阱/00-概述.md)是反面教材。[第八章](../08-终章/00-概述.md)回到哲学层面。

起点是一个事实判断（大模型改变了软件系统的基本性质），终点是一个价值判断（软件工程应该向什么方向演化）。

## 阅读建议

如果你是有经验的软件工程师，刚开始接触大模型应用开发，建议从第一章和第二章开始。这两章提供的认识论和方法论框架，会改变你看待后续所有技术细节的方式。

如果你已经在做大模型应用开发，遇到了"prompt 不稳定""系统不可靠""不知道怎么测试"等具体问题，可以直接跳到对应章节，但建议回头补读前两章。很多工程问题的根源不在技术层面，在认知层面。

如果你对框架和工具更感兴趣，这本书可能会让你不舒服。它会反复追问"为什么"而不是直接告诉你"怎么做"。这是刻意的选择：工具会过时，思维方式不会。

不同意书中的判断是完全正常的。事实上，如果你在每个问题上都同意作者的观点，那说明这本书对你的价值有限——它没有提供任何你原本不具备的视角。有价值的阅读体验是：在某些点上产生强烈的共鸣，在另一些点上产生强烈的反对，而反对的过程本身迫使你澄清自己的立场。
