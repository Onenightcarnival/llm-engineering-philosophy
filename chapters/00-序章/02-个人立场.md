# 个人立场声明

## 为什么需要这篇声明

每一本技术书籍背后都有作者的偏见。区别在于：有些书假装没有偏见，有些书把偏见明确写出来。

前者更危险。读者在不知不觉中接受了一套特定的价值预设，却以为自己读到的是客观事实。后者至少给了读者一个选择：在了解作者立场的前提下，自主决定哪些判断值得采信，哪些需要打折。

这篇声明的目的不是自我辩护，而是提供校准信息。知道望远镜的光轴偏了多少度，观测数据就仍然有用。

## 学术训练的烙印

计算数学和数据科学的双重训练，在思维方式上留下了不可逆的印记。

计算数学训练的核心不是具体的算法，而是一种将模糊问题精确化的直觉。面对一个含混的工程问题，第一反应是：定义变量、建立约束、判断解的存在性和唯一性。这种直觉在本书中的投射是：对"大致正确"的不满足，对模糊概念的不容忍，对"说不清但好像有道理"的表述的本能警觉。

更具体地说，计算数学中有两个概念深刻影响了本书的论证方式：

**收敛性。** 一个迭代过程是否在接近目标？接近的速度是多少？在什么条件下会发散？这套思维框架在讨论 prompt 优化、架构演进、系统调优时反复出现。一个没有收敛判据的优化过程，和一个没有止损条件的投资策略一样，是不负责任的。

**数值稳定性。** 理论上等价的两个计算路径，在有限精度算术下可能产生截然不同的结果。这个事实的工程含义是：语义等价不等于行为等价。两种"意思相同"的 prompt 表述，可能导致输出质量的数量级差异。对数值稳定性的敏感，让作者天然地不信任"理论上应该一样"这种说法。

数据科学的训练则提供了另一个维度：对不确定性的量化能力和对实验方法的尊重。"这个方案好不好"不是一个可以凭直觉回答的问题——好多少，在什么置信水平下，样本量是否足够，有没有混淆变量。这种思维方式在第七章（测试与可靠性）和第八章（数据科学与计算思维）中集中体现。

这两个学科的交叉地带，正好是大模型应用开发的核心困难所在：如何在概率性系统中做出可靠的工程决策。

## 工程经验的塑造

一线大厂的生产系统和独立的框架设计经历，留下的最重要的东西不是特定的技术栈知识，而是两种能力：

**对抽象层级的直觉。** 什么时候应该抽象，抽象到什么程度，在哪里放置抽象边界——这些判断不来自设计模式教科书，来自在大规模系统中反复犯错和修正的经历。抽象过度和抽象不足都有代价，而最优抽象层级取决于具体的约束条件，不存在普适的答案。

**对框架的祛魅。** 参照 Spring 的设计哲学独立设计过 Python 框架之后，对 IoC/DI、声明式配置、AOP、分层架构等模式的理解不再是"学会了怎么用"，而是"知道了为什么这么设计，以及这种设计在什么条件下会失效"。这种视角在本书中的投射是：对任何框架的第一反应不是"怎么用"，而是"它做了什么设计决策，这些决策的代价是什么"。

由此衍生出本书的一个核心态度：对"框架崇拜"的批判。

框架崇拜的症状是：用框架的抽象替代对问题的理解。不是先理解问题再选择工具，而是先选择框架再把问题塞进框架的模型里。当 LangChain 定义了 chain、agent、tool 这套抽象时，使用者开始用这套抽象来思考问题，而不是用问题本身的结构来思考问题。框架的抽象变成了认知的牢笼。

这不是说框架没有价值。框架的价值在于固化经过验证的设计决策，降低重复工作的成本。但框架不能替代思考。一个不理解 RAG 本质的开发者，用 LlamaIndex 写出的系统不会比手写的更可靠——只是更快地达到了一个不理解的状态。

## 决策哲学的迁移

十余年投资实战提供的不是金融知识，而是一套经过真金白银验证的决策框架。这套框架在本书中的迁移不是修辞，是方法论。

**Strategy 大于 Analysis。**

分析可以无穷无尽，但战略层面的判断才决定最终结果。这个原则在投资中的表现是：2022 年英伟达跌至 100 美元附近时的判断不是来自更精细的财务模型，而是来自对 AI 产业结构的理解——算力是这一轮技术革命的基础设施，英伟达在算力层的垄断地位短期内无法被替代。战略正确时，买入价格偏差 20% 无关紧要；战略错误时，即便买在最低点也会亏损。

迁移到软件工程：架构决策的重要性远大于实现细节。选择把 LLM 定位为核心引擎还是胶水层，这个战略判断的影响远大于 prompt 怎么写、用哪个框架、调什么参数。花 80% 的时间在战略层面想清楚，比花 80% 的时间在战术层面精雕细琢更有价值。

**知行合一。**

知道应该做什么和真正做到之间的距离，比大多数人想象的要远得多。2025 年 4 月关税战期间标普回测 4800 点，所有的基本面分析都指向"这是买入机会"，但恐惧让大多数人做出了相反的选择。知道和做到之间，隔着人性的弱点：恐惧、从众、对确定性的渴望、对复杂方案的迷恋。

迁移到软件工程：知道"应该用简单方案"和真正选择简单方案之间，同样隔着人性的弱点。复杂方案给人一种"考虑周全"的安全感。引入一个框架比自己写 50 行代码显得更"专业"。用微服务架构比用单体应用显得更"先进"。这些心理倾向和投资中的从众心理在结构上是同构的：都是用表面的复杂性来对冲内心的不确定感。

**不确定性是约束条件，不是敌人。**

波动性不是市场的缺陷，是市场的本质属性。试图消除波动性（比如通过过度对冲）的代价通常高于接受波动性。正确的做法是：承认波动性存在，量化它，在给定波动性水平下做出最优决策。

迁移到软件工程：LLM 的概率性输出不是缺陷，是本质属性。试图把 LLM 变成确定性函数（比如通过极低的 temperature 和极严格的约束）的代价通常是丧失 LLM 最有价值的能力。正确的做法是：承认不确定性存在，用类型系统和测试框架约束它到可接受的范围，在给定不确定性水平下设计系统。

这三条原则在第二章展开详细论证。

## 技术偏好的显式声明

以下偏好贯穿全书，在此显式列出，供读者在阅读时作为校准参考：

**声明式优于命令式。** 描述"要什么"比描述"怎么做"更不容易出错，更容易验证，更容易组合。SQL 优于手写循环遍历表，类型定义优于运行时检查，Pydantic 模型优于手动 JSON 解析。这个偏好在第三章（Code as Prompt）和第四章（类型系统与契约）中集中体现。

**类型系统优于运行时检查。** 能在编译期（或定义期）捕获的错误，不要留到运行时。类型注解不是可选的文档，是可执行的规格说明。在 LLM 应用中，Pydantic 模型定义本身就是最好的 prompt——这个观点在第四章详细论证。

**组合优于继承。** 通过组合小而专一的组件构建复杂系统，而不是通过继承层次来扩展功能。这个原则在传统软件工程中已经是共识，在 LLM 应用中更加重要——因为 LLM 调用本身就是天然适合组合的独立单元。

**显式依赖优于隐式约定。** 系统的每一个依赖关系都应该在代码中可见，而不是隐藏在配置文件、环境变量或框架的"约定优于配置"中。在大模型应用中，这意味着 prompt 的每一个组成部分、模型的每一个参数、上下文的每一个来源都应该是显式可追踪的。

**简单方案优于"看起来高级"的方案。** 如果 50 行代码能解决的问题，不要用一个框架。如果一个 API 调用能完成的任务，不要用 Agent 编排。复杂性不是免费的，每一层抽象都有认知成本和维护成本。选择复杂方案的唯一正当理由是：简单方案确实无法满足需求，而且这一点已经被证明而非推测。

**结构驱动优于流程驱动。** 好的结构自然地引导出正确的流程，反过来则不成立。先定义数据结构和接口契约，流程自然浮现；先画流程图再填充实现，通常导致结构和流程的不匹配。

这些偏好不是绝对真理。每一条都有适用边界，都有反例。声明它们的目的不是要求读者接受，而是让读者知道：当本书在两个同样合理的方案之间做出选择时，选择背后的倾向是什么。

## 这些偏见的局限性

任何立场都有盲区，诚实地标注盲区是学术训练的基本要求。

**数学训练的偏见：过度形式化。** 将模糊问题精确化是好习惯，但不是所有问题都适合精确化。有些工程问题的最优解法是"差不多就行"，过度追求形式化反而增加了不必要的复杂性。本书可能在某些地方过度使用了数学类比，读者需要自行判断哪些类比确实揭示了结构性同构，哪些只是作者的思维惯性。

**框架设计者的偏见：对框架的过度批判。** 造过框架的人容易对别人的框架过于苛刻。"我知道框架可以做得更好"这个想法，有时候会演变成"所有现有框架都不够好"。实际上，在大多数场景下，一个"不够好"的框架仍然比从零开始更经济。本书对框架的批判应该被理解为"在选择框架之前先理解问题"，而不是"不要使用框架"。

**投资者的偏见：过度强调决策。** 投资是一个决策密度极高的活动，每一个判断都直接对应盈亏。但软件工程不完全是这样——很多时候，"先做起来再说"比"想清楚再动手"更有效。本书可能在某些地方过度强调了"先想清楚"的重要性，对快速原型和迭代试错的价值有所低估。

**独立性的偏见：对协作的低估。** 独立管理资金、独立设计框架、独立写作——这种独立性是作者的特质，也是局限。软件工程在绝大多数场景下是团队活动，而团队协作中的很多问题（沟通成本、共识建设、妥协的艺术）在本书中的讨论深度不足。

标注这些盲区不是为了预防批评，而是为了帮助读者更准确地使用这本书：在作者的优势区域（形式化思维、架构决策、不确定性管理）给予更多信任，在作者的盲区（快速原型、团队协作、"差不多就行"的工程判断）保持更多警惕。

## 一个承诺

这本书中的每一个事实陈述，作者会尽力确保其准确性。每一个价值判断，作者会明确标注为价值判断而非事实陈述。每一个代码示例，要么是可运行的，要么明确标注为伪代码。每一个跨领域类比，要么指向具体的结构性同构和可操作的工程启示，要么不会出现。

不会写空话。不会用"可能""也许""在某些情况下"来对冲每一个判断。不会为了面面俱到而稀释观点的锋利度。

该说的说清楚，不该说的不说。
