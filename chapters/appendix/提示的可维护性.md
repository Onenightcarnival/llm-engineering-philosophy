# Prompt 的可维护性问题

## 生产系统中的 prompt 不是写完就结束的

一段 prompt 在原型阶段的生命周期是几天。在生产系统中的生命周期是几个月到几年。原型阶段的 prompt 只需要"能跑就行"；生产系统的 prompt 需要被维护、版本控制、回归测试、跨团队协作、以及在模型升级时被重新验证。

这意味着生产环境的 prompt 面临的核心问题是"怎么维护下去"。这个问题在传统软件工程中已经被深入研究了几十年——代码的可维护性是软件工程的中心课题之一。但 prompt 的可维护性几乎没有被系统性地讨论过，因为行业还停留在"prompt 是手艺品"而非"prompt 是工程制品"的认知阶段。

## 代码面临的可维护性问题，prompt 一个不少

把生产代码面临的经典可维护性挑战逐一映射到 prompt 上，会发现 prompt 不仅面临所有相同的问题，而且由于缺少编译器、类型检查器和 IDE 支持，每个问题的严重程度都被放大了。

| 挑战 | 代码中的解决方案 | Prompt 现状 | 差距 |
|------|----------------|------------|------|
| 版本管理 | Git + 语义化版本 + diff + blame | 嵌入代码字符串或散落在配置文件中 | 缺少 prompt 专用的语义级 diff 工具，缺少 prompt 变更对输出影响的量化评估 |
| 回归测试 | 单元测试 + 集成测试 + CI/CD | 手动运行几个案例看看效果 | 需要统计显著性检验而非简单的通过/失败判定，需要区分随机性和真实回归 |
| 重构 | IDE 重命名、提取函数、类型检查 | 手动搜索替换，全凭记忆追踪依赖 | prompt 中的引用关系是语义层面的，没有工具能自动检测断裂的引用 |
| 模块化 | 函数、类、模块、包 | 通常是一整段文本 | prompt 的分段拼接不等于模块化，各段之间通过注意力机制耦合，不像函数调用有清晰的接口隔离 |
| 文档与可读性 | 注释、文档字符串、类型注解 | prompt 本身就是自然语言，看似不需要文档 | 意图和实际效果之间有语义鸿沟：写的是什么清楚，为什么这样写、改了哪个词会有什么影响没有记录 |

**版本管理。** 代码有 Git，有 diff，有 blame，有 merge conflict 的解决机制。prompt 通常以字符串字面量嵌入在代码中，或者存储在配置文件甚至数据库里。它的变更历史经常不可追溯——谁在什么时候为什么改了这段 prompt？"把'请简短回答'改成了'请用不超过三句话回答'"这个变更看起来微不足道，但它可能在某些边界案例上导致输出质量的大幅波动。

**回归测试。** 代码有单元测试和集成测试，改了一行代码可以立即知道是否破坏了现有功能。prompt 的修改没有这样的安全网。修改了 prompt 中的一个措辞，唯一的验证手段是重新运行整个 LLM 评估流程——这通常需要几分钟到几小时，而且由于 LLM 的概率性，即使 prompt 没有变化，两次运行的结果也可能不同。如何区分"prompt 变更导致的差异"和"LLM 固有的随机性"？这个问题在多数团队中没有被正式回答。

**重构。** 代码有 IDE 的重构支持——重命名一个变量、提取一个函数、修改一个接口——工具会自动处理所有相关引用。prompt 没有这样的工具支持。一段 prompt 引用了系统中其他组件的输出（比如"根据上面的检索结果"），这种引用关系是隐式的、无法被工具追踪的。当被引用的组件变更了输出格式，prompt 的维护者可能完全不知道。

**模块化。** 代码可以被分解为函数、类、模块，每个单元有清晰的接口和职责。prompt 通常是一个扁平的字符串，缺乏内部结构。当 prompt 增长到几百字甚至几千字时，它变成了一个"大泥球"——修改任何部分都可能影响其他部分，因为 LLM 的注意力机制不保证只关注 prompt 中与当前任务相关的部分。

## 没有编译器的世界

代码的可维护性在很大程度上依赖于工具链的支撑。编译器捕获类型错误，linter 检查风格问题，formatter 统一格式，IDE 提供导航和重构——这些工具构成了代码质量的自动化防线。

prompt 没有这些工具。

更准确地说，prompt 没有等价的工具，因为 prompt 的"正确性"不像代码那样可以被形式化定义。一段代码是否有类型错误，有明确的是/否判定。一段 prompt 是否"正确"——这取决于 LLM 对它的解读，而 LLM 的解读是概率性的、随模型版本漂移的。

这个工具链的缺失有具体的工程后果：

**错误反馈的延迟。** 代码中的类型错误在写下那一行的瞬间就被 IDE 标红了。prompt 中的"错误"（措辞歧义、约束遗漏、依赖断裂）只有在 LLM 执行并产生不符合预期的输出时才会被发现——而且由于输出的概率性，同一个"错误"可能只在部分调用中显现，增加了排查难度。

**修改的不可预测性。** 修改一行代码的影响范围可以通过类型系统和调用链追踪来精确确定。修改 prompt 中的一个词的影响范围是不可预测的——因为 LLM 的注意力机制是全局的，prompt 中任何 token 的变化都可能影响输出中任何 token 的概率分布。这就是[第一章](../01-认识论/autoregressive-generation.md)讨论的"数值稳定性"问题的工程化体现：语义等价的修改不一定产生行为等价的结果。

**知识的不可传递性。** 一段好代码是自解释的——类型签名、变量名、函数结构共同传达了代码的意图。一段 prompt 的"好"在很大程度上是隐性知识——"为什么用'分析'而不是'评估'？""为什么示例放在指令之前而不是之后？""为什么这个约束要重复两次？"这些决策的理由通常只存在于编写者的脑中。当编写者离开，知识就丢失了。

## 声明式方法如何改善可维护性

Code as Prompt 的核心价值之一是将 prompt 的可维护性问题转化为代码的可维护性问题——而后者已经有了成熟的解决方案。

**版本管理回归标准流程。** 当 prompt 的核心逻辑用 Pydantic 模型表达时，它就是 Python 代码，自然地进入 Git 版本控制。字段的增删改可以通过标准的 code review 流程来管控。diff 清晰地显示了什么变了：新增了一个字段？修改了一个 Literal 枚举？调整了字段顺序？这些变更的语义在 diff 中一目了然，而不像自然语言 prompt 的 diff 那样需要仔细阅读才能理解影响。

```python
# 版本演进的可追踪性示例

# v1.0: 初始版本
class CustomerFeedbackV1(BaseModel):
    sentiment: Literal["positive", "negative"]
    summary: str = Field(description="反馈摘要")

# v1.1: 增加 neutral 选项（非破坏性变更）
class CustomerFeedbackV1_1(BaseModel):
    sentiment: Literal["positive", "negative", "neutral"]  # CHANGED
    summary: str = Field(description="反馈摘要")

# v2.0: 增加置信度和关键词（破坏性变更，输出结构改变）
class CustomerFeedbackV2(BaseModel):
    sentiment: Literal["positive", "negative", "neutral"]
    confidence: float = Field(ge=0.0, le=1.0, description="判断置信度")  # NEW
    summary: str = Field(description="反馈摘要")
    keywords: list[str] = Field(                                          # NEW
        min_length=1, max_length=5,
        description="支撑判断的关键词"
    )
```

在这个演进过程中，v1.0 到 v1.1 是非破坏性的——增加了一个枚举值，下游系统如果用 `if sentiment == "positive"` 的逻辑，不会被影响。v1.1 到 v2.0 是破坏性的——输出结构增加了新字段，下游系统需要更新。这种变更的分类与 API 版本管理的 semver 约定完全一致。

**回归测试变得可行。** Pydantic 模型的约束逻辑可以独立于 LLM 进行测试——这在前面已经展示过。更重要的是，模型定义本身可以作为回归测试的基线：如果一个 LLM 的输出通过了 v1.0 的验证但没有通过 v2.0 的验证，这意味着 LLM 的行为和新版本的期望之间存在差距，需要调整 prompt 或接受降级。

```python
from pydantic import ValidationError
from typing import Type


def regression_check(
    llm_outputs: list[str],
    old_schema: Type[BaseModel],
    new_schema: Type[BaseModel],
) -> dict:
    """
    Schema 演进的回归检查。

    用旧 schema 和新 schema 分别验证同一批 LLM 输出，
    量化 schema 变更对兼容性的影响。
    """
    results = {
        "total": len(llm_outputs),
        "old_schema_pass": 0,
        "new_schema_pass": 0,
        "both_pass": 0,
        "regression": 0,  # 旧通过新不通过 = 回归
    }

    for output in llm_outputs:
        old_ok = True
        new_ok = True
        try:
            old_schema.model_validate_json(output)
        except (ValidationError, Exception):
            old_ok = False

        try:
            new_schema.model_validate_json(output)
        except (ValidationError, Exception):
            new_ok = False

        if old_ok:
            results["old_schema_pass"] += 1
        if new_ok:
            results["new_schema_pass"] += 1
        if old_ok and new_ok:
            results["both_pass"] += 1
        if old_ok and not new_ok:
            results["regression"] += 1

    return results
```

**重构有了类型安全保障。** 当 prompt 的输出结构用 Pydantic 模型定义时，下游代码对输出的引用就是标准的 Python 属性访问——`result.sentiment`、`result.confidence`。IDE 的"查找所有引用"功能可以精确定位所有使用了特定字段的位置。重命名一个字段时，IDE 会自动更新所有引用点。这和重构普通 Python 代码完全一样——因为它就是普通 Python 代码。

**模块化通过类型组合实现。** 独立的类型定义可以通过嵌套和引用来组合。`Entity` 类型可以被 `DocumentAnalysis` 引用，也可以被 `EmailExtraction` 引用。每个类型定义的职责清晰、边界明确。修改 `Entity` 的字段会影响所有引用它的上层结构——而这种影响可以通过类型检查自动发现。

## Prompt 的"技术债务"

技术债务的概念在软件工程中有精确的含义：为了短期目标（快速交付、原型验证）而做出的技术决策，在长期会产生额外的维护成本。技术债务不是坏事——它是有意识的权衡。但不被意识到的技术债务是灾难。

Prompt 领域存在大量不被意识到的技术债务。

**隐性依赖债务。** 一段 prompt 可能隐式依赖于特定模型的行为特征。"在 GPT-4 上效果很好"的 prompt 在 Claude 上可能完全不同。"在 2024 年 3 月的模型版本上效果很好"的 prompt 在模型更新后可能退化。这些依赖没有被显式记录，当它们断裂时（模型升级、供应商切换），定位问题的成本极高。这类债务的检测难度极大——它通常只在模型切换或升级的瞬间暴露。应对的策略是将模型特异性行为封装在适配层中，让 prompt 的核心逻辑通过类型声明来表达，使其与具体模型解耦。

**措辞依赖债务。** 经过反复调优的 prompt 往往有大量"因为某次实验发现这个措辞效果更好"的修改。比如"请确保"被改成了"务必"，因为某次实验显示后者效果更好，但没有记录实验条件和样本量。这些修改的理由没有记录，效果没有被隔离验证。当需要修改 prompt 的其他部分时，维护者不敢动这些"祖传措辞"——因为不知道改了之后会不会破坏某些微妙的平衡。这类债务几乎不可见，因为它的存在证据就是"一个词的选择"，而这个选择背后的理由已经丢失了。解决之道是用声明式约束替代祈使句——`Field(min_length=1)` 比"请确保至少包含一项"更稳定，因为前者的语义是精确的、模型无关的。

**上下文污染债务。** 随着 prompt 的迭代，新的指令被追加，旧的指令没有清理。一段 prompt 可能从最初的 200 字增长到 1500 字，其中包含多段矛盾的格式要求、冗余的指令（同一个约束被用不同的措辞表达了三次），或者过时的指令（引用了不再存在的系统组件）。在传统代码中，死代码和矛盾逻辑会被编译器或 linter 检测到。在 prompt 中，这些问题默默地消耗注意力资源、增加 token 成本、并以不可预测的方式影响输出质量。这类债务的检测相对容易——prompt 的长度膨胀本身就是一个信号。治理手段是用 Pydantic 模型替代自然语言格式描述，确保每个约束有且仅有一个声明位置，消除重复和矛盾的可能性。

**结构侵蚀债务。** 最初清晰的 prompt 结构在多轮迭代后变成了"意大利面"——谁都不敢做大的修改。每次小修补都在已有结构上打补丁，最终整段 prompt 变成了考古遗迹：每一层措辞都是某个历史阶段的产物，但没有人记得哪一层对应哪个需求。这类债务在表面上最容易被观察到——只要 prompt 读起来让人困惑，结构侵蚀就已经发生了。修复的方式是定期将 prompt 重构为声明式定义，将结构逻辑从自然语言中提取到类型系统中，让类型定义承担结构表达的职责，让自然语言只承担补充说明的角色。

## 将 Prompt 视为代码的工程实践

承认 prompt 是工程制品而非手艺品之后，以下实践自然地从传统软件工程中迁移过来。

**实践一：prompt 的变更走 code review。** 每一次 prompt 修改都应该有对应的 pull request，说明变更的动机、影响范围和验证方法。对于声明式 prompt（Pydantic 模型），diff 是精确的；对于仍然是自然语言的 prompt 部分，至少应该在 PR 描述中说明"改了什么，为什么改，预期影响是什么"。

**实践二：prompt 的评估结果作为 CI 的一部分。** 在 CI 流水线中加入 prompt 评估：用一组固定的测试输入运行 LLM，检查输出是否满足约束，统计通过率。如果通过率低于阈值，CI 失败。这需要区分"随机波动"和"真实回归"——统计假设检验在这里有直接的应用价值。

**实践三：prompt 有明确的所有者。** 生产代码有 CODEOWNERS 文件，指定谁负责审核哪些文件的变更。prompt 应该有同样的机制。"这段 prompt 谁最熟悉""出了问题找谁"——如果这些问题的答案是"不知道"，那这段 prompt 的可维护性就是一个定时炸弹。

**实践四：自然语言 prompt 有结构化元数据。** 对于仍然必须使用自然语言的 prompt，至少应该附带结构化元数据：

```python
from dataclasses import dataclass, field
from datetime import date
from typing import Optional


@dataclass(frozen=True)
class PromptMetadata:
    """
    为自然语言 prompt 附加工程元数据。

    这个元数据是给维护者看的，LLM 不需要它。
    它将 prompt 的"隐性知识"显式化。
    """
    # 身份标识
    prompt_id: str
    version: str
    owner: str

    # 生命周期
    created_date: date
    last_modified_date: date
    last_validated_date: Optional[date]

    # 依赖
    target_models: tuple[str, ...]          # 在哪些模型上验证过
    depends_on: tuple[str, ...]             # 依赖的上游组件
    depended_by: tuple[str, ...]            # 下游消费者

    # 质量
    validation_pass_rate: Optional[float]   # 最近一次评估的通过率
    known_failure_patterns: tuple[str, ...] # 已知的失败模式

    # 决策记录
    design_decisions: tuple[str, ...] = field(default_factory=tuple)


# 示例
invoice_extraction_prompt_meta = PromptMetadata(
    prompt_id="invoice-extraction-v2",
    version="2.3.1",
    owner="data-team/yamato",
    created_date=date(2024, 6, 1),
    last_modified_date=date(2025, 1, 15),
    last_validated_date=date(2025, 1, 15),
    target_models=("gpt-4o-2024-08-06", "claude-3-5-sonnet-20241022"),
    depends_on=("ocr-pipeline/text-extraction",),
    depended_by=("accounting-service/invoice-processor",),
    validation_pass_rate=0.94,
    known_failure_patterns=(
        "多页发票的跨页金额提取失败率较高",
        "手写发票的日期格式解析不稳定",
    ),
    design_decisions=(
        "使用 few-shot 而非 zero-shot，因为发票格式变异大",
        "'金额' 字段使用 str 而非 float，因为需要保留货币符号和千分位",
    ),
)
```

这个元数据结构将原本散落在团队成员脑中的隐性知识变成了可追踪的显式记录。当维护者需要修改这段 prompt 时，`design_decisions` 告诉他们为什么 prompt 是这样设计的，`known_failure_patterns` 告诉他们需要特别关注哪些测试用例，`depends_on` 和 `depended_by` 告诉他们变更的影响范围。

## 可维护性的战略意义

回到本章的核心命题：提示工程是软件工程的一个分支。如果这个命题成立，那么软件工程中关于可维护性的所有认知都适用于 prompt。

软件工程几十年的经验告诉我们：一个系统 70% 的生命周期成本是维护成本，不是开发成本。prompt 的情况不会例外。一段 prompt 被写出来的时间是几小时，被维护的时间是几年。如果 prompt 的设计没有考虑可维护性，那么短期的开发效率提升会被长期的维护成本吞噬——而且利率很高。

声明式方法——用类型定义替代自然语言描述，用结构约束替代措辞技巧——本质上是将 prompt 从不可维护的文本制品转化为可维护的代码制品。这是工程经济学的必然结论。
