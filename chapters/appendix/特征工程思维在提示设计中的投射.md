# 特征工程思维在提示设计中的投射

## Prompt 设计就是特征工程

在传统机器学习中，特征工程是将原始数据转化为模型可利用的输入表示的过程。一个经验丰富的数据科学家知道：模型的上限往往不由算法决定，而由输入特征的质量决定。选择哪些特征输入模型、如何编码这些特征、如何处理特征之间的交互关系——这些决策对最终性能的影响，经常超过模型选择和超参数调优的影响。

LLM 应用中的 prompt 设计面临结构上同构的问题。每次 LLM 调用时，开发者需要决定：在有限的上下文窗口中放入什么信息。用户的原始输入、系统指令、历史对话、检索到的文档片段、结构化的元数据——这些信息共同构成了模型的"特征空间"。选择哪些信息放入 prompt、以什么格式组织、放在什么位置，本质上就是特征选择和特征工程。

这个类比不是修辞性的。它指向一套可操作的方法论：数据科学中积累了数十年的特征工程原则，可以直接迁移到 prompt 设计中。

## 上下文窗口作为特征预算

传统特征工程受到维度诅咒的约束——特征维度过高会导致模型性能下降（过拟合）和训练成本上升。开发者必须在信息量和维度之间取得平衡，这就是特征选择的核心问题。

LLM prompt 设计面临等价的约束：上下文窗口是有限的。即使当前模型的上下文窗口已经扩展到百万 token 级别，有限性的本质并未改变——只是预算变大了。更关键的是，上下文长度的增加并不意味着所有位置的信息都被平等利用。研究表明，模型对上下文中间位置的信息的利用效率低于开头和结尾。这意味着上下文窗口不是一个均匀的"特征空间"，而是一个具有位置权重的不均匀空间。

因此，prompt 设计中的"特征预算"不仅是总量的约束，还是分配的约束。高价值信息应该占据注意力集中的位置（开头和结尾），低价值但仍有用的信息可以放在中间，而无关信息应该被排除——无论上下文窗口是否还有剩余空间。

这与特征选择中"不是所有可用特征都应该使用"的原则完全一致。添加无关特征不是无害的——它不仅浪费空间，还引入噪声，稀释有用信号的权重。

## 信息增益视角

特征选择的经典方法之一是按信息增益（information gain）排序：一个特征对预测目标的信息贡献越大，它越应该被选入模型。信息增益的计算基于条件熵——加入这个特征后，目标变量的不确定性减少了多少。

将这个框架映射到 prompt 设计：一条上下文信息的价值，应该用它对输出质量的边际贡献来衡量。具体操作是 ablation study（消融实验）——逐一移除某条上下文信息，观察输出质量的变化。质量下降大的信息是高信息增益特征，应该保留；质量变化不大的信息是低信息增益特征，可以考虑移除。

```python
from dataclasses import dataclass


@dataclass
class ContextFeature:
    name: str
    content: str
    position: str  # "system", "prefix", "suffix"


def measure_information_gain(
    base_prompt: str,
    features: list[ContextFeature],
    test_inputs: list[str],
    quality_fn: callable,  # (prompt, input) -> float
) -> dict[str, float]:
    """通过消融实验衡量每个上下文特征的信息增益。

    对每个特征，比较包含它和移除它时的平均输出质量差异。
    差异越大，该特征的信息增益越高。
    """
    # 基线：所有特征都包含
    full_prompt = _assemble(base_prompt, features)
    baseline_scores = [quality_fn(full_prompt, inp) for inp in test_inputs]
    baseline = sum(baseline_scores) / len(baseline_scores)

    gains = {}
    for feature in features:
        reduced = [f for f in features if f.name != feature.name]
        reduced_prompt = _assemble(base_prompt, reduced)
        reduced_scores = [quality_fn(reduced_prompt, inp) for inp in test_inputs]
        reduced_avg = sum(reduced_scores) / len(reduced_scores)
        gains[feature.name] = baseline - reduced_avg  # 正值意味着移除后质量下降

    return gains


def _assemble(base: str, features: list[ContextFeature]) -> str:
    prefix = " ".join(f.content for f in features if f.position == "prefix")
    suffix = " ".join(f.content for f in features if f.position == "suffix")
    system = " ".join(f.content for f in features if f.position == "system")
    return f"{system}\n{prefix}\n{base}\n{suffix}"
```

这个方法的成本不低——每个特征需要一轮独立的评估。但它提供了做出有根据的裁剪决策的基础，而非凭直觉决定"这条信息可能有用"。[输出质量的统计度量](./输出质量的统计度量.md)讨论的质量度量方法为这里的 `quality_fn` 提供了具体实现。

## 特征交互效应

特征工程中一个被频繁低估的现象是特征交互：两个单独看都有价值的特征组合后可能产生冲突，两个单独看价值不大的特征组合后可能涌现新的预测能力。

Prompt 设计中存在完全对应的现象。系统指令中的"保持简洁"和检索到的详细文档片段之间存在冲突——前者要求压缩，后者提供了展开的素材，模型在两个矛盾的信号之间被迫折中，输出质量反而下降。这就是负交互效应。

另一方面，用户的历史偏好信息和当前查询的语义分析单独看可能贡献有限，但组合后模型能够更精确地理解用户意图——历史偏好约束了查询的解释空间，查询语义激活了偏好中相关的维度。这是正交互效应。

检测交互效应需要超越单特征消融，进行组合消融：不仅测试移除单个特征的效果，还测试同时移除或添加特征对的效果。如果两个特征的联合效应显著偏离各自独立效应的加和，就存在交互效应。形式化地：

```
interaction(A, B) = gain(A, B) - gain(A) - gain(B)
```

正值表示正交互（协同），负值表示负交互（冲突）。这个计算虽然简单，但在特征数量较多时组合空间会快速膨胀。实际工程中，通常只对已知可能冲突的特征对进行交互检测，而非穷举所有组合。

## 从特征工程到 Prompt 架构

特征工程思维最终指向一个更高层面的问题：prompt 的信息架构。[第五章](../05-架构与编排/)讨论的架构模式从系统级视角处理这个问题，而特征工程视角提供的是信息级的分析工具。

一个经过特征选择优化的 prompt 具有以下特征：每条信息都有可量化的质量贡献，信息之间的冲突被识别并处理，信息的位置分配与其重要性匹配，总信息量在预算约束内。这些都是可检验的工程标准。

## 局限性

特征工程类比的主要局限在于规模差异。传统特征工程处理的特征空间虽然可能很大（数千维），但每个特征的语义是明确的、可枚举的。Prompt 中的"特征"是自然语言文本，其语义空间是连续的、高维的、难以精确分割的。两段文本之间的"交互效应"远比两个数值特征之间的交互复杂——语义层面的冲突、互补和冗余难以用简单的数值指标完全捕获。

此外，消融实验的前提是质量评估函数的可靠性。如果 `quality_fn` 本身的噪声很大（参见[输出质量的统计度量](./输出质量的统计度量.md)关于度量噪声的讨论），那么信息增益的估计也会不可靠，基于这些估计做出的特征选择决策就可能是错误的。方法的精度受限于度量的精度——这是一个无法通过方法本身解决的约束。
