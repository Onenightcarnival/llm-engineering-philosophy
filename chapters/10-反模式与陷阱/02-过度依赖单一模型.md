# 过度依赖单一模型

## 单一供应商的供应链风险

供应链管理中有一条基本原则：关键零部件不能只有一个供应商。原因很简单：单一依赖将系统的命运交给了一个不可控的外部变量。供应商可能涨价、断供、质量下滑、被收购后改变策略——任何一种情况发生时，你都没有替代方案。

LLM 应用对单一模型的依赖具有完全相同的结构。但在 LLM 领域，这个风险比传统供应链更加剧烈，因为模型提供商的变化速度远超传统供应商。

## 模型变化的四种形态

**模型升级即破坏性变更。** 当 OpenAI 将 GPT-4 从 0314 版本升级到 0613 版本时，大量生产系统的输出质量发生了不可预期的变化。有些变好了，有些变差了，但关键是：你无法提前知道。模型升级不像软件版本升级有 changelog 可以审查——模型的行为变化是统计性的、弥散的、难以预测的。你的 prompt 在旧版本上精心调优的效果，在新版本上可能完全失效。

**模型下线。** 模型版本会被废弃。当你依赖的特定版本被停止服务时，你必须迁移到新版本，而迁移意味着重新调优所有 prompt、重新运行所有评估、重新验证所有边界条件。如果你的系统中有数十个不同的 prompt，每个都针对特定模型版本调优过，迁移的工作量是巨大的。

**定价变化。** 模型提供商可以随时调整价格。降价是好消息，但涨价或者改变计费方式（比如从按 token 计费改为按请求计费）可能让你的成本模型瞬间失效。如果你的商业模式建立在当前的 API 价格之上，价格变动就是商业风险。

**质量退化。** 这是最隐蔽的风险。模型可能在某次更新后在特定任务上的表现变差，而这种退化不会在 benchmark 上体现——benchmark 衡量的是平均表现，你关心的是你的特定用例。没有模型提供商会通知你"你的特定用例可能受影响"。

## 锁定发生在哪里

过度依赖单一模型的锁定是一个渐进的过程。它在以下几个层面悄然加深。

**Prompt 层面的锁定。** 不同模型对 prompt 的响应模式不同。针对 GPT-4 调优的 prompt 在 Claude 上可能表现更差，反之亦然。当你花了数周时间为一个模型精调 prompt 后，切换模型意味着这些工作可能需要重做。这种锁定是最常见的，也是成本最高的。

**API 层面的锁定。** 不同提供商的 API 接口不完全兼容。消息格式、参数命名、功能支持各有差异。如果你的代码直接调用特定提供商的 SDK，切换提供商需要修改所有调用点。

**能力层面的锁定。** 当你的系统依赖某个模型特有的能力——比如特定的 function calling 格式、特定的多模态输入方式、特定的上下文窗口长度——切换到不具备这些能力的模型就不仅是调优的问题，而是功能缺失的问题。

## 多模型策略

应对单一模型依赖的核心策略是：在架构层面保持模型可替换性。

```python
from abc import ABC, abstractmethod
from pydantic import BaseModel


class LLMResponse(BaseModel):
    content: str
    model: str
    input_tokens: int
    output_tokens: int


class LLMProvider(ABC):
    """模型提供者的抽象接口。

    关键设计决策：接口按能力（capability）而非提供商划分。
    不是 OpenAIProvider / AnthropicProvider，
    而是 TextGenerator / StructuredOutputGenerator。
    """

    @abstractmethod
    def generate(self, messages: list[dict], **kwargs) -> LLMResponse: ...


class ModelRouter:
    """根据任务特征路由到不同模型。

    不同任务对模型的要求不同：
    - 简单分类任务用便宜的小模型
    - 复杂推理任务用高能力的大模型
    - 延迟敏感的任务用响应快的模型
    """

    def __init__(self, providers: dict[str, LLMProvider]):
        self.providers = providers

    def route(self, task_type: str, messages: list[dict]) -> LLMResponse:
        provider_name = self._select_provider(task_type)
        return self.providers[provider_name].generate(messages)

    def _select_provider(self, task_type: str) -> str:
        # 路由策略可以是静态配置，也可以是动态决策
        routing_table = {
            "classification": "fast_cheap",
            "reasoning": "powerful",
            "generation": "balanced",
        }
        return routing_table.get(task_type, "balanced")
```

这个抽象层的价值不在于它的代码复杂度——它非常简单。价值在于它在架构层面确立了一个原则：业务逻辑不直接依赖具体的模型提供商。当需要切换模型时，修改的是 `LLMProvider` 的实现和 `ModelRouter` 的路由表，而不是散布在整个代码库中的 API 调用。

## 评估体系是多模型策略的前提

没有评估体系的多模型策略是空谈。你无法在不知道每个模型在你的任务上表现如何的情况下做出路由决策。

这意味着你需要为每个关键任务维护一个评估集（eval set），并定期在多个模型上运行评估。评估结果驱动路由决策：哪个模型在哪类任务上性价比最优。当某个模型升级后表现退化时，评估结果会捕获这个变化，路由策略随之调整。

第七章讨论的"评估即测试"框架在这里找到了直接的应用场景：评估不仅是质量保障手段，也是模型选择和路由的决策依据。

## 不是不信任，是不依赖

多模型策略不意味着对任何一个模型提供商的不信任。它意味着对单点依赖这种结构性风险的清醒认知。你可以把 80% 的流量发给你最信任的模型提供商，但保持 20% 的流量分散在替代方案上，确保切换路径是经过验证的、可执行的。

这和投资中的分散化原则在结构上同构：集中持仓将系统性风险暴露到不可接受的程度，分散化是对这一风险的结构性应对。第二章讨论的"不确定性是约束条件"在这里有直接的工程含义：模型提供商的未来行为是不确定的，正确的做法是在架构层面为这种不确定性预留缓冲。
