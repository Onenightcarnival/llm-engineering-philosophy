# 第十章 反模式与陷阱

## 核心问题

了解不应该做什么，和了解应该做什么同等重要。在一个快速演化、缺乏共识的领域，"不做什么"的指导价值不亚于"做什么"。

LLM 应用开发中存在大量看似合理实则有害的实践。它们之所以流行，往往是因为在原型阶段表现良好，在生产环境中才暴露问题——而此时技术债务已经积累到难以清理的程度。

## 文章

- [提示拼接的脆弱性](01-提示拼接的脆弱性.md) -- 字符串拼接构造 prompt 与 SQL 拼接构造查询有相同的结构性缺陷：注入风险、转义问题、可读性灾难。模板引擎只解决了语法问题而非结构问题。
- [过度依赖单一模型](02-过度依赖单一模型.md) -- 把所有赌注压在一个模型上，就像只有一个供应商的供应链。模型的四种变化形式（升级即破坏、下线、涨价、质量退化），以及模型路由器模式作为应对策略。
- [忽视成本模型](03-忽视成本模型.md) -- Token 不是免费的。成本的乘法结构（单价 x 数量 x 频次）在原型阶段被忽视，上线后才暴露。设计时的成本建模和五种优化策略。
- [把大模型当数据库用](04-把大模型当数据库用.md) -- LLM 的"知识"是训练数据的统计压缩，不是可靠的事实存储。依赖 LLM 进行事实性查询是危险的。正确的模式是 LLM 负责意图解析，数据库负责事实查询。
- [框架锁定](05-框架锁定.md) -- 过度依赖某个 LLM 框架导致的技术债务。抽象泄漏在 LLM 系统中被放大（两层不确定性叠加），以及用隔离模式保持切换能力。
- [把不确定性当缺陷](06-把不确定性当缺陷.md) -- 最隐蔽的反模式。试图通过无限堆叠约束来消除输出变异性，实际上是在对抗系统的本质属性。三种架构级替代方案。全书的收束论点——回到[第一章](../01-认识论/00-概述.md)的认识论和[第二章](../02-不确定性与决策/00-概述.md)的决策框架。

## 本章的推演逻辑

六篇文章按照危害的隐蔽程度排列：

前三篇讨论"可见的"反模式——提示拼接的脆弱性、模型依赖的风险、成本的失控。这些问题相对容易识别，因为它们会产生明确的症状（注入攻击、服务中断、账单超标）。

中间两篇讨论"半隐蔽的"反模式——误用 LLM 的能力边界、被框架绑架。这些问题的症状不那么直接（偶发的错误答案、日益增长的维护成本），需要更深入的分析才能归因。

最后一篇讨论"最隐蔽的"反模式——把不确定性当缺陷。这个反模式的隐蔽性在于：它看起来像是在追求质量（更多约束 = 更好的输出），实际上是在对抗系统的本质。它是全书核心论点的反面——理解了不确定性是约束条件而非缺陷，就理解了为什么这个反模式是最根本的错误。

本章的反模式清单与[第十一章](../11-终章/00-概述.md)形成对照：第十章说"不要做什么"，第十一章说"应该走向哪里"。
