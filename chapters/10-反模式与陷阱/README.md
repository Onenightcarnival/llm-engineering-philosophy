# 第十章 反模式与陷阱

## 核心问题

了解不应该做什么，和了解应该做什么同等重要。LLM 应用开发中存在大量看似合理实则有害的实践，值得专门讨论。

## 计划文章

- **Prompt 拼接的脆弱性** -- 字符串拼接构造 prompt 与 SQL 拼接构造查询有相同的结构性缺陷：注入风险、转义问题、可读性灾难。以及为什么模板引擎不是银弹。
- **过度依赖单一模型** -- 把所有赌注压在一个模型上，就像只有一个供应商的供应链。模型会升级、会下线、会涨价、会变笨。如何设计模型无关的抽象层。
- **忽视成本模型** -- Token 不是免费的。一个 prompt 的成本 = 输入 token + 输出 token，乘以调用频次。很多开发者在 prototype 阶段对成本毫无概念，上线后才发现账单不可接受。
- **把 LLM 当数据库用** -- LLM 不是知识库，不是搜索引擎，不是计算器。它的"知识"是训练数据的统计压缩，有截止日期，有偏差，有幻觉。依赖 LLM 进行事实性查询是危险的。
- **框架锁定** -- 过度依赖某个 LLM 框架导致的技术债务。框架的抽象泄漏、版本不兼容、社区衰退——这些风险在快速演化的 LLM 生态中被放大了。
- **把不确定性当 bug 而非特性** -- 最隐蔽的反模式。试图通过无限加长 prompt、堆叠规则来消除 LLM 输出的变异性，实际上是在对抗系统的本质。正确的做法是接受不确定性，在架构层面处理它。
