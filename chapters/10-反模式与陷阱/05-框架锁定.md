# 框架锁定

## 框架的诱惑与代价

第六章第四节已经从"过度设计"的角度讨论了 LLM 编排框架的问题。本文讨论一个不同但相关的问题：即使你选择了一个当前看来合理的框架，过度依赖它仍然会制造严重的技术债务。

框架锁定（framework lock-in）不是 LLM 领域的新问题。Java 世界经历过 Struts 到 Spring 的迁移阵痛，Python 世界经历过 Django 到 FastAPI 的取舍。但 LLM 生态的特殊性在于：这个领域的演化速度远超传统 Web 框架。传统 Web 框架的 major version 周期以年计，LLM 框架的 breaking change 可能以月计。这意味着框架锁定的风险被时间压缩了。

## 锁定的四个信号

**你的业务逻辑无法脱离框架运行。** 如果你的 prompt 构造、输出解析、流程编排全部使用框架特有的 API，那么你的业务逻辑已经和框架融为一体。要验证这一点很简单：尝试在不引入框架的情况下运行你的核心业务逻辑。如果做不到，你就被锁定了。

**框架升级变成了项目风险。** 当框架发布新版本时，你的第一反应不是"有什么新功能"，而是"会不会破坏现有代码"。如果每次框架升级都需要大量的适配工作，这说明你的代码与框架的耦合度超过了合理范围。

**你在绕过框架。** 当你发现自己在写代码绕过框架的限制——monkey patching、访问内部 API、用 hack 的方式实现框架不支持的功能——这说明框架的抽象不再适合你的需求，但你已经无法轻易脱离它。

**调试需要理解框架内部实现。** 当一个 bug 的定位需要你深入框架源码、理解框架内部的调用链和状态管理时，框架不再是"节省时间的工具"，而是"增加复杂度的负担"。

## 抽象泄漏的放大效应

所有抽象都会泄漏——这是 Joel Spolsky 二十年前就指出的。但在 LLM 领域，抽象泄漏的后果比传统软件更严重。

传统 Web 框架的抽象泄漏通常表现为：某个边界情况下的行为不符合文档描述，需要你了解底层实现来解决。这是烦人的，但通常是局部的。

LLM 框架的抽象泄漏有一个独特的维度：框架对 LLM 行为的抽象是建立在一个本身就不确定的系统之上的。当框架告诉你"使用 `Agent` 类来构建一个自主决策的系统"时，它隐藏了大量关于 prompt 工程、错误处理、成本控制的细节。这些细节在框架的抽象表面之下，但它们决定了系统的实际行为。当系统行为不符预期时，你需要同时理解框架的抽象层和 LLM 的概率性行为——两层不确定性的叠加让调试难度不是加法而是乘法。

## 什么时候用框架是合理的

批判框架锁定不等于主张完全不用框架。框架在以下场景中有合理的使用理由。

**探索阶段的快速原型。** 在你还不确定产品形态时，框架提供了快速实验的能力。关键的决策点在于：原型验证完成后，生产实现是否需要保留框架。很多团队在这个决策点上犯了惰性的错误——因为"原型已经用了框架"就继续在生产环境中使用它。

**标准化需求超过定制化需求。** 如果你的团队需要在多个项目中使用统一的 LLM 集成方式，而每个项目的需求都比较标准（基本的 RAG、简单的对话流程），框架提供的统一接口减少了重复工作。

**团队能力与时间约束。** 如果团队缺乏 LLM 应用开发经验，且上线压力大，框架降低了入门门槛。但这是一个短期决策，长期来看，团队应该逐步建立直接使用 LLM API 的能力。

## 降低锁定风险的设计原则

如果决定使用框架，以下原则可以降低锁定风险。

**隔离框架依赖。** 将框架的使用限制在基础设施层，不要让框架的类型和接口渗透到业务逻辑中。业务逻辑应该依赖你自己定义的接口，框架是这些接口的一个实现。

```python
from abc import ABC, abstractmethod
from pydantic import BaseModel


# 你的接口——不依赖任何框架
class RetrievalResult(BaseModel):
    content: str
    source: str
    relevance_score: float


class Retriever(ABC):
    @abstractmethod
    def retrieve(self, query: str, top_k: int = 5) -> list[RetrievalResult]: ...


# 框架实现——隔离在基础设施层
class LangChainRetriever(Retriever):
    """如果你用 LangChain，这是唯一依赖它的地方。"""

    def __init__(self, vectorstore):
        self._vectorstore = vectorstore

    def retrieve(self, query: str, top_k: int = 5) -> list[RetrievalResult]:
        docs = self._vectorstore.similarity_search(query, k=top_k)
        return [
            RetrievalResult(
                content=doc.page_content,
                source=doc.metadata.get("source", "unknown"),
                relevance_score=doc.metadata.get("score", 0.0),
            )
            for doc in docs
        ]


# 无框架实现——切换时只需替换这一个类
class DirectRetriever(Retriever):
    """直接使用向量数据库客户端，无框架依赖。"""

    def __init__(self, db_client):
        self._client = db_client

    def retrieve(self, query: str, top_k: int = 5) -> list[RetrievalResult]:
        results = self._client.search(query, limit=top_k)
        return [
            RetrievalResult(
                content=r["text"],
                source=r["metadata"]["source"],
                relevance_score=r["score"],
            )
            for r in results
        ]
```

**保持切换路径畅通。** 定期验证你能够在不使用框架的情况下实现核心功能。这不是说你真的要维护两套实现，而是说你要确保业务逻辑没有与框架的内部机制产生不可分离的耦合。

**控制框架的使用范围。** 不要因为引入了一个框架就把所有功能都用它来实现。如果你只需要向量检索，就只用框架的向量检索部分，不要顺便把 prompt 管理、输出解析、对话历史管理也交给它。每增加一个依赖框架的功能点，切换成本就增加一分。

在一个每隔几个月就会出现新范式的领域，保持架构的灵活性不是过度设计，而是生存策略。框架是工具，不是归宿。
