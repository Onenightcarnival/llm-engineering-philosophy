# 忽视成本模型

## 原型阶段的成本盲区

几乎所有 LLM 应用的开发都从一个让人兴奋的原型开始：写一个 prompt，调一次 API，看到模型返回了令人满意的结果。API 调用的成本是几分钱，完全不值得关注。

然后你上线了。用户量从 1 个变成 1000 个，每个用户每天发起 10 次请求，每次请求的 prompt 包含 2000 个 input token，模型返回 500 个 output token。月底账单到了，数字可能让你重新审视整个项目的商业可行性。

这个故事反复发生的原因是：开发者在原型阶段建立了"API 调用很便宜"的直觉，而这个直觉在规模化后是完全错误的。单次调用确实便宜，但成本是乘法结构：

**总成本 = (输入 token 单价 x 输入 token 数 + 输出 token 单价 x 输出 token 数) x 调用频次**

原型阶段通常只关注公式中的单价部分，而忽略了 token 数和频次这两个乘数。当乘数从"手动测试几次"变成"生产环境中每天十万次调用"时，成本可以轻松增长五个数量级。

## 成本盲点的具体来源

**上下文注入的隐性成本。** RAG 系统中每次调用都会注入检索到的上下文片段。如果检索返回 5 个文档片段，每个 500 token，那么每次调用额外增加 2500 个 input token。这部分成本在 prompt 设计时很容易被忽略，因为开发者关注的是"检索结果是否相关"，而不是"检索结果有多长"。

**对话历史的累积成本。** 多轮对话系统中，每一轮都需要将之前的对话历史放入上下文。第 10 轮对话的 input token 数量是第 1 轮的数倍。如果不主动截断或压缩历史，成本随对话轮次线性增长。

**重试和冗余调用。** 当模型输出不符合预期时，常见的做法是重试。如果重试率是 20%，实际成本就是预期成本的 1.2 倍。加上验证调用（用另一个 LLM 检查输出质量）、fallback 调用（主模型失败时切换到备选模型），实际的 API 调用次数可能是业务逻辑所需的 2-3 倍。

**输出 token 的价格差异。** 大多数提供商对输出 token 的收费高于输入 token。如果你的任务需要模型生成长文本（摘要、报告、代码），输出 token 的成本可能占总成本的主要部分。这一点在比较不同模型的定价时尤其重要：不能只看 input token 的单价。

## 成本估算应该是设计的一部分

成本估算不应该在开发完成后才进行，而应该是系统设计阶段的一个必要环节，与功能设计、性能设计并列。

```python
from dataclasses import dataclass


@dataclass
class CostEstimate:
    """单次调用的成本估算。"""
    avg_input_tokens: int
    avg_output_tokens: int
    input_price_per_1k: float   # 每 1000 input token 的价格（美元）
    output_price_per_1k: float  # 每 1000 output token 的价格（美元）

    @property
    def cost_per_call(self) -> float:
        input_cost = self.avg_input_tokens / 1000 * self.input_price_per_1k
        output_cost = self.avg_output_tokens / 1000 * self.output_price_per_1k
        return input_cost + output_cost

    def monthly_cost(self, daily_calls: int) -> float:
        return self.cost_per_call * daily_calls * 30

    def report(self, daily_calls: int) -> str:
        monthly = self.monthly_cost(daily_calls)
        return (
            f"单次调用成本: ${self.cost_per_call:.4f}\n"
            f"日调用量: {daily_calls:,}\n"
            f"预估月成本: ${monthly:,.2f}"
        )


# 设计阶段就应该做的估算
rag_query = CostEstimate(
    avg_input_tokens=3000,   # system prompt + 检索上下文 + 用户问题
    avg_output_tokens=500,
    input_price_per_1k=0.01,
    output_price_per_1k=0.03,
)

print(rag_query.report(daily_calls=100_000))
# 单次调用成本: $0.0450
# 日调用量: 100,000
# 预估月成本: $135,000.00
```

13.5 万美元的月成本足以让大多数项目重新评估架构决策。而这个数字不是上线后才能知道的——它在设计阶段就可以估算。

## 优化策略

成本优化不是事后补救，而是设计决策的一部分。

**按任务复杂度选择模型。** 不是所有任务都需要最强的模型。简单的分类任务、格式转换、关键词提取——这些任务用小模型或专用模型就足够了，成本可能只有大模型的十分之一。第二篇文章讨论的多模型策略在这里有直接的成本含义。

**缓存。** 相同或相似的查询不需要每次都调用模型。语义缓存（基于查询的语义相似度而非精确匹配来命中缓存）可以显著降低重复查询的成本。缓存命中率的提升直接转化为成本的下降。

**压缩上下文。** 检索到的上下文片段不需要全文注入。摘要、截断、关键句提取——这些预处理步骤会减少 input token 数量。对话历史也可以定期压缩：用一次 LLM 调用将长对话历史压缩为结构化摘要，后续调用使用摘要而非原始历史。

**控制输出长度。** 通过 prompt 设计和 API 参数（`max_tokens`）控制模型的输出长度。如果任务只需要一个分类标签，就不要让模型生成一段解释。输出 token 比输入 token 贵，控制输出长度是成本优化中最直接的手段。

**批量处理。** 对延迟不敏感的任务可以积攒后批量调用。部分提供商对批量调用提供折扣。即使没有折扣，批量处理也允许你在一个 prompt 中处理多个输入，分摊 system prompt 的 token 成本。

## 成本是架构约束

成本不是一个独立于技术决策的商业问题。它是架构约束的一部分，和延迟、吞吐量、可靠性同等重要。

一个在功能上完美但在成本上不可持续的系统，不是一个好的系统。成本约束应该在设计阶段就被纳入决策框架：选择模型时考虑性价比而非只考虑能力，设计 prompt 时考虑 token 效率而非只考虑效果，选择架构模式时考虑调用次数而非只考虑优雅性。

第二章讨论的"不确定性是约束条件"的思维框架在这里可以直接迁移：成本也是约束条件。正确的做法不是忽略它然后祈祷账单可以接受，而是在给定成本约束下做出最优的技术决策。
