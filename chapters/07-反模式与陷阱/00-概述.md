# 第七章 反模式与陷阱

## 核心问题

了解不应该做什么，和了解应该做什么同等重要。在一个快速演化、缺乏共识的领域，"不做什么"的指导价值不亚于"做什么"。

LLM 应用开发中存在大量看似合理实则有害的实践。它们之所以流行，往往是因为在原型阶段表现良好，在生产环境中才暴露问题——而此时技术债务已经积累到难以清理的程度。

## 文章

- [提示拼接的脆弱性](01-提示拼接的脆弱性.md) -- 字符串拼接构造 prompt 与 SQL 拼接构造查询有相同的结构性缺陷：注入风险、转义问题、可读性灾难。模板引擎只解决了语法问题而非结构问题。
- [模型选择的工程约束](02-模型选择的工程约束.md) -- 模型选择是一个多维约束优化问题。单一模型依赖的供应链风险（升级/下线/涨价/退化四种变化形态）与成本的乘法结构是同一个决策空间中的不同维度，需要在设计阶段统一纳入架构考量。
- [能力边界的误判](03-能力边界的误判.md) -- 两种对称的错误：高估 LLM 的能力（把大模型当数据库用，依赖统计压缩进行事实性查询）和高估框架的抽象（框架锁定，让两层不确定性叠加放大调试难度）。本质都是把工具放在它不适合的位置。
- [把不确定性当缺陷](04-把不确定性当缺陷.md) -- 最隐蔽的反模式。试图通过无限堆叠约束来消除输出变异性，实际上是在对抗系统的本质属性。三种架构级替代方案。全书的收束论点——回到[第一章](../01-认识论/00-概述.md)的认识论和[第二章](../02-不确定性与决策/00-概述.md)的决策框架。
- [何时微调而非继续提示](05-何时微调而非继续提示.md) -- 一个实际的工程决策。Prompt 工程的能力天花板、微调的成本结构与风险分析、"prompt 优先、微调兜底"的决策框架。当 prompt 优化已经触顶，继续在 prompt 层面投入就是一种反模式。

## 本章的推演逻辑

五篇文章按照危害的隐蔽程度排列，分为三层：

第一篇讨论"可见的"反模式——提示拼接的脆弱性。这个问题有明确的安全症状（注入攻击、结构破坏），相对容易识别和修复。

第二、三篇讨论"半隐蔽的"反模式——模型选择的约束和能力边界的误判。这些问题的症状是弥散的（偶发的错误答案、日益增长的维护成本、账单超标），需要更深入的分析才能归因到架构决策。

第四、五篇讨论"最隐蔽的"反模式——把不确定性当缺陷，以及在 prompt 已经触顶时仍在 prompt 层面投入。这两个反模式藏在开发者的心智模型里，不体现为任何一行可以被指出的错误代码，而是体现为错误的目标设定。第四篇是全书核心论点的反面——理解了不确定性是约束条件而非缺陷，就理解了为什么这个反模式是最根本的错误。

本章的反模式清单与[第八章](../08-终章/00-概述.md)形成对照：第七章说"不要做什么"，第八章说"应该走向哪里"。
