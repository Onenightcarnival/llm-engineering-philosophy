# 把大模型当数据库用

## 一个诱人的错觉

大模型看起来"什么都知道"。你问它某个 API 的参数列表，它能答上来。你问它某个历史事件的时间，它能给出日期。你问它某家公司的产品线，它能列出来。这种表象制造了一个诱人的错觉：大模型是一个自然语言接口的知识库。

这个错觉的危害在于，它让开发者把 LLM 放到了一个它在结构上不适合的位置。数据库存储的是数据，查询返回的是精确匹配。LLM 存储的是训练数据的统计压缩，"查询"返回的是条件概率分布上的采样。

这两者之间的区别是本质上的：LLM 根本不是在做查询。

## 统计压缩不是存储

当 LLM 回答"Python 的 `os.path.join` 函数接受哪些参数"时，它没有从某个参数数据库中检索出答案。它做的是：基于训练数据中关于 `os.path.join` 的大量文本，生成一个在统计上最可能的 token 序列。如果训练数据中关于这个函数的描述是一致的，模型的回答通常是正确的。但"通常正确"和"保证正确"之间有一条工程上不可忽略的鸿沟。

**训练数据截止日期。** 模型的"知识"定格在训练数据的时间点。API 可能已经更新、公司可能已经改名、法律可能已经修订——模型对这一切一无所知，更危险的是，它不会告诉你它不知道。它会以同样自信的语气给出过期信息和当前信息。

**频率偏差。** 模型对训练数据中频繁出现的信息"记忆"更好，对罕见信息的回答更不可靠。主流编程语言的 API 文档比小众语言的更准确，热门公司的信息比冷门公司的更可靠。这意味着模型的准确率不是均匀分布的——恰好在你最需要准确信息的长尾场景中，模型最不可靠。

**幻觉。** 这是最根本的问题。当模型对某个问题"不知道"时，它不会说"我不知道"——它会生成一个在统计上"看起来像正确答案"的输出。这就是幻觉：形式上完美、内容上虚构的回答。一个编造的 API 参数名、一个不存在的函数、一个虚构的论文引用——它们在文本层面和真实信息没有任何可区分的特征。

## 危险的使用模式

以下几种模式是"把大模型当数据库用"的典型表现。

**事实性查询。** 直接向 LLM 询问事实性信息并将其作为系统输出的一部分。例如：让 LLM 回答用户关于产品规格、法律条款、医疗信息的问题。这些信息的准确性至关重要，而 LLM 无法保证准确性。

**数值计算。** 第一章已经详细分析了为什么 LLM 不是计算器。但在实践中，开发者仍然会要求 LLM 计算统计数据、财务指标或数学表达式，然后直接使用返回的数值。

**状态查询。** 询问 LLM 关于"当前"状态的信息——某个服务是否可用、某个 API 的最新版本号、某个库的最新版本。模型的回答基于训练数据，不是基于实时状态。

## 正确的职责划分

LLM 的正确用法是理解自然语言意图并将其转化为结构化查询。

```python
from pydantic import BaseModel, Field
from typing import Literal
from datetime import date


class ProductQuery(BaseModel):
    """LLM 的输出：结构化的查询意图，不是查询结果。"""
    product_category: str
    attributes: list[str] = Field(
        description="用户关心的产品属性，如价格、规格、库存"
    )
    comparison: bool = Field(
        description="用户是否在进行产品对比"
    )
    time_constraint: Literal["current", "historical"] = "current"


def handle_product_question(user_input: str):
    """正确的职责划分：LLM 解析意图，数据库提供事实。"""

    # 第一步：LLM 将自然语言转化为结构化查询意图
    query = llm_parse(user_input, output_type=ProductQuery)

    # 第二步：用结构化查询去查真正的数据源
    results = product_database.query(
        category=query.product_category,
        attributes=query.attributes,
    )

    # 第三步（可选）：LLM 将结构化结果组织为自然语言回答
    response = llm_generate(
        system="基于以下产品数据回答用户问题。只使用提供的数据，不要编造信息。",
        context=results,
        user_query=user_input,
    )
    return response
```

这个模式的关键在于：LLM 在第一步中做的是意图解析（它擅长的事），不是事实回答（它不可靠的事）。事实性信息来自数据库——一个确定性的、可审计的、有明确数据来源的系统。LLM 在第三步中做的是自然语言生成（它擅长的事），但生成的依据是数据库提供的事实，不是它自己的"记忆"。

## 混合模式与防护措施

完全禁止 LLM 回答任何事实性问题在某些场景下不现实。用户期望对话系统能回答常识性问题，而为每一个可能的问题都准备一个结构化数据源也不经济。

务实的做法是分级处理：

**高风险事实：必须外部验证。** 涉及法律、医疗、财务、安全的事实性信息，必须来自权威数据源，LLM 只负责意图解析和结果组织。这是不可妥协的底线。

**中风险事实：标注不确定性。** 产品信息、技术文档、历史数据——使用 RAG 从可信数据源检索，但在输出中标注信息来源和时效性。让用户知道信息的依据是什么，而不是伪装成全知全能。

**低风险事实：允许模型直接回答，但标注免责。** 常识性问题、非关键性的背景信息——可以依赖模型的"知识"，但系统应该明确告知用户这些信息未经实时验证。

把大模型当数据库用，本质上是对模型能力边界的误判。第一章讨论的大模型不擅长什么，在这里有最直接的工程后果：事实性准确不是大模型的能力范围，把事实性问题交给不保证事实准确性的系统，是架构层面的错误，不是可以通过更好的 prompt 修复的问题。
