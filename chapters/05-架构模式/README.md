# 第五章 架构模式

## 核心问题

LLM 应用并非没有架构可言。当复杂度超过一定阈值，结构设计就不再是可选项。问题在于：哪些架构模式是这个领域真正需要的，哪些是从传统软件工程中生搬硬套的。

## 计划文章

- **RAG 的本质：检索增强还是上下文注入** -- RAG 被过度包装了。剥开框架的外衣，它的核心操作不过是"在调用 LLM 之前，把相关信息塞进上下文"。理解这个本质，才能判断何时需要 RAG，何时不需要。
- **Agent 的结构分解** -- Agent 不是一个神秘的概念。它是"LLM + 工具调用 + 循环控制"的组合。本文拆解 Agent 的最小结构，讨论状态机模型、工具绑定模式和终止条件设计。
- **LLM 作为胶水层 vs LLM 作为核心引擎** -- 两种根本不同的架构定位。前者意味着 LLM 负责连接和转换，核心逻辑仍由确定性代码承载；后者意味着 LLM 本身就是业务逻辑。大多数应用应该选择前者。
- **状态管理与会话设计** -- 无状态调用 vs 有状态会话。上下文窗口不是数据库。会话状态的序列化、压缩和恢复策略。为什么"把所有历史消息都塞进去"是最懒也最脆弱的方案。
