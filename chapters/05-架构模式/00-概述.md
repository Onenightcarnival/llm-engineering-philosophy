# 第五章 架构模式

## 核心问题

LLM 应用并非没有架构可言。当复杂度超过一定阈值，结构设计就不再是可选项。问题在于：哪些架构模式是这个领域真正需要的，哪些是从传统软件工程中生搬硬套的。

前四章讨论的都是单次 LLM 调用层面的问题：如何理解 LLM 的本质、如何做工程决策、如何设计 prompt、如何用类型系统约束输出。本章将视角扩展到系统层面：当一个应用包含多个 LLM 调用、外部工具、数据存储和业务逻辑时，如何组织它们之间的关系。

## 文章

- [RAG 的本质：检索增强还是上下文注入](01-RAG的本质.md) -- RAG 被过度包装了。剥开框架的外衣，它的核心操作不过是"在调用 LLM 之前，把相关信息塞进上下文"。理解这个本质，才能判断何时需要 RAG、何时不需要，以及检索质量为什么是系统的真正瓶颈。
- [Agent 的结构分解](02-Agent的结构分解.md) -- Agent 不是一个神秘的概念。它是"LLM + 工具调用 + 循环控制"的组合。本文拆解 Agent 的最小结构，讨论工具绑定原则、终止条件设计和状态机形式化。
- [LLM 作为胶水层 vs LLM 作为核心引擎](03-胶水层与核心引擎.md) -- 两种根本不同的架构定位。前者意味着 LLM 负责连接和转换，核心逻辑仍由确定性代码承载；后者意味着 LLM 本身就是业务逻辑。定位决定了测试策略和可靠性模型。
- [状态管理与会话设计](04-状态管理与会话设计.md) -- 上下文窗口不是数据库。会话状态的分层管理、对话历史的压缩策略、无状态与有状态架构的选择标准——这些设计决策的质量决定了系统在长对话场景下的表现。

## 本章的推演逻辑

四篇文章覆盖了 LLM 应用架构的四个核心维度：

第一篇讨论数据维度——如何为 LLM 提供它不具备的外部知识，RAG 的本质和边界。第二篇讨论控制维度——当 LLM 需要与外部世界交互时，Agent 模式的最小结构和设计原则。第三篇讨论定位维度——LLM 在整个系统中应该扮演什么角色，这个战略决策如何影响所有后续的技术选择。第四篇讨论状态维度——会话系统如何管理跨调用的状态，上下文窗口的局限性如何影响架构设计。

本章的架构模式是静态的结构描述。当这些结构需要按照特定的逻辑被组合和执行时，就进入了[第六章](../06-工作流与编排/00-概述.md)的领域：工作流与编排。
