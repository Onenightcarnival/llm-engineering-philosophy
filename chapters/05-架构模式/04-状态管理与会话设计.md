# 状态管理与会话设计

## 上下文窗口不是数据库

LLM 的上下文窗口经常被当作一个"临时数据库"来使用：把所有历史消息、系统指令、检索结果、工具输出全部塞进去，让 LLM 自行理解和利用。这种做法在原型阶段看似有效，但它的问题会随着对话长度和系统复杂度的增加而指数级暴露。

上下文窗口与数据库的根本区别在于三点：

**容量有限且昂贵。** 每个 token 都占用推理资源，无论它是否被"关注"到。128k token 的上下文窗口看起来很大，但一次包含系统 prompt、对话历史、检索结果和工具输出的调用，很容易消耗数万 token。成本与上下文长度近似线性相关。

**没有索引。** 数据库通过索引实现 O(log n) 的查找。上下文窗口没有索引——LLM 需要通过注意力机制"扫描"所有内容来找到相关信息。虽然长上下文模型在技术上能处理大量内容，但注意力的稀释效应意味着放在上下文末尾的信息可能比放在开头的信息获得更少的"关注"。

**无持久性。** 上下文窗口的内容随每次 API 调用重新构建。上一次调用的上下文不会自动延续到下一次调用——开发者必须显式管理状态的序列化和恢复。

## 会话状态的分层管理

有效的状态管理需要区分不同类型的状态，并为每种类型选择合适的存储和传递机制。

**系统状态：** 应用的全局配置、用户权限、功能开关。这些信息在会话期间不变或很少变化，应该作为 system prompt 的一部分。优化策略：只包含与当前任务相关的系统信息，而非完整的系统配置。

**对话状态：** 用户和 LLM 之间的消息历史。这是最占空间的部分，也是最需要管理的部分。天真的做法是保留所有历史消息；工程化的做法是有策略地压缩和裁剪。

**任务状态：** 当前正在执行的任务的中间结果、已调用的工具、已获取的信息。在 Agent 场景中尤其重要。任务状态应该被结构化存储（而非作为自由文本混在对话历史中），以便于查询和恢复。

**外部状态：** 数据库记录、文件内容、API 响应。这些状态不应该被缓存在上下文窗口中——每次需要时从源头获取最新值，避免使用过期数据。

## 对话历史的压缩策略

当对话超过一定长度，完整保留所有历史消息会导致上下文溢出或成本失控。压缩是必须的，问题是如何压缩才能最小化信息损失。

**滑动窗口。** 只保留最近 N 轮对话。简单直接，但丢失了早期对话中可能仍然相关的上下文。适合聊天类应用，不适合需要长期记忆的场景。

**摘要压缩。** 用 LLM 将早期对话压缩为摘要，用摘要替代原始消息。信息保留率高于滑动窗口，但摘要本身可能丢失细节，而且生成摘要需要额外的 LLM 调用。

```python
def compress_history(messages: list[dict], max_recent: int = 5) -> list[dict]:
    if len(messages) <= max_recent:
        return messages

    old_messages = messages[:-max_recent]
    recent_messages = messages[-max_recent:]

    summary = call_llm(
        f"将以下对话历史压缩为简要摘要，保留关键信息和决策：\n"
        + format_messages(old_messages)
    )

    return [
        {"role": "system", "content": f"之前的对话摘要：{summary}"},
        *recent_messages
    ]
```

**选择性保留。** 根据消息的"重要性"决定保留还是丢弃。包含关键决策、用户偏好声明、重要事实的消息应该被保留；寒暄、重复确认、格式调整等消息可以丢弃。重要性的判断可以用规则（包含特定关键词）或 LLM 辅助评估。

没有完美的压缩策略。每种策略都在上下文长度、信息保留和计算成本之间做权衡。正确的做法是根据应用场景选择策略，而非追求一个通用方案。

## 无状态 vs 有状态的架构选择

**无状态架构：** 每次 LLM 调用是独立的。所有必要的上下文在每次调用时完整传入。优势是简单——不需要状态存储、不存在状态不一致问题、天然支持水平扩展。代价是每次调用的上下文可能很大（重复传入不变的信息）。

**有状态架构：** 在服务端维护会话状态，LLM 调用只传入增量信息。优势是减少了 token 消耗，支持更长的"有效记忆"。代价是需要状态存储和管理，引入了状态一致性问题，水平扩展需要解决状态亲和性。

选择标准与传统 Web 架构的 session 管理决策同构：

- 交互轮次少（< 5 轮）、不需要长期记忆：无状态。
- 交互轮次多、需要跨轮次记忆、需要中断后恢复：有状态。
- 需要水平扩展但也需要状态：有状态 + 外部状态存储（Redis、数据库）。

## 会话设计的常见陷阱

**把所有信息都塞进 system prompt。** System prompt 应该包含角色定义和全局约束，不是百科全书。过长的 system prompt 稀释了 LLM 对关键指令的注意力，同时增加了每次调用的基础成本。

**在对话历史中混合不同类型的状态。** 工具调用结果、中间计算步骤、系统通知——这些信息混在用户消息和 LLM 回复之间，让对话历史变得难以解析和压缩。应该用结构化的方式分离不同类型的消息。

**假设 LLM 会"记住"上下文中的所有信息。** LLM 不会均匀地关注上下文中的所有内容。关键信息应该出现在上下文的开头或末尾（注意力的 U 形分布），重要约束应该在需要时重复强调，而不是只在 system prompt 中提一次。

这些陷阱的共同根源是把上下文窗口当作一个可靠的、无限的、均匀的存储空间。它不是。理解它的局限性是设计健壮会话系统的前提。
